## 📖 Story

**Como** desenvolvedor da plataforma Loyola Digital X,
**Quero** um service `kiwify.ts` que autentique na Kiwify (OAuth2 com cache de token), liste produtos recorrentes, varra o stream de vendas em janelas de 90 dias e compute o dashboard de recorrência (Pull-MVP),
**Para** que as rotas (35.3) sirvam as métricas derivadas de vendas sem que o frontend precise conhecer o contrato da Kiwify.

---

## ✅ Acceptance Criteria

1. **Auth OAuth2 com cache de token** (`getKiwifyToken(clientId, clientSecret)`):
   - `POST https://public-api.kiwify.com/v1/oauth/token`, body `application/x-www-form-urlencoded` com `client_id`+`client_secret`.
   - Cache em memória (`Map` por par de credenciais), refresh ~5min antes do `expires_in` (default 86400s).
   - Em falha, lança `Error` **sem vazar** client_secret/token. `clearKiwifyTokenCache()` para testes.

2. **Cliente HTTP** `kiwifyGet<T>(token, accountId, path, params?)`:
   - Headers `Authorization: Bearer <token>` **+ `x-kiwify-account-id: <accountId>`**.
   - Monta querystring ignorando `undefined/null/""`. Em `!res.ok`, lança `Error` com status + detalhe da Kiwify (sem PII).

3. **Cripto helpers** `encryptKiwifySecret`/`decryptKiwifySecret` (wrappers sobre `services/encryption.ts`).

4. **`listKiwifyProducts(token, accountId)`** → lista distinct `{ id, name }` dos produtos com **`payment_type === "recurring"`**, paginando `/v1/products` por `page_number`/`page_size` até `pagination.count`. Ordenado por name.

5. **`fetchSalesWindowed(token, accountId, { productId, status?, from, to, fullDetails? })`** → varre `/v1/sales` respeitando o **teto de 90 dias**: fatia `[from,to]` em janelas ≤90d e, dentro de cada janela, pagina por offset (`page_number`/`page_size`) até esgotar `pagination.count`. Guard contra loop infinito (MAX_PAGES). Retorna todas as vendas.

6. **`fetchSalesCount(token, accountId, { productId, status, from, to })`** → conta sem baixar tudo: soma `pagination.count` das janelas de 90d (1 chamada por janela, `page_size=1`).

7. **`fetchKiwifyStats(token, accountId, { from, to })`** → `GET /v1/stats` (refund_rate, chargeback_rate, total_sales).

8. **Agregação pura testável** `aggregateKiwifyDashboard(input)` (sem rede) e orquestrador `computeKiwifyDashboard(token, accountId, { productId, months })` que produz `KiwifyDashboard` conforme o contrato em Dev Notes — incluindo os **gaps honestos** (`activeSubscriptions: null`, `churnRate: null`).

9. **Helpers de data:** janelas de 90 dias entre `from`/`to` em formato `YYYY-MM-DD` (a Kiwify usa data, não epoch ms — diferente da Hotmart). `monthsAgo(months)` para o default 12m.

10. **Testes unitários** (`kiwify-service.test.ts`): cripto round-trip, `distinctRecurringProducts`, janelamento de 90 dias (ex: 12 meses → 5 janelas), `aggregateKiwifyDashboard` com fixtures (receita por moeda, buckets de status, novos vs renovação, MRR 30d). **Sem rede** (mockar fetchers).

11. **Qualidade:** `npm run typecheck`, `npm run lint`, `npm test` (kiwify) passam.

## 📊 Complexity Estimate

**Story Points:** 6 (Medium) · ~1.5 dia. Espelha `services/hotmart.ts`, mas com **3 diferenças do adaptador**: datas `YYYY-MM-DD` (não epoch ms), paginação por **offset** (não cursor), e métricas derivadas de **vendas** (não de subscriptions). Agregação pura → alta testabilidade.

## ⚠️ Risks & Mitigations

| Risk | Severity | Mitigation |
|------|----------|-----------|
| Teto de 90 dias esquecido → API 400 | HIGH | `fetchSalesWindowed` **sempre** fatia em janelas ≤90d (AC5); teste cobre 12m→5 janelas. |
| Janela larga + muitas vendas → lento | MEDIUM | Para contagens usar `fetchSalesCount` (page_size=1, lê só `pagination.count`); baixar itens só quando necessário (novos vs renovação). |
| MRR/receita confundir moedas | LOW | Agrupar por `currency`; primária BRL. `net_amount` em **centavos** → dividir por 100 na borda de exibição (manter centavos no backend). |
| Vazar secret/PII em erro/log | HIGH | Mensagens de erro só status+detalhe Kiwify; nunca logar token/secret/customer. |

## 🔧 Dev Notes

### Contrato de saída — `KiwifyDashboard` (nomes EXATOS)

```ts
interface MoneyByCurrency { currency: string; value: number; } // value em CENTAVOS
interface KiwifyChargeBucket { count: number; value: MoneyByCurrency[]; }

interface KiwifyDashboard {
  recurringRevenue: MoneyByCurrency[];   // Σ net_amount de paid/approved no período
  mrrApprox: MoneyByCurrency[];          // receita recorrente dos últimos 30 dias
  charges: {
    paid: KiwifyChargeBucket;            // status paid + approved
    refunded: KiwifyChargeBucket;        // refunded + refund_requested + pending_refund
    chargeback: KiwifyChargeBucket;      // chargedback
    pending: KiwifyChargeBucket;         // waiting_payment + pending + processing
    refused: { count: number };          // refused (sem valor — não houve receita)
  };
  refundRate: number;                    // de /stats (%)
  chargebackRate: number;                // de /stats (%)
  newVsRenewal: { new: number; renewal: number }; // parent_order_id vazio vs preenchido
  statusDistribution: Array<{ status: string; count: number }>;
  currencyPrimary: string;               // "BRL"
  // GAPS honestos — não disponíveis via pull (fase 2 / webhooks):
  activeSubscriptions: null;
  churnRate: null;
}
```

### Buckets de status (enum oficial 11) → mapeamento

- **paid:** `paid`, `approved` · **refunded:** `refunded`, `refund_requested`, `pending_refund` · **chargeback:** `chargedback` · **pending:** `waiting_payment`, `pending`, `processing` · **refused:** `refused`. (`authorized` ⊂ pending até captura — tratar como pending.)

### Adaptador — diferenças vs Hotmart (NÃO copiar cego)

| Aspecto | Hotmart | **Kiwify** |
|---|---|---|
| Datas | epoch ms UTC | **`YYYY-MM-DD`** (string) |
| Paginação | cursor (`next_page_token`) | **offset** (`page_number`/`page_size`, `pagination.count`) |
| Header extra | — | **`x-kiwify-account-id`** obrigatório |
| Janela | `accession_date` (default 30d) | **teto rígido 90d** por chamada → janelar |
| Assinatura | `/subscriptions` com status | **só `/sales`** → derivar de vendas recurring |
| Valores | unidade da moeda | **centavos** |

### MRR aproximado

`mrrApprox` = `recurringRevenue` recalculado para a janela dos **últimos 30 dias** (mesma lógica de soma, janela curta). Documentar no card como **aproximação** (receita recorrente do mês corrente ≠ MRR contratual real, que exigiria estado de assinatura).

### Reuso

`services/encryption.ts` (encrypt/decrypt AES-256-GCM). Padrão de service: espelhar estrutura de `services/hotmart.ts` (seções: tipos, cripto, helpers de data, auth+cache, cliente HTTP, fetchers, agregação pura, orquestração).
