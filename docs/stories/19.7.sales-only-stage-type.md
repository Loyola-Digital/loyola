# Story 19.7 — Novo tipo de etapa "Vendas" (sales-only)

**Epic:** 19 — Funnel Stages
**Status:** Ready
**Priority:** Medium
**Estimate:** M
**Depende de:** 19.4 (stage type paid/free), 19.5 (stage_sales_spreadsheets), 19.6 (stage financial dashboard)

---

## User Story

Como gestor do Loyola X, quero criar uma etapa do tipo **"Vendas"** dentro de um funil (em adição aos tipos atuais "Gratuita" e "Paga"), para registrar uma planilha de vendas avulsa (ex: Produto Principal vendido fora do funil de captação) e ver um dashboard focado **só em vendas** — sem aba de tráfego/leads — com cards de total e faturamento, agrupamentos Por Canal de Origem e Por Medium (Adset), e gráfico de vendas por dia.

---

## Contexto Técnico

### Modelo atual (não muda)

- `funnel_stages.stage_type` enum: `"free" | "paid"`
- Etapa `paid` tem 2 planilhas via `stage_sales_spreadsheets` com `subtype: "capture" | "main_product"` (UNIQUE em `(stage_id, subtype)`)
- Dashboard de etapa paga renderiza tráfego (Meta Ads) **+** 2 sections de vendas (capture + main_product) — ver `launch-dashboard.tsx` e `perpetual-dashboard.tsx`

### Mudança proposta

- Adicionar valor `"sales"` ao enum `stage_type` (3º tipo)
- Reusar `stage_sales_spreadsheets` com novo `subtype: "sales"` — 1 planilha por stage (UNIQUE constraint isola por `stage_id`, sem misturar dados)
- Etapa `sales` renderiza dashboard **simplificado** — só vendas, sem aba de tráfego
- StageCard exibe rótulo/ícone diferente pra etapa Vendas

### Por que reutilizar `stage_sales_spreadsheets`

- 90% da lógica de mapping + leitura + agregação (`/sales-data`) já existe e funciona pra `subtype="capture"` e `"main_product"`
- Adicionar `"sales"` ao subtype reusa rota, hook e wizard sem duplicar
- Isolamento por `stage_id` garante que dados não se misturam entre etapas (UNIQUE `(stage_id, subtype)` + cada etapa tem `stage_id` próprio)

---

## Acceptance Criteria

1. **AC1 — Schema:** Migration adiciona `'sales'` ao enum `funnel_stages.stage_type` (PostgreSQL `ALTER TYPE ADD VALUE`). Drizzle schema atualizado.
2. **AC2 — Dialog "Nova Etapa":** O dialog em `packages/web/app/(app)/projects/[id]/funnels/[funnelId]/page.tsx` mostra **3 opções** (Gratuita / Paga / **Vendas**) em vez de 2. A opção Vendas tem hint claro: "Apenas planilha de vendas, sem tráfego pago".
3. **AC3 — Criar etapa sales:** Backend (`/api/projects/.../stages` POST) aceita `stageType: "sales"`. Etapa criada aparece no grid de StageCards do funil junto com as outras (mesmo local — não em seção separada).
4. **AC4 — Card visual:** `StageCard` mostra label "Vendas" + ícone distinto (ex: 💰 ou `<DollarSign>`) quando `stage.stageType === "sales"`.
5. **AC5 — Página da stage (sales):** Quando `stage.stageType === "sales"`, a página `/projects/[id]/funnels/[funnelId]/stages/[stageId]` renderiza:
   - **Sem** componentes de tráfego/Meta Ads (sem `LaunchDashboard`/`PerpetualDashboard` da aba tráfego, sem campanhas vinculadas, sem leads/qualificação)
   - **Com** uma única seção `StageSalesSpreadsheetSection` com `subtype="sales"` (botão "Conectar planilha", reconectar, desconectar) — reutilizando wizard `StageSalesWizardDialog`
   - **Com** um `StageSalesSection` (também `subtype="sales"`) renderizando: cards (Total Vendas, Faturamento Bruto, Ticket Médio), tabela "Por Canal de Origem", tabela "Por Medium (Adset)" (resolvendo utm_medium → adset_name via `useFunnelAdsetsMap`), tabela "Por Forma de Pagamento", e gráfico de vendas por dia (`StageSalesByDayChart` ou similar).
6. **AC6 — Sem "Por Content (Ad)":** A tabela "Por Content (Ad)" NÃO deve ser renderizada na etapa sales (rate limit Meta API, removida em PR #109/#110).
7. **AC7 — `subtype="sales"` no backend:** Rota `GET /api/projects/.../stages/.../sales-data` aceita `subtype="sales"` e retorna o mesmo shape (`StageSalesData`) que pra `capture`/`main_product`. Validação Zod do query schema atualizada.
8. **AC8 — Wizard de mapping:** Wizard `StageSalesWizardDialog` aceita `subtype="sales"` e persiste o mapping em `stage_sales_spreadsheets` com esse subtype.
9. **AC9 — Isolamento de dados:** Criar 2 etapas `sales` no mesmo funil → cada uma conecta uma planilha diferente → os dashboards não se misturam (cada um lê só do próprio `stage_id`).
10. **AC10 — Sem regressão:** Etapas existentes (`free` e `paid`) continuam funcionando sem alteração visual ou de comportamento.

## Scope

**IN:**
- Migration que adiciona `'sales'` ao enum `stage_type`
- Drizzle schema atualizado (e tipos compartilhados em `packages/shared/src/types/funnel.ts`)
- 3º botão no dialog "Nova Etapa"
- Lógica de renderização condicional na página da stage pra `stage_type="sales"` (dashboard simplificado)
- StageCard com visual pra "Vendas"
- Backend aceita `subtype="sales"` em `/sales-data` e em `/stage-sales-spreadsheets` (criar/listar/desconectar)
- Wizard `StageSalesWizardDialog` aceita prop `subtype="sales"`

**OUT:**
- Mudanças no modelo (não cria tabela nova, só reusa `stage_sales_spreadsheets`)
- Novo dashboard de tráfego pago dentro de etapa sales
- Tabela "Por Content (Ad)" (foi removida por rate limit)
- Cruzamento com leads de outra etapa
- Suporte a múltiplas planilhas por etapa sales (1 por stage é suficiente)

## Risks

- **Migration enum:** `ALTER TYPE ADD VALUE` em Postgres é irreversível direto — se houver erro, rollback exige criar novo enum. Risco baixo.
- **Mapping confuso:** UI pode confundir o usuário se o botão "Vendas" não tiver hint claro distinguindo de "Paga" (que também tem planilha de vendas). Resolver com texto descritivo no botão.
- **Stage sales sem planilha conectada:** dashboard mostra empty state — não pode quebrar render.
