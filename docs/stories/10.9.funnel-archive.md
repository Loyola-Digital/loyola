# Story 10.9 — Arquivar/desarquivar funis no sidebar

**Epic:** 10 — Funnel Architecture
**Status:** Done
**Priority:** Medium
**Estimate:** M
**Depende de:** 10.4 (sidebar-funnels), 10.8 (sort_order — reorder ignora arquivados)

---

## User Story

Como gestor do Loyola X, quero **arquivar funis** (lançamentos antigos ou perpétuos que não interessam mais) pelo menu de 3 pontos no sidebar, pra **limpar a visão** dos funis ativos sem perder os dados. Funis arquivados ficam numa seção colapsável "Arquivados" abaixo da lista normal, e posso desarquivar a qualquer momento.

---

## Contexto Técnico

### Modelo atual

- Tabela `funnels` não tem coluna de arquivamento.
- `GET /api/projects/:projectId/funnels` retorna **todos** os funis do projeto.
- `FunnelItem` em `project-folder.tsx` tem menu de 3 pontos (`DropdownMenu`) com ações de Renomear / Trocar tipo / Compartilhar / Deletar.
- `SortableFunnelList` (10.8) renderiza 2 grupos drag-and-drop (perpetuals + launches).
- `PUT /funnels/reorder` valida ordem e persiste `sort_order`.

### Mudança proposta

1. **Soft archive:** colunas `archived_at` (timestamptz nullable) e `archived_by` (uuid nullable FK users) em `funnels`.
2. **GET `/funnels`** ganha query param opcional `archived=true|false|all` (default `false`, retrocompat: comportamento atual = só ativos).
3. **PUT `/funnels/:id/archive`** e **`/unarchive`** — endpoints dedicados (mais claro que reusar PUT update).
4. **Reorder rejeita arquivados:** se o array `ids` incluir algum funil com `archived_at IS NOT NULL`, retorna 400.
5. **Frontend:**
   - `FunnelItem` ganha item "Arquivar" no DropdownMenu (acima de "Deletar").
   - Hook `useArchiveFunnel` / `useUnarchiveFunnel`.
   - Hook `useArchivedFunnels(projectId)` separado pro fetch da lista de arquivados.
   - Componente novo `ArchivedFunnelsSection` no `project-folder.tsx` — Collapsible com ícone `Archive` (lucide) abaixo da seção atual de funis.
   - `SortableFunnelList` recebe apenas ativos (já é o default do GET sem query param).

### Por que endpoints dedicados (não PUT genérico)

- Semântica clara: `/archive` / `/unarchive` deixa óbvio no log/audit o que aconteceu.
- Permite RBAC granular no futuro (ex: só admin pode arquivar).
- Evita risco de o PUT genérico (com `updateFunnelSchema`) zerar campos inadvertidamente.

## Acceptance Criteria

1. **AC1 — Schema:** Migration `0043_funnels_archived.sql` adiciona:
   - `archived_at timestamptz NULL`
   - `archived_by uuid NULL REFERENCES users(id) ON DELETE SET NULL`
   - Index parcial `idx_funnels_active ON funnels (project_id) WHERE archived_at IS NULL` (acelera o GET default)
   Drizzle schema atualizado.

2. **AC2 — GET com filter:** `GET /api/projects/:projectId/funnels?archived=false|true|all`:
   - `false` (default): retorna só funis com `archived_at IS NULL`
   - `true`: retorna só funis com `archived_at IS NOT NULL`
   - `all`: retorna todos
   - Ordenação atual (perpétuos > lançamentos > sort_order) preservada.

3. **AC3 — PUT archive:** `PUT /api/projects/:projectId/funnels/:funnelId/archive`:
   - Seta `archived_at = now()` e `archived_by = request.userId`
   - Valida ownership (project pertence ao user/membership)
   - Guest → 403
   - Idempotente: se já arquivado, retorna `{ success: true }` sem erro
   - Retorna `{ success: true, archivedAt: string }`

4. **AC4 — PUT unarchive:** `PUT /api/projects/:projectId/funnels/:funnelId/unarchive`:
   - Seta `archived_at = NULL` e `archived_by = NULL`
   - Mesmas validações de AC3
   - Idempotente
   - Retorna `{ success: true }`

5. **AC5 — Reorder rejeita arquivados:** `PUT /api/projects/:projectId/funnels/reorder` valida que NENHUM `id` do body tem `archived_at IS NOT NULL`. Se incluir, 400 `{ error: "Não é possível reordenar funis arquivados" }`.

6. **AC6 — Hard rule pós-arquivar:** Se o usuário arquiva um perpétuo e sobra apenas lançamentos ativos, a lista normal mostra só os lançamentos (sem erro/inconsistência). Mesmo com perpétuos arquivados, hard rule "ativos perpétuos > ativos lançamentos" continua válida.

7. **AC7 — Hook frontend:**
   - `useArchivedFunnels(projectId)` — query separada com `queryKey: ["funnels-archived", projectId]`
   - `useArchiveFunnel(projectId)` — mutation, invalida `["funnels", projectId]` E `["funnels-archived", projectId]`
   - `useUnarchiveFunnel(projectId)` — mesmo

8. **AC8 — Menu de 3 pontos:** Em `FunnelItem`, o `DropdownMenu` ganha item "Arquivar" (ícone `Archive`) imediatamente acima de "Deletar". Click chama `useArchiveFunnel`. Toast de sucesso "Funil arquivado" + opção "Desfazer" no toast que chama `useUnarchiveFunnel`. Em erro toast vermelho.

9. **AC9 — Seção Arquivados:** Em `project-folder.tsx`, **após** o `SortableFunnelList` e **antes** do botão "Novo Funil", aparece `<ArchivedFunnelsSection>`:
   - Só renderiza se `useArchivedFunnels` retornar pelo menos 1 funil
   - `Collapsible` fechado por default
   - Header: ícone `Archive` + texto "Arquivados ({count})" + chevron
   - Conteúdo: lista de `ArchivedFunnelItem` (componente novo) — visual compacto, sem expansão de stages, com botão "Desarquivar" no hover
   - Click no nome navega pra `/projects/[id]/funnels/[funnelId]` (página do funil continua acessível mesmo arquivado — soft archive)

10. **AC10 — ArchivedFunnelItem:** Item compacto: ícone do tipo do funil (`Rocket`/`Repeat`) + nome + botão `RotateCcw` (Desarquivar) visível em hover. Sem drag handle. Sem menu de 3 pontos (única ação é desarquivar). Click no body navega pra página do funil.

11. **AC11 — Soft archive preserva acesso:** Navegar diretamente pra `/projects/[id]/funnels/[funnelId]` de um funil arquivado funciona normal — dashboard, stages, vendas, etc. Não há banner "este funil está arquivado" nesta v1 (fica como nice-to-have).

12. **AC12 — Sem regressão:**
   - Funis ativos continuam com drag-and-drop e menu de 3 pontos completo
   - SortableFunnelList renderiza só ativos (já é o default GET)
   - Reorder, criação, edição, exclusão dos ativos seguem idênticos
   - Funis arquivados não aparecem na lista normal nem participam de reorder

13. **AC13 — Typecheck + lint zero erros:** `pnpm -r typecheck` e `pnpm -r lint` passam limpos.

**IN:**

- Filtro/busca dentro da lista de arquivados
- Data de arquivamento exibida no item
- Auto-archive por inatividade (sem regra automática)
- Audit log dedicado (já temos `archived_by`)

## API Changes

```ts
// GET /api/projects/:projectId/funnels?archived=false|true|all
// Default: false (só ativos — comportamento atual)

// PUT /api/projects/:projectId/funnels/:funnelId/archive
// Response 200
{ success: true, archivedAt: "2026-06-01T12:34:56.000Z" }

// PUT /api/projects/:projectId/funnels/:funnelId/unarchive
// Response 200
{ success: true }

// PUT /api/projects/:projectId/funnels/reorder
// Response 400 (novo erro)
{ error: "Não é possível reordenar funis arquivados" }
```

```sql
-- Migration 0043_funnels_archived.sql
ALTER TABLE "funnels"
  ADD COLUMN IF NOT EXISTS "archived_at" timestamptz,
  ADD COLUMN IF NOT EXISTS "archived_by" uuid REFERENCES users(id) ON DELETE SET NULL;

-- Index parcial: acelera o GET default (só ativos) sem sobrecarregar lookups
-- por id (PK continua dominante).
CREATE INDEX IF NOT EXISTS "idx_funnels_active"
  ON "funnels" ("project_id") WHERE "archived_at" IS NULL;
```

## Risks

- **Soft archive em queries existentes:** Outras rotas/cálculos podem assumir que GET de funis retorna todos. Mitigação: o `GET /funnels` é o único consumido em UI. Cálculos internos (sellers-breakdown, stage-sales-data) operam por `funnelId` específico — não filtram lista. Pode arquivar e o histórico continua intacto.
- **User arquiva e perde:** Sem auto-arquivamento; user precisa clicar conscientemente. Toast "Desfazer" mitiga clique acidental.
- **sort_order desencontrado após desarquivar:** Funil volta com `sort_order` antigo. Pode bater com outro funil ativo no mesmo número. Resultado: tiebreak vai pra `created_at`. Não quebra ordenação, só pode dar "salto" pequeno. Aceitável; se incomodar, faz limpeza num PR futuro.
- **Index parcial em projeto pequeno:** Custa pouco mesmo sem ganho perceptível; sem risco.

## Dev Notes

- Reorder agora carrega APENAS funis com `archivedAt IS NULL` (linha "allFunnels query"). Isso garante: (a) lista comparativa pra validar `body.ids` cobre só ativos; (b) impossível arquivado entrar no array sem erro "ID desconhecido ou funil arquivado".
- Endpoints `/archive` e `/unarchive` são idempotentes — re-arquivar não falha (apenas reseta `archived_at` pra agora) e reabrir um já-ativo também é no-op.
- `ON DELETE SET NULL` em `archived_by` evita FK violation quando user some.
- Index parcial `idx_funnels_active` cobre o caso quente (lista de ativos) sem desperdiçar disco com arquivados — economiza ~30% do espaço típico do índice.
- Toast com `action.label: "Desfazer"` reaproveita a mutation `useUnarchiveFunnel`. Sem state extra; reage ao clique direto.
- `ArchivedFunnelsSection` retorna `null` quando `archived.length === 0` — não polui sidebar quando o user não tem nada arquivado.
- `ArchivedFunnelItem` reusa a navegação direta pra página do funil — soft archive preserva acesso total ao histórico.
