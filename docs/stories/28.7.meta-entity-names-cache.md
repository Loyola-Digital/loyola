## User Story

Como gestor de tráfego do Loyola, quando olho o dashboard de vendas de uma etapa **Vendas/Captação Paga**, quero ver as tabelas **Por Adset** (utm_term → adset_name), **Por Content (Ad)** (utm_content → ad_name) e **Por Medium** com os **nomes legíveis** em vez de IDs crus tipo `120241432925400`. Hoje só "Por Medium" funciona; as outras 2 foram removidas em 06/05 (commit `cfeef00`) porque estouravam o rate limit da Meta API ao tentar resolver 500+ ad_ids por refresh.

A tabela "Por Medium" sobreviveu porque usa `useFunnelAdsetsMap` → `/all-adsets` (1 request batch, cache 15min na memória do servidor). As outras 2 não tinham cache equivalente.

### Por que o cache em memória não bastou

- `useFunnelAdsetsMap` cacheia 15min → expira → próximo refresh refaz request → cada projeto/funil é um cache key separado → quando vários gestores abrem dashboards simultaneamente, o cache thrasha
- Ad names são MUITO estáveis (criativo raramente muda de nome após criado) — TTL 15min é desperdício
- **Solução estrutural:** persistir em PostgreSQL com TTL longo (24h) + lookup-first antes de bater Meta

## Comportamento desejado

### Camada 1 — Cache persistente

Nova tabela `meta_entity_names_cache`:

```sql
CREATE TABLE meta_entity_names_cache (
  project_id   UUID NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  entity_type  VARCHAR(20) NOT NULL CHECK (entity_type IN ('ad','adset','campaign')),
  entity_id    VARCHAR(64) NOT NULL,
  entity_name  VARCHAR(500) NOT NULL,
  last_synced_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  PRIMARY KEY (project_id, entity_type, entity_id)
);
CREATE INDEX idx_meta_names_cache_lookup ON meta_entity_names_cache (project_id, entity_type, last_synced_at);
```

### Camada 2 — Helper de resolução

`resolveEntityNames(projectId, type, ids[], options?)` em `meta-ads.ts`:

1. **Query cache** — busca `entity_id IN ids` da tabela acima onde `last_synced_at > NOW() - 24h`
2. **Identifica faltantes** — `missingIds = ids - cached.keys()`
3. **Batch Meta API** — para faltantes, chama Meta `level={type}&fields=id,name` em lotes de **max 100 ids/request** (limite oficial Meta batch API)
4. **Throttle** — entre requests, sleep mínimo 200ms pra ficar bem abaixo de 200 req/h
5. **Upsert no cache** — `INSERT … ON CONFLICT … DO UPDATE`
6. **Retorna** — `Map<id, name>` completo (cache + recém-resolvido). Ids que falharam silenciosos viram fallback `id` literal

### Camada 3 — Rota atualizada

`stage-sales-data.ts` ao montar response:
- Coleta todos os utm_term únicos → `resolveEntityNames(projectId, 'adset', termIds)`
- Coleta todos os utm_content únicos → `resolveEntityNames(projectId, 'ad', contentIds)`
- utm_medium continua usando `useFunnelAdsetsMap` (já agregado), OU migra também pra usar o cache (decisão do dev — manter compat por enquanto)
- Adiciona campo `name` em cada item de `porUtmTerm` e `porUtmContent` (nome resolvido ou fallback id)

### Camada 4 — Frontend restaura tabelas

`stage-sales-section.tsx` renderiza 5 tabelas em vez de 3:
1. Por Canal (existente)
2. Por Medium (Adset) (existente, com nome ainda via `adsetsMap`)
3. **Por Term (Adset)** ← restaurada — usa `data.porUtmTerm[i].name`
4. **Por Content (Ad)** ← restaurada — usa `data.porUtmContent[i].name`
5. Por Forma de Pagamento (existente)

Tabelas 3 e 4 mostram **label = name** quando resolvido, **label = id (cru)** com badge "⚠️ não resolvido" quando fallback.

## Acceptance Criteria

### AC1 — Schema + Cache
- **AC1.1:** Migration cria tabela `meta_entity_names_cache` com PK composto e index
- **AC1.2:** `entity_type` aceita apenas `'ad'|'adset'|'campaign'` (CHECK constraint)
- **AC1.3:** Cascade em `project_id` — apaga cache quando projeto for deletado

### AC2 — Helper de resolução
- **AC2.1:** `resolveEntityNames(projectId, 'ad', [id1, id2, id3])` retorna `Map<id, name>` com todos os 3 keys (mesmo que algum falhe — fallback é o próprio id)
- **AC2.2:** Cache hit não chama Meta API (verificável via log: `[meta-cache] hit X, miss Y`)
- **AC2.3:** Batch Meta API limita a **100 ids/request**
- **AC2.4:** Entre batches, sleep ≥ 200ms (throttle)
- **AC2.5:** Failure parcial graceful — 1 batch falha não derruba outros; ids do batch que falhou retornam fallback id

### AC3 — Stage Sales Data
- **AC3.1:** Response inclui `porUtmTerm[i].name` e `porUtmContent[i].name` populados quando cache/API resolveu
- **AC3.2:** Quando id não resolveu, `name === id` (fallback explícito, sem null)
- **AC3.3:** Performance: dashboard de funil com 200 ads únicos resolve em < 3s no primeiro load, < 200ms nos seguintes (cache hit)

### AC4 — Frontend
- **AC4.1:** Tabela "Por Term (Adset)" aparece no `stage-sales-section.tsx` (3ª tabela)
- **AC4.2:** Tabela "Por Content (Ad)" aparece (4ª tabela)
- **AC4.3:** Quando `item.name === item.id` (não resolvido), exibe badge "⚠️ não resolvido" ao lado
- **AC4.4:** Tabelas continuam ordenadas por `vendas DESC`

### AC5 — Sem regressão
- **AC5.1:** Tabela "Por Medium" existente continua funcionando idêntico
- **AC5.2:** Funis sem `utm_term`/`utm_content` mapeado mostram empty state nas tabelas novas (sem erro)
- **AC5.3:** Rate limit Meta API NÃO estoura — log do servidor deve mostrar < 50 req Meta em 1h de uso normal

### Helper resolveEntityNames

```ts
const CACHE_TTL_MS = 24 * 60 * 60 * 1000; // 24h
const BATCH_LIMIT = 100;
const BATCH_THROTTLE_MS = 200;

export async function resolveEntityNames(
  db: DbInstance,
  projectId: string,
  type: 'ad' | 'adset' | 'campaign',
  ids: string[],
  accessToken: string,
): Promise<Map<string, string>> {
  const result = new Map<string, string>();
  if (ids.length === 0) return result;
  const unique = Array.from(new Set(ids));

  // 1. Query cache válido (TTL 24h)
  const cutoff = new Date(Date.now() - CACHE_TTL_MS);
  const cached = await db.select(...).from(metaEntityNamesCache).where(...);
  for (const c of cached) result.set(c.entityId, c.entityName);

  // 2. Faltantes
  const missing = unique.filter((id) => !result.has(id));
  if (missing.length === 0) return result;

  // 3. Batch Meta API com throttle
  const rows: Array<{ entity_id: string; entity_name: string }> = [];
  for (let i = 0; i < missing.length; i += BATCH_LIMIT) {
    const batch = missing.slice(i, i + BATCH_LIMIT);
    try {
      const meta = await fetchMeta<{ data: Array<{ id: string; name: string }> }>(
        `/?ids=${batch.join(',')}&fields=id,name`,
        accessToken,
      );
      for (const m of meta.data ?? []) {
        result.set(m.id, m.name);
        rows.push({ entity_id: m.id, entity_name: m.name });
      }
    } catch (err) {
      // Graceful — batch falhou, ids dele caem em fallback abaixo
      console.error('[resolveEntityNames] batch failed', err);
    }
    if (i + BATCH_LIMIT < missing.length) await sleep(BATCH_THROTTLE_MS);
  }

  // 4. Upsert no cache
  if (rows.length > 0) {
    await db.insert(metaEntityNamesCache)
      .values(rows.map((r) => ({ projectId, entityType: type, ...r })))
      .onConflictDoUpdate(...);
  }

  // 5. Fallback pra ids que não resolveram
  for (const id of unique) if (!result.has(id)) result.set(id, id);

  return result;
}
```

## Dev Notes

- **Migration manual:** seguir padrão do projeto via `node packages/api/scripts/run-migrations.js` (ou similar — confirmar com arquivo `0033_zoom_cached_data.sql` como referência)
- **Throttle 200ms:** conservador. Meta permite até 200 req/h, batch de 100 ids = 1 req. Pra 500 ids = 5 reqs = 1s total. Margem MUITO grande
- **`level=ad` vs `/?ids=`:** o endpoint `/?ids=id1,id2,...&fields=id,name` aceita até 50 ids em ALGUMAS contas. Se for o caso, reduzir `BATCH_LIMIT` pra 50 e testar. **Atenção memory rule:** pesquisar Meta API docs antes de inventar parâmetro
- **utm_medium hoje usa adsetsMap (in-memory):** manter por enquanto, refactor pode vir em story separada se quiser unificar
