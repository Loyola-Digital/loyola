## 📖 Story

**Como** desenvolvedor da plataforma Loyola Digital X,
**Quero** as tabelas `kiwify_connections` (credenciais OAuth2 criptografadas por projeto) e `kiwify_cache` (cache SWR do dashboard) declaradas no schema Drizzle, com migrations versionadas,
**Para** que as stories seguintes do Epic 35 (service, rotas, frontend) tenham onde persistir, com segurança, o `client_id`, `client_secret` e `account_id` da Kiwify de cada projeto e o payload agregado do dashboard.

---

## ✅ Acceptance Criteria

1. **`kiwifyConnections` declarada no schema Drizzle** (`packages/api/src/db/schema.ts`), espelhando `hotmartConnections`, com colunas:
   - `id` uuid PK default `gen_random_uuid()` NOT NULL
   - `project_id` uuid NOT NULL UNIQUE, FK → `projects(id)` ON DELETE cascade
   - `client_id_encrypted` text NOT NULL · `client_id_iv` text NOT NULL
   - `client_secret_encrypted` text NOT NULL · `client_secret_iv` text NOT NULL
   - `account_id_encrypted` text NOT NULL · `account_id_iv` text NOT NULL
   - `created_at` / `updated_at` timestamptz default `now()` NOT NULL
   - Index `idx_kiwify_connections_project` em `(project_id)`.

2. **`kiwifyCache` declarada no schema Drizzle**, espelhando `hotmartCache`:
   - `project_id` uuid NOT NULL · `cache_key` varchar(200) NOT NULL · `data` jsonb NOT NULL · `computed_at` timestamptz default `now()` NOT NULL
   - PK composto `(project_id, cache_key)` · FK `project_id` → `projects(id)` ON DELETE cascade.

3. **Migration `0054_kiwify_connection.sql`** criada (espelha `0051_hotmart_connection.sql`): CREATE TABLE IF NOT EXISTS + UNIQUE inline + FK com guard idempotente (`information_schema.table_constraints`) + CREATE INDEX IF NOT EXISTS + `--> statement-breakpoint` entre statements. FK nomeada `kiwify_connections_project_id_projects_id_fk`.

4. **Migration `0055_kiwify_cache.sql`** criada (espelha `0053_hotmart_cache.sql`): CREATE TABLE IF NOT EXISTS com PK composto + FK com guard idempotente. FK nomeada `kiwify_cache_project_id_projects_id_fk`.

5. **Reuso de cripto documentado** (sem novo código de cripto): Dev Notes referencia `services/encryption.ts`. `account_id` é criptografado por **uniformidade do padrão** (não é segredo, mas mantém schema/service simétricos); o token JWT é derivado em runtime e NUNCA armazenado.

6. **Migrations aplicadas e verificadas** no banco de dev via `node scripts/run-migration.mjs src/db/migrations/0054_kiwify_connection.sql` e `...0055_kiwify_cache.sql` — sucesso + tabelas/index/FK confirmados via `information_schema`/`pg_indexes`.

7. **Qualidade:** `npm run typecheck` e `npm run lint` passam na API após editar o schema.

## 🔧 Dev Notes

### Contrato de dados (nomes EXATOS — não inventar)

`kiwify_connections`: ver AC1. `kiwify_cache`: ver AC2. `cache_key` = `"dashboard:<productId>:<months>"` ou `"products:<months>"`.

> A Kiwify usa OAuth2 `client_credentials` → JWT (expira 24h). Toda chamada exige header `x-kiwify-account-id: <account_id>`. Guardamos `client_id`, `client_secret` e `account_id` criptografados (+IV); o **token NÃO é persistido** (cache em memória no service, story 35.2).

### Declaração Drizzle — espelhar `hotmartConnections` e `hotmartCache` (schema.ts)

Mesmo formato do bloco Hotmart (Story 34.1/34.x), trocando `hotmart`→`kiwify` e adicionando o par `account_id_encrypted`/`account_id_iv`. Imports já presentes (`pgTable, uuid, text, timestamp, index, varchar, jsonb, primaryKey`).

### Migrations — espelhar `0051_hotmart_connection.sql` (connection) e `0053_hotmart_cache.sql` (cache)

`0054_kiwify_connection.sql`: CREATE TABLE com as 3 pares encrypted/iv + UNIQUE("project_id") inline + bloco DO $$ de FK idempotente + CREATE INDEX `idx_kiwify_connections_project`. `0055_kiwify_cache.sql`: CREATE TABLE com PK composto `("project_id","cache_key")` + bloco DO $$ de FK idempotente. `--> statement-breakpoint` entre statements.

### Segurança

NUNCA commitar/logar `client_secret` nem o token. Em dev, credenciais só no `.env` (gitignored). Colunas guardam apenas valores criptografados.
