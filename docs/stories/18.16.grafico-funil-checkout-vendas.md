## Story

Como gestor de tráfego do Loyola X, quero visualizar no **gráfico de funil de conversão** da aba Meta Ads as etapas de **Visitas do Checkout** e **Vendas** (com suas taxas de conversão entre etapas), para entender completamente o caminho da lead até a venda final.

## Contexto

### Problema atual

Hoje o gráfico de funil (Story 18.4) mostra 4 etapas: Impressões → Link Clicks → Landing Page Views → Leads Únicos.

Mas o funil é incompleto: não mostra o que acontece **após a lead chegar** — quantos levam ao checkout e quantos viram vendas efetivas.

O gestor de tráfego quer ver o **funil completo**: Impressões → Cliques → Landing Page → Leads → **Checkout** → **Vendas**, com taxas de conversão entre cada etapa.

### Solução proposta

Estender o gráfico de funil para incluir 2 etapas adicionais:
- **5ª etapa: Visitas do Checkout** (number of checkout landing_page_views)
- **6ª etapa: Vendas** (number of sales from planilha de captação paga)

E renderizar as **taxas de conversão** entre cada etapa no mesmo padrão visual (ex: "15.5%", "8.2%", etc.).

### Decisão arquitetural

Estender o componente de gráfico de funil existente (Story 18.4) para aceitar dados de checkout e vendas. Frontend-only. Usar dados:
- Checkout: Meta Ads API (`landing_page_view`)
- Vendas: Planilha de Captação Paga
- Taxas: Calculadas como `(etapa_N+1 ÷ etapa_N) × 100`

## Escopo

### IN
- Estender gráfico de funil com 2 novas etapas: **Checkout** e **Vendas**
- Ordem das etapas: Impressões → Link Clicks → Landing Page → Leads → **Checkout** → **Vendas**
- Exibir taxa de conversão **entre cada etapa** (ex: "Leads → Checkout: 15.5%", "Checkout → Vendas: 8.2%")
- Respeitar filtro `DayRangePicker` (mesmo `days` para todas etapas)
- Usar mesmo visual pattern do gráfico existente (barras, cores, labels)
- Se planilha captação não vinculada, renderizar etapas até Leads normalmente; Checkout e Vendas exibem "—" com tooltip

### OUT
- Alterações em outras etapas do funil (Impressões, Cliques, LP Views, Leads) — mantém atual
- Mudanças em cards KPI
- Configuração de eventos de tracking novos
- Mudanças em backend ou estrutura de dados

## Dados e Fórmulas

### Etapas do Funil

```
Etapa 1: Impressões
  Fonte: Meta Ads API
  Valor: Σ impressions

Etapa 2: Link Clicks
  Fonte: Meta Ads API
  Valor: Σ actions[action_type="link_click"]

Etapa 3: Landing Page Views
  Fonte: Meta Ads API
  Valor: Σ actions[action_type="landing_page_view"] (todas)

Etapa 4: Leads Únicos
  Fonte: Planilha de Leads
  Valor: count(total_leads)

Etapa 5: Visitas Checkout
  Fonte: Meta Ads API
  Valor: Σ actions[action_type="landing_page_view"] (checkout pixel)

Etapa 6: Vendas
  Fonte: Planilha de Captação Paga
  Valor: count(where "Venda ingressos" = 1)
```

### Taxas de Conversão Entre Etapas

```
Tx. Impressões → Cliques    = (Link Clicks ÷ Impressões) × 100
Tx. Cliques → LP Views      = (LP Views ÷ Link Clicks) × 100
Tx. LP Views → Leads        = (Leads ÷ LP Views) × 100
Tx. Leads → Checkout        = (Visitas Checkout ÷ Leads) × 100
Tx. Checkout → Vendas       = (Vendas ÷ Visitas Checkout) × 100

Divisão por zero → "—"
```

### Período Filtrado
- Aplicar `DayRangePicker` a todas as etapas (Meta API e Planilhas)

## Riscos Técnicos

| Risco | Severidade | Impacto | Mitigação |
|-------|-----------|--------|-----------|
| **Evento landing_page_view checkout indisponível/mapeado incorretamente** | High | Etapa 5 (Checkout) com valores errados, taxa Leads→Checkout incorreta | Validar mapeamento do pixel de checkout; se indisponível, propor alternativa (ex: pixel de página de sucesso) |
| **Performance com 6 etapas + 5 taxas** | Medium | Gráfico lento ou não renderiza em datasets grandes | Implementar memoization; considerar lazy-loading de dados; testar com 6 meses de dados |
| **Planilha de captação não vinculada** | Medium | Etapas 5-6 exibem "—", funil incompleto | Comportamento esperado (AC4); tooltip explicativo |
| **Filtro de data inconsistência entre sources** | High | Numeradores e denominadores em períodos diferentes, taxas incorretas | Garantir `DayRangePicker` aplica mesmo `days` a Meta API e ambas planilhas (feedback obrigatório) |
| **Divisão por zero em múltiplas taxas** | Low | Algumas taxas exibem "—", é comportamento esperado | Validado em AC3; implementar gracefully |
| **Refatoração do componente gráfico** | Medium | Quebra de visual consistency se não manter padrão existente | Reutilizar componente existente (18.4); preservar cores, fonts, spacing |

## Acceptance Criteria

### AC1: Novas etapas no gráfico
- [ ] Gráfico renderiza 6 etapas (não mais 4): Impressões, Cliques, LP, Leads, Checkout, Vendas
- [ ] Ordem das etapas corretas, da esquerda para direita
- [ ] Valores numéricos corretos para cada etapa

### AC2: Taxas de conversão
- [ ] Taxa entre cada etapa é exibida (ex: "15.5%" entre Leads e Checkout)
- [ ] Cálculo correto: `(etapa_N+1 ÷ etapa_N) × 100`
- [ ] Divisão por zero retorna "—" sem erro
- [ ] Taxas com 1 a 2 casas decimais

### AC3: Filtro de período
- [ ] Ao alterar `DayRangePicker`, todas as 6 etapas são recalculadas
- [ ] Meta API reflete novo período
- [ ] Planilhas filtradas por data no novo período

### AC4: Tratamento de dados incompletos
- [ ] Se planilha de captação não vinculada: etapas 1-4 normais, etapas 5-6 exibem "—" + tooltip
- [ ] Se Meta API indisponível: exibir aviso, não renderizar gráfico parcial

### AC5: Visual consistency
- [ ] Gráfico segue padrão visual do funil existente (cores, fontes, espaçamento)
- [ ] Labels de etapas claros e legíveis
- [ ] Taxas exibidas próximo aos "degraus" do funil ou em tooltip

### AC6: Documentação e accessibility
- [ ] Tooltips explicam cada etapa (fonte de dados)
- [ ] Tooltip em taxa de conversão mostra fórmula (ex: "Checkout → Vendas: 25 ÷ 200 = 12.5%")
