# Story 27.2 — Fix: dados Meta Ads do dia atual não aparecem na tabela diária

**Epic:** 27 — Anotações em Métricas
**Status:** Draft
**Priority:** High
**Estimate:** S

---

## User Story

Como gestor de tráfego do Loyola X, quero ver os dados de spend, cliques, impressões e leads pagos da Meta no **dia atual** assim que rodam, e não só no dia seguinte — pra acompanhar performance em tempo quase real (como Stract Ads / UTMfy fazem) e tomar decisões enquanto o dia ainda corre.

---

## Bug observado

Tabela "Dados diários" em 06/05/2026:

| Dia | Investimento | Cliques | Impressões | Leads pagos | Leads org | Sem track |
|---|---|---|---|---|---|---|
| 01-05/05 | R$ 52-322 | 44-207 | 3K-18K | 31-45 | 10-23 | 0-29 |
| **06/05** (hoje) | **R$ 0,00** | **0** | **0** | **17** | **9** | **2** |

Leads (planilha Google Sheets) chegam normais, mas Meta Ads retorna tudo zero pro dia atual.

---

## Causa raiz (investigada)

3 problemas combinados em `packages/api/src/services/meta-ads.ts` e `traffic-analytics.ts`:

### Problema 1 — Filtro descarta dia atual
`fetchCampaignDailyInsightsForIds` (linhas 363-366) e funções irmãs filtram dias com `impressions > 0`. Quando Meta ainda não populou o dia atual (lag normal de 30-60 min), o dia some inteiro:

```ts
const filtered = allResults.filter((d) => {
  const impressions = parseFloat(d.impressions || "0");
  return impressions > 0;  // ← derruba o dia atual
});
```

### Problema 2 — Cache de 15 min estagna
`traffic-analytics.ts:92` usa cache in-memory com TTL de 15 min para todas as queries. Quando a primeira chamada do dia volta vazia (Meta ainda processando), o resultado vazio fica cacheado por 15 min antes de tentar de novo. Combina mal com Problema 1.

### Problema 3 — Timezone UTC vs conta Meta
`dateRangeFromDays()` (linhas 30-37) calcula `since`/`until` em UTC. Meta API interpreta o range no timezone da conta (provavelmente America/Sao_Paulo). O comentário do próprio código admite o trade-off (linhas 25-28). Janela problemática: antes de 03:00 BRT, "hoje" UTC ainda é "ontem" pra Meta.

Tools como Stract Ads / UTMfy resolvem passando `time_range` JSON com `time_zone` explícito.

## Acceptance Criteria

1. **AC1 — Dia atual visível durante o dia**
   - Quando Meta tem dados parciais (1+ impressions) do dia atual, esses dados aparecem na tabela diária.
   - Quando Meta retorna 0 ainda, o dia aparece com R$ 0 / 0 impressões em vez de sumir.

2. **AC2 — Refresh agressivo pra hoje**
   - Cache server-side do dia atual: TTL ≤ 2 min.
   - Cache de dias passados (range que NÃO inclui hoje): mantém TTL atual de 15 min.

3. **AC3 — Timezone correto**
   - `since`/`until` enviados pra Meta calculados em `America/Sao_Paulo` (hardcoded — Loyola atende só clientes brasileiros).
   - `time_range` enviado como JSON com `time_zone` explícito (`{"since":"...","until":"...","time_zone":"America/Sao_Paulo"}`).
   - Funciona corretamente em qualquer hora do dia, incluindo madrugada (00:00-03:00 BRT).

4. **AC4 — Sem regressão em dias passados**
   - Tabela diária mostra os mesmos números pra dias 01-05/05/2026 que mostrava antes do fix.
   - Comparativo período anterior continua funcionando.

5. **AC5 — Sem regressão em outros dashboards**
   - `/api/traffic/analytics/*` (campaign-daily, campaign-daily-bulk, account, campaign, adset, ad-level)
   - LaunchDashboard, PerpetualDashboard
   - Cross-funnel daily table

## Dev Notes

### Por que `Intl.DateTimeFormat` com SP hardcoded em vez de chamar Meta `/act_{id}?fields=timezone_name`?
- 1 chamada extra por request = +200ms overhead
- Cache do TZ traria mais complexidade (invalidação?)
- Loyola atende só clientes brasileiros. SP cobre 100% dos casos com 0 custo.
- Se um dia precisar generalizar, migrar para coluna `timezone_name` em `meta_ads_accounts` é trivial (uma migration + buscar no validateMetaAdAccount).

### Por que `time_range` JSON em vez de `&since=&until=`?
Meta API aceita ambos, mas `time_range` permite passar `time_zone` explícito:
```
time_range={"since":"2026-05-06","until":"2026-05-06","time_zone":"America/Sao_Paulo"}
```
URL-encoded antes de inserir na URL.

### Cache TTL adaptativo
```ts
function getCacheTtl(includeToday: boolean): number {
  return includeToday ? 2 * 60 * 1000 : 15 * 60 * 1000;
}
```
`includeToday` = `true` quando `endDate || dateRangeFromDays(days).until` ≥ hoje no fuso da conta.
