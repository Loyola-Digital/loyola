## 📖 User Story

Como **gestor de tráfego**, quero que a tabela **"Desempenho de Testes de LPs"** mostre **uma linha por LP** (LPA, LPB, LPC, LPD) identificada pelo **Campaign Name da API Meta Ads**, com **LP View real** (Landing Page Views da API), **Connect Rate correto** e **contagem de Leads vinda da planilha n8n** — além de **filtros Hot/Cold temáticos** iguais aos de Criativos — para que eu **compare a performance real de cada LP por público**. E quero **ocultar Impressões/Cliques** na tabela de Criativos para reduzir ruído.

**Escopo:** 1 ajuste trivial em Criativos + reformulação fullstack da tabela de LPs (backend Meta Ads + frontend).

---

## 🔍 Complexity Estimate

**T-Shirt Size:** L (13-21 Story Points)
- **Drivers:** Mudança de fonte de identificação (utm_term → Campaign Name), parsing de `landing_page_view` no backend, exposição de novos campos no endpoint, nova agregação (1 linha por LP), filtros hot/cold efetivos
- **Estimated Timeline:** 3-5 dias (@dev)
- **Primary Complexity:** Backend (parsing de actions + campaign_name no endpoint) + reagregação no hook + filtros

## 📐 Scope

### IN (Inclusos)
- **Criativos:** ocultar colunas **Impressões** e **Cliques**
- **LPs:** tabela **única**, **1 linha por LP** (coluna "LP" em vez de "Dia"), sem título "LPA" no topo
- **LPs:** identificar a LP pelo **Campaign Name** da API (contém `lpa`/`lpb`/`lpc`/`lpd`, case-insensitive)
- **LPs:** **LP View** real = `landing_page_view` do array `actions` da API Meta Ads (backend)
- **LPs:** **Connect Rate** = LP View ÷ Link Clicks
- **LPs:** **Leads** = contagem da planilha n8n (`n8n-leads-lp-cap-grat`), por `utm_content` contendo `lpX`
- **LPs:** filtros **Hot/Cold/Todos** efetivos (Campaign Name + `utm_term`), com **botões temáticos** iguais aos de Criativos

### OUT (Exclusos)
- Granularidade diária na tabela de LPs (agora é agregado por LP, não por dia)
- Novas LPs além de LPA-LPD (o padrão `lp[a-z]` cobre futuras automaticamente, mas validação é só pra A-D)
- Mudança nas fórmulas de CPM/CPC/CTR/CPV/CPL/ROAS (mantidas)
- Exportação / edição inline

## ⚠️ Risks & Mitigations

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|-----------|
| Endpoint atual não expõe `campaign_name` nem `landing_page_view` | **High** | **High** | Estender a rota `stage-creative-performance.ts` para incluir os 2 campos; é o coração desta story |
| Campaign Name sem `lpX` em algumas campanhas | **Medium** | **Medium** | LP "não identificada" → agrupar em bucket "Outros"/ignorar + logar warning (confirmar com Danilo) |
| `landing_page_view` ausente em parte dos ads | **Medium** | **Medium** | Default 0 quando action não existe; Connect Rate trata divisor 0 → "—" |
| Identificação de Lead por `utm_content` vs `utm_term` | **Medium** | **High** | Usar `utm_content` (col 5) contendo `lpX` para atribuir LP do lead (ver AC6); validar com dados da Fernanda |
| Connect Rate > 100% (LP View > Clicks) | **Low** | **Low** | Possível por rastreamento; exibir valor real, sem clamp (apenas documentar) |
| Regressão na 18.45 (mesma branch) | **Low** | **Medium** | LPs: validar que reordenação Leads/CPL/Vendas/CPV da 18.45 continua coerente na nova tabela única |

### AC2: Tabela de LPs — Tabela única, 1 linha por LP

**Arquivos:** `packages/web/lib/components/funnels/lp-performance-table.tsx` + `packages/web/lib/hooks/useLpPerformanceData.ts` (+ componente que renderiza múltiplas tabelas)

- [x] Ordenação: por Investimento desc (ou alfabética por LP — confirmar; default: Investimento desc).

### AC3: Identificação da LP por Campaign Name

**Arquivos:** backend `packages/api/src/routes/stage-creative-performance.ts` + `packages/api/src/services/meta-ads.ts` + hook frontend

- [x] O endpoint deve **expor o `campaign_name`** de cada ad/criativo (hoje é buscado da API em `meta-ads.ts:700` mas não chega ao frontend).
- [x] A LP de cada gasto é identificada por **`campaign_name` contendo `lp[a-z]`** (case-insensitive). Ex: campanha "FZ - LPA - Hot" → **LPA**.
- [x] Agregar métricas (spend, impressions, clicks, landing_page_views) por LP.
- [x] Gasto cujo Campaign Name **não** contém nenhum `lpX` → **atribuir a LPA** (decisão Danilo 2026-06-12). Ou seja, LPA é o bucket default quando não há identificação explícita.

### AC4: LP View real (Landing Page Views da API Meta Ads) — BACKEND

**Arquivos:** `packages/api/src/services/meta-ads.ts` + `packages/api/src/routes/stage-creative-performance.ts`

- [x] Parsear o array `actions` retornado pela API (já buscado em `meta-ads.ts:700`) e extrair `action_type === 'landing_page_view'` → valor numérico.
- [x] Expor esse valor por ad/criativo no endpoint como `landingPageViews` (ou `lpViews`).
- [x] No hook `useLpPerformanceData`, substituir o hardcode `lpViews += 0` (linha ~140) pela soma do valor real por LP.
- [x] Quando a action não existir para um ad, considerar 0.
- [x] **Referência de parsing:** padrão similar em `packages/api/src/services/traffic-analytics.ts` (`parseLeadsFromActions`).

### AC5: Connect Rate = LP View ÷ Link Clicks

**Arquivo:** `packages/web/lib/utils/lp-metrics-calculator.ts` (função `calculateConnectRate`)

- [x] Alterar a fórmula de Connect Rate para **`(LP View ÷ Link Clicks) × 100`**.
- [x] Tratar divisor 0 (Link Clicks = 0) → exibir "—".
- [x] **Nota:** hoje Connect Rate = conversões ÷ cliques, e `conversoes = clicks`, resultando em ~100% sempre (bug). Esta AC corrige isso.
- [x] Pode resultar > 100% (LP View > Clicks por rastreamento) — exibir valor real, sem clamp.

### AC6: Leads via planilha n8n (utm_content)

**Arquivos:** `packages/web/lib/hooks/useCrossReferenceLeads.ts` + `useLpPerformanceData.ts`

- [x] Contar Leads a partir da aba **`n8n-leads-lp-cap-grat`** (planilha "Painel de Controle" — já detectada em `useCrossReferenceLeads.ts:46-48`).
- [x] Atribuir cada Lead a uma LP pela coluna **`utm_content`** (`= content = co=`, col índice 5): se `utm_content` contém `lpa` → **LPA**, `lpb` → **LPB**, etc. (case-insensitive).
- [x] A coluna **Leads** da tabela mostra a contagem de leads daquela LP.
- [x] **Leads é a única métrica externa à API** — todas as demais (Investimento, LP View, Cliques, etc.) vêm do Meta Ads.

### AC7: Filtros Hot/Cold temáticos (Campaign Name + utm_term)

**Arquivos:** `lp-performance-table.tsx` (UI) + `useLpPerformanceData.ts` (lógica) + `creative-metrics-calculator.ts` (classificação)

- [x] Substituir o `<Select>` atual de público (`lp-performance-table.tsx:89-101`) pelos **botões temáticos** idênticos aos de Criativos (`stage-creative-performance-table.tsx:251-267`):
  - Container: `flex items-center gap-1 rounded-md border border-border/40 p-0.5`
  - Opções: `Todos` | `🔥 Hot` | `❄️ Cold`

- [x] Classificação de temperatura olha **Campaign Name E `utm_term`** (`= term = t=`): se qualquer um contém `hot` → **Hot**, contém `cold` → **Cold** (case-insensitive).
- [x] Filtro **efetivo**: ao selecionar Hot/Cold, a tabela recalcula mostrando só os gastos/leads daquele público (hoje o filtro existe mas não filtra — `useLpPerformanceData.ts:47-50`).

- **"Só LPA" (AC3):** o endpoint agrupa por `ad_name`, colapsando as LPs de um mesmo criativo. Resolvido com um corte por LP (`lpBreakdown`) **sobre insights por campanha** (`fetchCampaignInsights`), agrupando por `campaign_name`.
- **Connect Rate (AC5):** alinhado ao card do funil (61.6%). Fonte trocada de nível-anúncio (inflava link_click ~2×) para **nível-campanha**, e métrica trocada para **`inline_link_clicks`** (o "Inline Link Clicks" do Ads Manager) — bateu (~62%).
- **Tx Conv.:** `Leads pagos ÷ Link Clicks` (consistente com a Dados Diários).
- **LPC/LPD sumindo:** `fetchCampaignInsights` trazia só a 1ª página (~25 campanhas, sem filtro). Adicionado `filtering` por `campaign.id` + `limit=500` — respeita as campanhas selecionadas em "Gerenciar campanhas do funil" e traz as de baixo volume.
- **Layout:** header e hover da tabela de LPs alinhados ao padrão das demais tabelas (removido `bg-slate-700`/`text-white`/`hover:bg-gray-50`).
