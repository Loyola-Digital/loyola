# Story 36.4: Job de sync/refresh da performance de criativos (scheduler)

**Epic:** 36 - Meta Ads Creative Intelligence API + MCP
**Story:** 36.4
**Status:** Draft
**Priority:** Medium
**Complexity:** Medium (~1 dia)
**Points:** 5
**Assignee:** @dev

---

## Valor de Negócio

A decisão do Lucas foi **sync + cache no banco** (não chamada ao vivo). Para os endpoints (36.3) e o MCP (36.6) entregarem dados úteis, o cache de performance precisa estar **fresco**. Esta story garante que um job periódico mantém `meta_insights_daily_cache` / criativos atualizados, sem depender de alguém abrir o dashboard. É o que faz a API "saber" da performance de hoje sem latência de Graph API no request.

## Dependências

- **Sem bloqueio** — pode rodar em paralelo com 36.2/36.3.
- **Reuso:** padrão do scheduler existente `packages/api/src/plugins/meta-names-scheduler.ts` (já agenda backfill de nomes via `META_BACKFILL_HOUR`), `meta-insights-cache.ts`, `meta-ads.ts`.

---

## Contexto

- Já existe um scheduler plugin para nomes de entidades Meta (`meta-names-scheduler.ts`) controlado por `META_BACKFILL_HOUR` (env). É o molde a copiar.
- Já existem caches: `meta_entity_names_cache`, `meta_ad_creatives_cache`, `meta_insights_daily_cache`.
- O que falta: garantir que os **insights diários de performance** (não só nomes/criativos) são puxados regularmente para todas as contas Meta ativas (`meta_ads_accounts`).

---

## Aceitação

### AC1: Job agendado de refresh de insights
- [ ] Plugin/scheduler (molde `meta-names-scheduler.ts`) que, em intervalo configurável, para cada conta em `meta_ads_accounts` (não revogada), puxa insights diários recentes via `meta-ads.ts` e popula/atualiza `meta_insights_daily_cache`.
- [ ] Intervalo configurável por env (ex.: `META_PERF_SYNC_HOUR` ou intervalo em minutos) — seguir o padrão do `META_BACKFILL_HOUR`.
- [ ] Janela de refresh sensata (ex.: últimos 7d diários por execução, para corrigir dados que o Meta ajusta retroativamente) — confirmar janela.

### AC2: Robustez
- [ ] Respeita o limite de 90d/chunk do Meta (lógica já existe em `meta-ads.ts`).
- [ ] Falha de uma conta não derruba as outras (try/catch por conta, log do erro).
- [ ] Não duplica linhas no cache (upsert por `adId`+`date`).
- [ ] Decripta o token da conta via padrão existente (`decryptAccountToken`).

### AC3: Observabilidade
- [ ] Log estruturado: contas processadas, linhas atualizadas, erros — alinhado com a tab `settings/audit` existente, se aplicável.
- [ ] Endpoint ou flag para disparo manual do refresh (admin-only) — útil para debug e primeira carga.

---

## Out of Scope
- Os endpoints de leitura (36.3) e o ROAS (36.5).
- Backfill histórico massivo (>1 ano) — só janela recente recorrente; backfill grande é tarefa pontual manual.

---

## Riscos

| Risco | Prob. | Impacto | Mitigação |
|-------|-------|---------|-----------|
| Rate limit da Graph API com muitas contas | Média | Médio | Janela curta por execução + chunk 90d existente + espaçar contas |
| Job escrever em prod sem querer durante dev | Média | Médio | Gate por env; desabilitável; disparo manual separado |
| Dados retroativos do Meta divergirem | Média | Baixo | Refresh de janela de 7d corrige automaticamente |

---

## Notas Técnicas
- **Não reinventar:** o cliente que puxa insights já existe (`meta-ads.ts`); esta story orquestra o *quando*, não o *como* puxar.
- **Upsert:** garantir `ON CONFLICT (adId, date)` (ou equivalente Drizzle) para idempotência.
