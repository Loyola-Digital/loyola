## Story

Na seção "Por Content (Ad)" do dashboard de vendas, anúncios FZ (ex.: `120251580272850273` → `ad04-fz-l2-jun26--se-você-tem-filhos`) aparecem como **ID cru / `[Unresolved]`** em vez do nome resolvido — **mesmo sem nenhum erro nos logs e com o token funcionando** (puxa campanhas normalmente).

### Causa-raiz (investigada, NÃO é a hipótese do Danilo)

A hipótese inicial (rate limit / timeout / credencial inválida em `resolveEntityNames`) foi **descartada por evidência**:

1. **`resolveEntityNames` funciona.** O cache `meta_entity_names_cache` tem 7 nomes resolvidos com sucesso — todos do projeto **DG & CPDF** (IDs terminando em `...0208`). Zero linhas envenenadas (`name === id`).
2. **A Meta resolve os IDs FZ perfeitamente** — porém **só** com o token da conta `act_2863980937018503`:
   - `act_2863980937018503` → `200 { "120251580272850273": { name: "ad04-fz-l2-jun26..." }, ... }`
   - `act_382129543871900` (BBE) → `200 {}`
   - `act_3717530711643512` (DG) → `200 {}`
3. A Meta codifica a conta dona no **sufixo do ID**: `...0208` = DG, `...0273` = `act_2863980937018503` (FZ & MFB). Cada token só enxerga os **próprios** anúncios.

**Mecânica da falha silenciosa:** quando o `projectId` da consulta está vinculado a uma conta que **não é dona** do anúncio, `GET /?ids=...&fields=id,name` retorna **`200 {}`** (sem a chave do ID). Em `meta-ads.ts:1146`, `entry?.name` é `undefined` → o ID cai no fallback `id === name` (linha 1184). **Nenhum erro é lançado**, portanto nada é logado e nada é cacheado. É exatamente o sintoma observado.

> Conclusão: **não é bug de código de resolução.** É vínculo de conta: o anúncio de venda pertence a uma conta Meta diferente da que está vinculada ao projeto que exibe aquela venda.

## Conclusão da Investigação (2026-06-09)

Confirmado pelo usuário: o funil é **FZ & MFB → `fz-l2-jun-26` (Captação Paga)** — o projeto **correto**, vinculado a `act_2863980937018503` (dona dos anúncios).

Simulação end-to-end do endpoint para esse projeto:
- Projeto FZ tem **exatamente 1** conta vinculada (sem ambiguidade no `.limit(1)` do `getProjectMetaToken`).
- `getProjectMetaToken(FZ)` retorna o token de `act_2863980937018503`.
- Esse token resolve os 2 IDs → `200` com os nomes corretos.

**Logo: o backend resolve de ponta a ponta com o banco atual. Não há bug de código nem de vínculo neste funil.**

Os IDs crus observados pelo Danilo **não são reproduzíveis** pelo código atual contra o banco atual. Causa mais provável: **processo da API rodando desatualizado** (o usuário roda a API separadamente). Evidência: cache tem nomes do DG & CPDF (02–05/jun) mas **zero do FZ** — a resolução FZ nunca chegou a rodar/cachear no processo atual.

**Ação recomendada (operacional, não código):**
1. Reiniciar o processo da API.
2. Reabrir `fz-l2-jun-26 > Captação Paga`.
3. Verificar `meta_entity_names_cache` — devem surgir linhas `ad` do FZ (hoje 0).

Se após restart + reload ainda mostrar ID cru → reabrir esta story para investigar o processo em execução (versão do build, env do processo). AC2 (multi-conta por projeto) e AC3 (log de não-resolvidos) permanecem como melhorias válidas independentemente.

## Atualização — causa do "ainda nada" + rate limit (2026-06-09)

Com a API reiniciada, os IDs FZ continuaram crus E o app passou a **estourar o rate limit**. Header da Meta revelou o motivo de fundo: `ads_api_access_tier: "development_access"` — o app Meta está no **tier de desenvolvimento**, com limite de chamadas baixíssimo.

**Ciclo vicioso identificado:** o dashboard resolvia **ID-por-ID** a cada carga; como nada ficava cacheado (rate-limited antes de terminar), toda carga reconsultava centenas de IDs → estourava o limite → não cacheava → repetia.

### Solução implementada

1. **Backfill em massa** (`packages/api/scripts/backfill-meta-names.mjs`): em vez de N lookups `/?ids=`, puxa a conta inteira pelos edges `/act_<id>/ads|adsets|campaigns` (paginado, ~17 chamadas/conta) e faz upsert em `meta_entity_names_cache` por projeto. Executado nas 3 contas:
   - act_2863980937018503 (FZ&MFB): 5841 ads, 1080 adsets, 662 campaigns
   - act_382129543871900 (BBE): 426 ads, 72 adsets, 24 campaigns
   - act_3717530711643512 (DG&CPDF): 5059 ads, 1415 adsets, 593 campaigns
   - **Total no cache: 11326 ads, 2567 adsets, 1279 campaigns** — incluindo os 7 IDs da tela.
2. **TTL do cache 24h → 30 dias** (`stage-sales-data.ts`, `traffic-analytics.ts`): nomes são estáveis; TTL longo evita reconsultar a Meta e re-estourar o rate limit. Refresh de nomes passa a vir do backfill.

### Ação recorrente recomendada
Rodar `node scripts/backfill-meta-names.mjs` periodicamente (ex.: semanal) — ou agendar via @devops — pra manter nomes novos no cache. Enquanto o app Meta estiver em `development_access`, resolução on-demand em massa não é viável.
