## Story 31.2 — Pull on-demand de tasks + cache 5min (5 pts)

**User story:** Como dashboard, preciso pegar tasks de N listas ClickUp em uma chamada com cache 5min, pra renderizar cards sem hammer no rate-limit ClickUp.

**Implementação:**
- `GET /api/sprint-dashboard/tasks?listIds=a,b,c` — guest-blocked
- Cache em-memória (Map<key, {tasks, expiresAt}>) chave = listIds ordenados
- TTL 5min (`TASKS_CACHE_TTL_MS = 5 * 60 * 1000`)
- Promise.allSettled paralelo por lista (1 falha não derruba o resto)
- Retorna `ClickUpTaskSimple[]` simplificado (id, name, status, statusColor, tags, url, dueDate, startDate, assignees, listId, listName)
- `?bypassCache=1` força refetch

**Files:** `packages/api/src/routes/sprint-dashboard.ts`

## Story 31.3 — Write-through PUT status (2 pts)

**User story:** Como user, quando marco task como done no dashboard Loyola, quero que vire done no ClickUp imediatamente.

**Implementação:**
- `PUT /api/sprint-dashboard/task/:taskId/status` body `{status: string}`
- Chama `fastify.clickupService.updateTaskStatus(taskId, status)` → `PUT /task/{id}` no ClickUp
- Após sucesso, invalida cache (`tasksCache.clear()`) pra próxima leitura refletir
- Service `clickup.ts` ganha `updateTaskStatus(taskId, status)`

**Files:** `packages/api/src/routes/sprint-dashboard.ts`, `packages/api/src/services/clickup.ts`

## Story 31.5 — Builder UI (5 pts)

**User story:** Como user, quero adicionar/editar/remover blocos do dashboard via UI, selecionando listas ClickUp + filtros + visual.

**Implementação:**
- Dialog `SprintBuilderDialog` aberto pelo botão "Configurar"
- Lista N blocos editáveis (cor + título + remover + mover ↑↓)
- Cada bloco tem 2 painéis colapsáveis:
  1. **List picker** — tree de spaces → folders → lists com busca, checkbox múltiplo
  2. **Filtros + agrupamento** — chips de status (carregados via `/list/:id/statuses`), input de tags, input de assignee IDs, select de groupBy
- `GET /api/sprint-dashboard/hierarchy` — devolve workspace inteiro (paralelo: folders + folderless lists por space)
- `GET /api/sprint-dashboard/list/:listId/statuses` — statuses da lista pro picker de filtro
- Save valida que todo bloco tem ≥ 1 lista, chama `PUT config`, invalida queries

**Files:**
- `packages/web/components/sprint-dashboard/sprint-builder-dialog.tsx`
- `packages/api/src/routes/sprint-dashboard.ts` (endpoints `hierarchy` + `list/:id/statuses`)

## Story 31.6 — Métricas-resumo do header (2 pts)

**User story:** Como user, quero ver eventos próximos (tasks com due_date no horizonte) e total de projetos ativos no header do dashboard.

**Implementação:**
- `GET /api/sprint-dashboard/metrics`:
  - Lê todos os listIds da config
  - Pull de tasks (reusa cache 5min de 31.2)
  - Filtra `dueDate` no horizonte `[hoje−14d, hoje+14d]`, ordena crescente, top 10
  - Conta blocos com ≥ 1 task em status não-final (`done|closed|cancelado|concluído|complete`)
- Header da página renderiza 1 card "Projetos ativos" + N cards de eventos (date em destaque, list name + task name)
- Cards de eventos linkam pra URL do ClickUp

**Files:**
- `packages/api/src/routes/sprint-dashboard.ts` (endpoint `/metrics`)
- `packages/web/app/(app)/sprint-dashboard/page.tsx` (componente `MetricsHeader`)
