## Story

### Problema
Hoje os cards do Top Criativos mostram:
- Métrica principal (CPL pago, CTR, etc.)
- Breakdown de leads por origem (Pagos/Org/S.orig)
- Dados qualitativos da pesquisa (💰 faturamento declarado, 👤 profissão, etc.)

**Falta:** quanto R$ aquele criativo efetivamente faturou. O gestor não consegue dizer "criativo X traz mais leads, mas criativo Y converte em mais receita" com os dados atuais.

Hoje o único dado de venda no dashboard está no **card KPI agregado** (Faturamento do dashboard inteiro) e na **StageSalesSection** (totais por canal/forma de pagamento). Nenhum desses é por criativo.

### Solução Proposta
Cruzamento em 3 etapas, server-side, retornando um map `{ [adId]: { faturamentoBruto, faturamentoLiquido, vendas } }`:

1. **Meta Ads → Planilha de leads:** para cada `ad_id`, buscar leads cujo `utm_content` bate (usando `normalizeNumericId` pra `_12345...`).
2. **Planilha de leads → Planilha de vendas:** para cada lead, pegar `email` e cruzar com `email` da planilha de vendas (subtype `capture`) do stage. Dedup via email (lowercase+trim).
3. **Agregação:** somar `valorBruto` / `valorLiquido` e contar vendas únicas (por email) por ad.

Frontend: nova linha no card, logo abaixo do breakdown Pagos/Org/S.orig, formato:
```
💵 Faturado: R$ 4.250,00 · 8 vendas
```
Se zero vendas → omite a linha (não mostra "R$ 0,00").

### User Story
**Como** gestor de tráfego do DGPG 02,
**Eu quero** ver quanto R$ cada criativo efetivamente faturou,
**Para que** eu possa priorizar criativos pelo ROI real e não só pelo volume de leads.

## Acceptance Criteria

### AC-1: Endpoint Novo Retorna Map de adId → Financeiro
**Given** a stage tem uma planilha de leads (`funnel_spreadsheets` type=leads) e uma planilha de vendas (`stage_sales_spreadsheets` subtype=capture)
**When** o frontend chama `GET /api/projects/:id/funnels/:funnelId/stages/:stageId/creative-revenue?days=N`
**Then** o backend retorna `{ byAdId: { [adId]: { faturamentoBruto: number, faturamentoLiquido: number, vendas: number } }, totalVendas: number, faturamentoBruto: number, faturamentoLiquido: number }`

### AC-2: Cruzamento utm_content → email
**Given** há um lead na planilha de leads com `utm_content = "120225436094170123"` e `email = "joao@x.com"`
**And** há uma venda na planilha de vendas com `email = "joao@x.com"` e `valorBruto = 497`
**When** o endpoint é chamado
**Then** `byAdId["120225436094170123"].faturamentoBruto` inclui R$ 497
**And** `byAdId["120225436094170123"].vendas` inclui 1

### AC-3: Normalização de utm_content Numérico com Underscore
**Given** o lead tem `utm_content = "_120225436094170123"` (formato Google Sheets texto)
**When** o cruzamento é feito
**Then** `normalizeNumericId` remove o `_` e casa com `ad_id = "120225436094170123"` do Meta
**Reuso:** função existente em `lib/utils/normalize-answer.ts` (Story 21.5).

### AC-4: Dedup de Email na Agregação de Vendas
**Given** o mesmo email comprou 2x na planilha de vendas (R$ 100 + R$ 50)
**When** o endpoint agrega o ad desse lead
**Then** `faturamentoBruto` soma R$ 150 (todos os produtos)
**And** `vendas` conta 1 (pessoa única)
**Rationale:** mesma semântica da Story 21.4 (`stage-sales-data.ts` já faz isso).

### AC-5: Filtro de Dias Aplicado
**Given** o parâmetro `?days=30` foi passado
**When** o endpoint cruza vendas
**Then** só considera vendas cuja `dataVenda` está nos últimos 30 dias
**And** vendas fora do range são ignoradas no cruzamento

### AC-6: Card Exibe Linha de Faturamento
**Given** `byAdId[c.ids[0]]` (ou qualquer id do agregado) tem `faturamentoBruto > 0`
**When** o card renderiza
**Then** exibe linha abaixo do breakdown de leads, formato: `💵 Faturado: R$ X.XXX,XX · N venda(s)`
**And** usa `Intl.NumberFormat("pt-BR", { style: "currency", currency: "BRL" })`

### AC-7: Sem Vendas → Linha Omitida
**Given** o criativo não tem vendas cruzadas (`faturamentoBruto === 0`)
**When** o card renderiza
**Then** a linha de faturamento **NÃO** é exibida (não mostra "R$ 0,00")
**Rationale:** evitar ruído visual; gestor entende ausência como "sem venda".

### AC-8: Agregação por Múltiplos ad_ids do Mesmo Criativo
**Given** o criativo agregado tem `c.ids = ["ad_1", "ad_2", "ad_3"]` (vária variações do mesmo nome)
**When** o card renderiza
**Then** soma `byAdId["ad_1"] + byAdId["ad_2"] + byAdId["ad_3"]` no mesmo bucket
**And** dedup de email cruza **entre** os ad_ids (se mesmo cliente comprou por 2 ads, conta 1 venda)

### AC-9: Stage Sem Planilhas Retorna Vazio Graciosamente
**Given** o stage não tem planilha de leads OU não tem planilha de vendas
**When** o endpoint é chamado
**Then** retorna `{ byAdId: {}, totalVendas: 0, faturamentoBruto: 0, faturamentoLiquido: 0, semDados: true }`
**And** o card **não exibe** a linha de faturamento (nenhum criativo vai ter match)

### AC-10: Typecheck + Lint PASS
**Given** a implementação está completa
**When** `pnpm typecheck` e `pnpm lint` são executados em `packages/api` e `packages/web`
**Then** ambos retornam 0 erros

## Dev Notes

### Arquitetura sugerida do endpoint

```typescript
// packages/api/src/routes/creative-revenue.ts (novo)
export default fp(async function creativeRevenueRoutes(fastify) {
  fastify.get(
    "/api/projects/:projectId/funnels/:funnelId/stages/:stageId/creative-revenue",
    async (request, reply) => {
      // 1. Validar access ao projeto (reuso do pattern de funnel-stages.ts)
      // 2. Buscar planilha de leads: funnel_spreadsheets WHERE stage_id = X AND type = 'leads' LIMIT 1
      //    (se null → retornar semDados: true)
      // 3. Buscar planilha de vendas: stage_sales_spreadsheets WHERE stage_id = X AND subtype = 'capture'
      //    (se null → retornar semDados: true)
      // 4. Ler ambas via readSheetData (cache 5min)
      // 5. Construir map salesByEmail: Map<email, { bruto, liquido, vendas }>
      //    - Iterar rows de vendas, aplicar filtro de dias (parseDate + cutoff)
      //    - Dedup por email, somar bruto/liquido, count++
      // 6. Construir map byAdId: Map<adId, { faturamentoBruto, faturamentoLiquido, vendas, emails: Set<string> }>
      //    - Iterar rows de leads, aplicar filtro de dias
      //    - Normalizar utm_content com normalizeNumericId
      //    - Se salesByEmail tem o email do lead → somar no adId
      //    - Set<email> pra dedup dentro do mesmo ad
      // 7. Colapsar em { faturamentoBruto, faturamentoLiquido, vendas }
      // 8. Retornar { byAdId, totalVendas, faturamentoBruto, faturamentoLiquido }
    }
  );
});
```

### Pontos de atenção técnica

- **Helper compartilhado:** `parseDate` e `parseNumber` podem ser extraídos pra util compartilhado em vez de duplicar de `stage-sales-data.ts`. Se ficar apertado, duplicar é aceitável — já tem precedente no projeto.
- **Performance:** planilhas de até ~5k rows cada, 2 passadas O(n+m). Irrelevante.
- **Cache:** `readSheetData` já cacheia 5min. Cache adicional no endpoint seria over-engineering.
- **Separação de arquivo:** criar `creative-revenue.ts` separado em vez de estender `stage-sales-data.ts` pra manter scope limpo. Plugin registrado em `app.ts`.

### Frontend

```typescript
// packages/web/lib/hooks/use-creative-revenue.ts (novo)
export function useCreativeRevenue(projectId, funnelId, stageId, days) {
  const apiClient = useApiClient();
  return useQuery({
    queryKey: ["creative-revenue", projectId, funnelId, stageId, days],
    queryFn: () => apiClient<CreativeRevenueData>(
      `/api/projects/${projectId}/funnels/${funnelId}/stages/${stageId}/creative-revenue?days=${days}`
    ),

    staleTime: 5 * 60 * 1000,

```

No componente `TopCreativesGallery`, receber `projectId, funnelId, stageId, days` (já tem tudo menos talvez stageId — verificar props) e no render do card somar `c.ids.reduce((s, id) => s + (revenueData.byAdId[id]?.faturamentoBruto ?? 0), 0)`.

### Desambiguação de dedup entre ads

Se o mesmo email foi lead de **2 ad_ids diferentes** do mesmo criativo agregado:
- `salesByEmail["joao@x.com"] = { bruto: 497, vendas: 1 }`
- lead de `ad_1` e `ad_2` ambos têm email joao@x.com
- `byAdId.ad_1` e `byAdId.ad_2` **ambos** contam essa venda?
  - **Sim** no nível do backend (é atribuição multi-touch simples — primeiro-touch vs último-touch é fora de escopo)
  - Frontend soma as duas → **dupla contagem**
- **Solução:** frontend deduplica por email usando um Set ao somar:
  ```ts
  const seenEmails = new Set<string>();
  let bruto = 0, vendas = 0;
  for (const id of c.ids) {
    for (const email of revenueData.byAdId[id]?.emailsSet ?? []) {
      if (seenEmails.has(email)) continue;
      seenEmails.add(email);
      bruto += salesByEmail[email].bruto;
      vendas++;
    }
  }
  ```
- Isso implica que o **endpoint precisa retornar também a lista de emails por ad** (ou um map global salesByEmail) pra o frontend poder deduplicar.
- **Decisão de design:** endpoint retornar `emails: string[]` por ad + `salesByEmail: Record<string, { bruto, liquido }>` no root. Aumenta payload mas resolve dedup corretamente. @dev decide o shape final no PR.

## Risks

- **Médio — Complexidade de dedup entre ad_ids:** quando o mesmo email é atribuído a 2+ ads do mesmo criativo agregado, simples soma geraria dupla contagem. Mitigação: retornar `emails: string[]` por ad no endpoint e deduplicar no frontend. Documentado em Dev Notes.
- **Baixo — Performance com muitos ads:** se o agregado tem 50+ ad_ids, somar rows do hashmap é O(n) — irrelevante. Planilhas grandes (5k+ rows) lidas em 2 passadas, também OK.
- **Baixo — Cache desalinhado:** `readSheetData` cacheia 5min; o user pode editar planilha e esperar ver o valor novo imediatamente. Mitigação: aceitar delay ou adicionar botão "Refresh" em iteração futura.
- **Baixo — Inconsistência entre Faturamento do card (por criativo) e Faturamento do KPI topo (total do stage):** podem divergir se alguns ads têm leads sem venda atribuída. Esperado — não é regressão.
