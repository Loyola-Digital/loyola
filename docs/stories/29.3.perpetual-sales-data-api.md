# Story 29.3 — API `/perpetual/sales-data` (agregação + daily + utm_source)

**Status:** Ready for Review
**Points:** 3
**Depende de:** 29.1

## User Story

Como **dashboard do perpétuo**, preciso de endpoints que leiam a planilha conectada (Story 29.1) e devolvam métricas agregadas (totalVendas, faturamentoBruto/Líquido, ticketMédio), distribuição por **utm_source bruto** (sem normalização "Pago/Orgânico"), distribuição por forma de pagamento, e série diária de receita — para que as Stories 29.4/29.5 abasteçam KPIs e gráficos no front.

## Dev Notes

### Diferenças vs stage-sales-data

| Aspecto | stage-sales-data | perpetual-sales-data |
|---|---|---|
| Escopo | `(stageId, subtype)` | `funnelId` (1 planilha) |
| Filtro do tipo | `subtype IN (capture, main_product, sales)` | `type = 'perpetual_sales'` |
| Tabela | `stageSalesSpreadsheets` | `funnelSpreadsheets` (reuso) |
| porUtmSource | Normalizado em Pago/Orgânico/SemTrack | **Bruto** (`google`, `instagram_organic`, etc.) |
| Meta names resolution | Sim (utm_medium → adset_name) | **Não** (fora de escopo desta story; pode entrar em 29.4 se necessário) |
| hot-cold/conversion | Sim | Não aplicável |

### Por que `porUtmSource` bruto?

Em perpétuo, o time precisa enxergar a granularidade de origem (cada source vira uma linha do gráfico). Normalização Pago/Orgânico/SemTrack faz sentido pra lançamento (onde a separação binária ajuda), mas pra perpétuo perde a riqueza que o time quer (qual parceria, qual landing page, qual canal específico está performando).

### Bucket "(sem origem)"

Linhas com utm_source vazio/null/"-"/"na" — usar `sanitizeUtmValue` retornando null → bucket `"(sem origem)"`. Não esconder essas linhas (representam tráfego direto/orgânico não-trackeado, importante pra ROAS real).

### Performance

`readSheetData` já tem cache interno em `google-sheets.ts` (30s TTL). Não preciso adicionar caching adicional.
