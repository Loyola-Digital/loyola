## Story

**As a** Loyola Digital admin,
**I want** REST endpoints to manage projects and filter Instagram accounts by project,
**so that** resources can be organized and accessed per client.

---

## Acceptance Criteria

1. [ ] `POST /api/projects` — Criar projeto: `{ name, clientName, description?, color? }` → 201 com projeto criado
2. [ ] `GET /api/projects` — Listar projetos do usuário autenticado (scoped por `created_by`)
3. [ ] `GET /api/projects/:id` — Detalhes do projeto (ownership check)
4. [ ] `PUT /api/projects/:id` — Atualizar `name`, `clientName`, `description`, `color`
5. [ ] `DELETE /api/projects/:id` — Remover projeto (cascade deleta recursos vinculados)
6. [ ] `POST /api/instagram/accounts` — Agora aceita `project_id` opcional no body
7. [ ] `GET /api/instagram/accounts?project_id=` — Filtra contas por projeto; sem filtro = todas as contas do user
8. [ ] `GET /api/projects/:id/instagram/accounts` — Atalho: contas do projeto
9. [ ] `GET /api/projects/:id/conversations` — Conversas do projeto
10. [ ] Todos os endpoints protegidos por auth (Clerk JWT)
11. [ ] Projetos scoped por `created_by` — GET retorna apenas projetos do user autenticado
12. [ ] Error responses padronizadas: `{ error: string, code?: string }`
13. [ ] Testes de integração para cada endpoint

## Dev Notes

### Architecture Context

- **Fastify Routes:** Plugin pattern idêntico a `routes/instagram.ts` [Source: packages/api/src/routes/instagram.ts]
- **Auth Middleware:** `requireAuth` / `authPlugin` já existe em `middleware/auth.ts`
- **Validation:** Zod `.safeParse()` no route handler — padrão do projeto
- **DB:** Drizzle com tabela `projects` criada na Story 4.1

### API Contract

**POST /api/projects**
```json
// Request
{ "name": "Projeto XYZ", "clientName": "Cliente XYZ Ltda", "description": "...", "color": "#d4a843" }
// Response 201
{ "id": "uuid", "name": "Projeto XYZ", "clientName": "Cliente XYZ Ltda", "color": "#d4a843", "isActive": true, "createdAt": "..." }
```

**GET /api/projects**
```json
[{ "id": "uuid", "name": "...", "clientName": "...", "color": "#...", "isActive": true, "createdAt": "..." }]
```

**GET /api/instagram/accounts?project_id=uuid**
```json
// Filtra accounts pelo project_id; sem ?project_id retorna todas do user
[{ "id": "uuid", "accountName": "...", "instagramUsername": "...", "projectId": "uuid" }]
```

#### Concerns (advisory — não bloqueiam merge)

**1. AC6/AC7 sem teste explícito do novo campo** (Medium):
`POST /api/instagram/accounts` com `projectId` fornecido e `GET ?project_id=uuid` com filtro ativo não têm casos de teste dedicados em `instagram-routes.test.ts`. Funcionalidade correta no código, mas branch `projectId != null` não coberta por teste. Adicionar em `instagram-routes.test.ts` em Story futura.

**2. GET /:id/conversations e GET /api/projects sem paginação** (Low):
Retornam todos os itens sem `limit`/`offset`. Padrão consistente com demais endpoints do projeto (`GET /api/instagram/accounts`), mas pode ser risco de performance em volume. Avaliar paginação quando EPIC-4 estiver completo.

**3. PUT usa `Record<string, unknown>` no set()** (Low):
Bypassa type-checking do Drizzle no update. Valores vêm de Zod-validated data, então seguro na prática. Pode ser revisado para `Partial<typeof projects.$inferInsert>` em refactor futuro.

**4. `isActive` não exposto no PUT** (Low / design intencional):
Não há forma de desativar projeto via `PUT /api/projects/:id`. AC4 não exige. Se houver caso de "arquivar projeto", avaliar endpoint dedicado em Story separada.
