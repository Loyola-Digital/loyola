## Story

**As a** developer on the Loyola Digital X project,
**I want** a Mind Engine service that loads mind artifacts from the filesystem using a 3-tier strategy and assembles complete system prompts for Claude API,
**so that** the Chat API (Story 1.2.2) can send properly formatted requests to Claude with each mind's full persona.

### Mind Engine Architecture (Section 6)

**Tiered Loading Strategy:**

| Tier | When | Artifacts | Tokens |
|------|------|-----------|--------|
| 1 | Every message | COGNITIVE_OS + COMMUNICATION_DNA | ~5K |
| 2 | Conversation start | Tier 1 + Frameworks + Value Equation | ~11.6K |
| 3 | On-demand | Tier 2 + Antipatterns + Case Library + Heuristics | ~16-20K |

**Prompt Structure:**

```
[Identity Header]
You are {name}. Respond as this person would.
Squad: {squadDisplayName}

---

[COGNITIVE_OS.md content]

---

[COMMUNICATION_DNA.md content]

---

[Tier 2+ artifacts if applicable]

---

[TASK_DELEGATION_PROMPT]
```

**TASK_DELEGATION_PROMPT** (exato da architecture doc):

```
## Task Delegation Capability

When the user asks you to create, schedule, or delegate a task:
1. Acknowledge the request in your persona's voice
2. Summarize the task clearly
3. End your message with exactly this JSON block:

\`\`\`json:task
{
  "action": "create_task",
  "title": "Task title here",
  "description": "Detailed description",
  "priority": 2,
  "tags": ["tag1", "tag2"]
}
\`\`\`

The system will detect this block and prompt the user to confirm task creation in ClickUp.
Do NOT mention ClickUp by name — just say "I can create a task for that."
```

### Content-Engine Agents vs MMOS Minds

| Aspect | MMOS Minds | Content-Engine Agents |
|--------|-----------|----------------------|
| Structure | Multi-file (8-14 artifacts) | Single-file (YAML + markdown) |
| Loading | Tiered (3 tiers) | Single file load (tier 1 = tudo) |
| Config | `docs/config.json` | Embedded in YAML block |

O `MindEngine` abstrai essa diferenca — para agents, tier 1 ja carrega tudo.

### ArtifactCache Design

```typescript
// LRU cache config (from architecture doc section 6.5)
{
  max: 100,              // Max 100 artifacts in memory
  maxSize: 50_000_000,   // 50MB max total
  sizeCalculation: (value) => Buffer.byteLength(value),
  ttl: 1000 * 60 * 30,  // 30min TTL
}
```

**Package:** `lru-cache` (npm) — popular, zero-dependency, TypeScript native.

### Key Implementation Notes

1. **MindEngine depende do MindRegistry** — precisa de `fastify.mindRegistry.getById()` para obter metadata
2. **ArtifactCache e separado** do MindRegistry — registry indexa paths, cache armazena conteudo
3. **Tier 3 e on-demand** — nao carregado no system prompt, injetado via mensagem quando necessario
4. **lru-cache e a unica dependencia nova** — package maduro, 0 deps, TypeScript nativo
5. **Agents carregam tudo no tier 1** — arquivo unico ja e compacto o suficiente
6. **TASK_DELEGATION_PROMPT sempre incluido** — em toda conversa, independente do tier

### Observations (Non-Blocking)

1. **Dead exclusion comparisons in prompt-builder.ts:74-81** — The tier 2 OFFER/AXIOM loop excludes standard keys using `k !== "cognitiveOS"` etc., but `k` is already lowercased so these comparisons never match camelCase strings. This is benign because no standard key contains "offer" or "axiom", so the short-circuit `k.includes("offer") || k.includes("axiom")` already filters them out. Dead code, zero impact on correctness.

2. **No tier 3 test coverage** — Fixtures lack antipatterns/caseLibrary/heuristics artifacts, so tier 3 loading path is untested. Acceptable for now since tier 3 is on-demand and the code path mirrors tier 1/2. Future stories adding real squad data should add tier 3 tests.

3. **PromptBuilder tests create individual Fastify instances per test** — Each test in the PromptBuilder describe block creates/closes its own Fastify instance. Works correctly but adds slight overhead. Could be optimized with beforeAll/afterAll pattern like the integration suite. Non-blocking.
