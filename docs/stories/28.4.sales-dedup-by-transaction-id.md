## User Story

Como gestor de vendas do Loyola, quando configuro a planilha do **Produto Principal** no funil e ela tem 15 vendas, quero ver **15 vendas** no dashboard. Hoje aparece **5** — perda direta de 67% da visibilidade do faturamento.

---

## Bug atual

### Sintoma reportado (Lucas, 13/05/2026)

- Planilha conectada como `subtype = main_product`
- 15 linhas, 15 emails únicos, todas com `Evento = paid`
- Produto: "Meu Filho Bilíngue - New"
- Datas: 11/05 a 13/05/2026
- Coluna `ID` (transaction_id) tem valor único por venda: `52nIkvD`, `YHHVJ7H`, `iYqUspR`, etc.
- Dashboard mostra **5 vendas** (`emailMap.size === 5`)

### Causa raiz suspeita (ordenada por probabilidade)

| # | Hipótese | Como validar |
|---|----------|--------------|
| H1 | **Dedup por email** ≠ dedup por transação. Mesmo com 15 emails únicos, alguma normalização ou edge case está colapsando 10 keys (improvável dado código `.trim().toLowerCase()`, mas precisa confirmar com debug counters) | Adicionar response debug: `{ totalRows, validRows, afterEmailFilter, afterDateFilter, uniqueEmails }` |
| H2 | **`columnMapping.email` aponta pra coluna errada** (ex: `Instagram` ou `Cidade` quase sempre vazias). 10 linhas têm email vazio → `if (!email) continue` em [stage-sales-data.ts:212](packages/api/src/routes/stage-sales-data.ts#L212) | Lucas abrir o mapping atual + comparar com headers da planilha. Debug counter `skippedEmailEmpty` confirma |
| H3 | **`dataVenda` mapeada errada** e `parseDate` falha → cutoff descarta linhas. Mas Lucas usa `days=30` padrão; cutoff seria 13/04, todas vendas (11-13/05) deveriam passar — improvável | Debug counter `skippedDateInvalid` + `skippedDateOutOfRange` |
| H4 | **Múltiplas planilhas com mesmo `subtype=main_product`** e uma delas tem dados antigos/parciais — query line 137-145 carrega TODAS e deduplica entre elas (mesmo email em planilhas diferentes é tratado como vendas separadas via `${spreadsheetId}|${email}`, mas vazias podem deixar emailMap zoado) | Debug response inclui `spreadsheetsLoaded: [{id, rows}]` |

### Causa de design (independente da causa raiz operacional)

Mesmo se H1-H4 forem resolvidas, **o algoritmo de dedup por email é semanticamente errado** pra produtos com recompras/retries do Kiwify:

```ts
// stage-sales-data.ts:229-244
const compositeKey = `${spreadsheet.id}|${email}`;
const existing = emailMap.get(compositeKey);
if (existing) {
  existing.bruto += bruto;  // SOMA — mascarando uma 2ª venda como 1
  existing.liquido += liquido;
}
```

A coluna `Transaction` ou `ID` da planilha tem um valor único por **transação real**. Deduplicação correta é por `transaction_id`, não por email.

## Comportamento desejado

### Fix 1 — Dedup por `transaction_id` quando mapeado

1. Adicionar campo opcional `transactionId` ao `SaleColumnMapping` (shared types + wizard UI + backend)
2. Backend: se `mapping.transactionId` presente e coluna válida → usa `transaction_id` como chave de dedup
3. Fallback: se `transactionId` não mapeado ou coluna vazia → comportamento atual (dedup por email) — sem regressão

### Fix 2 — Debug response (gated por query param)

1. Adicionar `?debug=1` na rota `GET /api/projects/:projectId/funnels/:funnelId/stages/:stageId/sales-data`
2. Quando `debug=1`, response inclui campo extra `debug` (não muda contrato do `StageSalesData`):

```ts
debug: {
  spreadsheetsLoaded: Array<{ id, name, totalRows, validRows }>,
  totalRowsRead: number,
  skippedEmailEmpty: number,
  skippedDateInvalid: number,
  skippedDateOutOfRange: number,
  uniqueDedupeKeys: number,  // antes/depois de dedup
}
```

3. Lucas pode chamar `?debug=1` no devtools pra ver exatamente onde os 10 somem

## Acceptance Criteria

### AC1 — Mapping de `transaction_id`
- **AC1.1:** `SaleColumnMapping` (shared types) tem campo opcional `transactionId?: string`
- **AC1.2:** Wizard `stage-sales-wizard-dialog.tsx` mostra novo campo "ID da Transação (opcional)" — não required
- **AC1.3:** Backend aceita e persiste `transactionId` no JSONB `columnMapping` de `stage_sales_spreadsheets`

### AC2 — Dedup correta
- **AC2.1:** Quando `mapping.transactionId` presente e índice da coluna >= 0 e valor da célula não vazio → chave de dedup é `${spreadsheet.id}|${transactionId.trim()}`
- **AC2.2:** Quando `transactionId` está ausente ou célula vazia para uma linha específica → fallback pra email (comportamento atual)
- **AC2.3:** Comparado com cenário atual sem `transactionId` mapeado: zero regressão (mesma contagem)
- **AC2.4:** Após Lucas re-mapear sua planilha apontando `transactionId → ID`, dashboard mostra **15 vendas** (atende espec)

### AC3 — Debug response
- **AC3.1:** Query param `?debug=1` adiciona campo `debug` no response sem afetar campos existentes
- **AC3.2:** Counters refletem realidade: somatório de `validRows + skippedEmailEmpty + skippedDateInvalid + skippedDateOutOfRange = totalRowsRead`
- **AC3.3:** `uniqueDedupeKeys` mostra contagem após dedup (deve bater com `totalVendas` exibido)
- **AC3.4:** Quando `?debug=1` ausente, response é exatamente o mesmo de hoje (campo `debug` omitido, não null)

### AC4 — Sem regressão
- **AC4.1:** Tipos `StageSalesData` não mudam (campo `debug` é additive e opcional)
- **AC4.2:** Funis sem `transactionId` mapeado continuam funcionando idêntico
- **AC4.3:** Stage Sales na perpetual dashboard não é afetada (mesmo hook, contrato preservado)

### AC5 — Validação manual (Lucas)
- **AC5.1:** Após deploy, Lucas mapeia `ID → transactionId` no wizard
- **AC5.2:** Recarrega dashboard, confirma 15 vendas exibidas
- **AC5.3:** Se ainda mostrar < 15, chama `?debug=1` e investiga junto com @dev qual counter é alto
