# Story 28.1 — Match de campanhas por fase da etapa (além do código do funil)

**Epic:** 28 — Match Inteligente de Campanhas
**Status:** Ready
**Priority:** High
**Estimate:** S

---

## User Story

Como gestor de tráfego do Loyola, quando crio uma **etapa** num funil, quero que o sistema **auto-popule** as campanhas certas baseado **não só** no código do funil (ex: `dg-pg02`) **mas também** no prefixo da fase (ex: `vendas-captacao`, `vendas-principal`, `leads`), pra que cada etapa receba só as campanhas que pertencem a ela — em vez do mesmo pool de candidatas misturadas em todas as etapas.

---

## Bug / cenário atual

- Cria funil "DG-PG02 Abril" com `matchCode = dg-pg02`
- Cria etapa "Captação Paga" (stageType `paid`) → backend grava `campaigns: []`
- Cria etapa "Venda Principal" (stageType `paid`) → mesma coisa
- `GET /orphan-campaigns` `byStage` retorna **mesma lista** pra ambas: todas as campanhas com `dg-pg02` no nome
- Gestor tem que escolher manualmente quais vão pra qual stage. Pior: pode mover errada por engano

---

## Comportamento desejado

Quando vc cria uma etapa, o backend:

1. Olha o `funnel.matchCode` (ex: `dg-pg02`)
2. Resolve o `phaseSuffix` da etapa baseado em `stageType` + nome (heurística — ver tabela)
3. Filtra todas as campanhas Meta da conta vinculada que casam com **ambos** `matchCode` E `phaseSuffix` no nome (case-insensitive)
4. Grava direto em `stage.campaigns` (auto-populate)

E no `GET /orphan-campaigns` o `byStage` passa a usar o mesmo filtro: só sugere campanhas que casam com `matchCode` + `phaseSuffix` da stage específica.

---

## Heurística `resolveStagePhaseSuffix`

Helper puro (sem I/O):

```ts
function resolveStagePhaseSuffix(
  funnelType: "launch" | "perpetual",
  stageType: "paid" | "free" | "sales" | "cpl",
  stageName: string,
): string | null {
  if (funnelType === "perpetual") return null;
  if (stageType === "cpl") return null;

  const lower = stageName.toLowerCase();
  const isCapture = /capta[cç][aã]o|captura|leads/.test(lower);
  const isMainProduct = /principal|vendas|produto/.test(lower);

  if (stageType === "free" && isCapture) return "leads";
  if (stageType === "paid" && isCapture) return "vendas-captacao";

  if (isMainProduct) return "vendas-principal";  // free, paid, sales
  if (stageType === "sales") return "vendas-principal";

  return null;  // ambíguo → não filtra extra
}
```

Tabela de exemplos:

| Funnel type | Stage type | Stage name | `phaseSuffix` |
|---|---|---|---|
| launch | free | "Captação Gratuita" | `leads` |
| launch | paid | "Captação Paga" | `vendas-captacao` |
| launch | paid | "Venda Principal" | `vendas-principal` |
| launch | sales | "Vendas Produto Principal" | `vendas-principal` |
| launch | cpl | "CPL Aula 1" | `null` |
| launch | free | "Etapa Estranha" (não bate heurística) | `null` |
| perpetual | qualquer | qualquer | `null` |

`null` = não aplica filtro extra (mantém comportamento legacy de só código).

## Acceptance Criteria

1. **AC1 — Auto-popular ao criar stage:**
   - Criar etapa "Captação Paga" (paid) em funil `dg-pg02` → `stage.campaigns` é populado com campanhas Meta cujo nome contém `dg-pg02` E `vendas-captacao`
   - Criar etapa "Venda Principal" (paid) → populado com campanhas que contém `dg-pg02` E `vendas-principal`
   - Criar etapa "Captação Gratuita" (free) → populado com campanhas que contém `<code>` E `leads`
   - Se nenhuma campanha bate → `stage.campaigns = []` (igual hoje)

2. **AC2 — `/orphan-campaigns` byStage filtrado:**
   - Pra cada stage, `orphans` retornados respeitam o filtro `matchCode` E `phaseSuffix` (quando o suffix existir)
   - Stages com `phaseSuffix = null` (cpl, perpétuo, nomes ambíguos) mantém comportamento legacy: só `matchCode`

3. **AC3 — Sem regressão em funis existentes:**
   - Funis perpétuos: `byStage` continua mostrando o pool por código
   - Etapas CPL: sem filtro extra
   - Etapas com nome que não bate heurística (ex: "Etapa 1"): cai no fallback `null`, sem regressão

4. **AC4 — Não popula campanhas em conflito:**
   - Auto-popular só executa quando o usuário **não passou** `campaigns` explícito no POST. Se cliente passar `campaigns: [...]` no body, respeita o que foi passado (não sobrescreve).

5. **AC5 — Helper unitestado:**
   - `resolveStagePhaseSuffix` tem teste com pelo menos 1 caso de cada linha da tabela acima

## Dev Notes

### Onde chamar `findMatchingCampaignsForStage`

No POST de stage (`funnel-stages.ts:193+`), depois do parse mas antes do `insert`:

```ts
const phaseSuffix = resolveStagePhaseSuffix(funnel.type, body.stageType, body.name);
const finalCampaigns =
  body.campaigns.length > 0
    ? body.campaigns
    : await findMatchingCampaignsForStage(fastify.db, funnel, phaseSuffix);
```

`findMatchingCampaignsForStage`:
- Se `phaseSuffix` é null → ainda retorna campanhas que matcham `matchCode` (igual sugestão de funil hoje)? Ou retorna `[]`?
- **Decisão:** retorna `[]` quando `phaseSuffix` é null. Auto-popular só quando heurística é confiante. Senão usuário escolhe manual.

### Onde atualizar `byStage`

`funnels.ts:539-551`:
```ts
for (const stage of stages) {
  const phaseSuffix = resolveStagePhaseSuffix(funnel.type, stage.stageType, stage.name);
  const selectedHere = new Set((stage.campaigns ?? []).map((c) => c.id));
  const orphansHere = matching
    .filter((c) => !selectedHere.has(c.id))
    .filter((c) => !phaseSuffix || c.name.toLowerCase().includes(phaseSuffix))
    .sort(...);
  byStage[stage.id] = { stageName: stage.name, orphans: orphansHere };
}
```

### Performance

- `findMatchingCampaignsForStage` reaproveita `fetchCampaigns(metaAccountId, accessToken)` (já existe). Não bate `/insights` — só lista de campanhas, sem time_range. Bem mais leve que `fetchAllAdInsights`.
- Cache de 15 min do `getCached`/`setCache` pode ser aproveitado adicionando uma chave nova.
