## 📖 Story

**Como** desenvolvedor backend do dashboard de Assinaturas (Epic 34),
**Quero** um service `hotmart.ts` que autentique via OAuth2 client_credentials, consuma as APIs de assinaturas/vendas da Hotmart e calcule todas as métricas de recorrência (MRR, LTV, LT, retenção, churn, renovações),
**Para** que as rotas (Story 34.3) tenham uma camada de serviço pronta, testada e segura, sem reinventar autenticação, paginação ou as fórmulas de métricas em cada endpoint.

---

## ✅ Acceptance Criteria

1. **Arquivo `packages/api/src/services/hotmart.ts` criado** exportando todas as funções do contrato de service (ver Dev Notes), com tipos TypeScript explícitos (sem `any` implícito).

2. **Autenticação OAuth2 `client_credentials`:**
   - `getHotmartToken(clientId, clientSecret)` faz `POST https://api-sec-vlc.hotmart.com/security/oauth/token?grant_type=client_credentials&client_id=..&client_secret=..` com header `Authorization: Basic base64(clientId:clientSecret)`.
   - Resposta `{ access_token, token_type, expires_in }` — só o `access_token` é usado em chamadas seguintes (`Authorization: Bearer <access_token>`).
   - Cache em memória por par de credenciais (Map), com refresh automático ~5 min (300s) **antes** do `expires_in`. Token reutilizado enquanto válido.
   - Em falha de auth, lança `Error` com mensagem legível **sem** vazar `clientSecret` nem o Basic.

3. **Cliente HTTP genérico `hotmartGet(token, path, params)`:**
   - Base `https://developers.hotmart.com/payments/api/v1`.
   - Monta querystring a partir de `params` (ignora `undefined`/`null`/`""`).
   - Header `Authorization: Bearer <token>`, `Accept: application/json`.
   - Em `!res.ok`, lança `Error` com status e detalhe quando disponível.

4. **`fetchSubscriptionCount(token, { productId, status, accessionFrom, accessionTo })`:**
   - Chama `/subscriptions` com `product_id`, `status` (opcional), `accession_date`, `end_accession_date`, `max_results` mínimo.
   - Retorna `page_info.total_results` (contagem sem baixar todos os itens).

5. **`fetchSubscriptionsSummary(token, { productId, accessionFrom })`:**
   - Chama `/subscriptions/summary` com auto-paginação via `page_token`/`next_page_token` (cursor, não offset) até esgotar.
   - Retorna o array completo de `items` (`lifetime`, `status`, `plan`, `product`, `price`, `date_next_charge`, ...).

6. **`fetchSalesSummaryByStatus(token, { productId, status, from, to })`:**
   - Chama `/sales/summary` com `transaction_status` (quando informado), `product_id`, `start_date`, `end_date`.
   - Retorna `items` agrupados por moeda (`{ total_items, total_value: { value, currency_code } }`).
   - Documenta a pegadinha: **sem** `transaction_status` a API retorna só APPROVED+COMPLETE.

7. **`listHotmartProducts(token, accessionFrom)`:**
   - Deriva produtos **distintos** (`{ id, name }`) varrendo `/subscriptions/summary` (o endpoint v2 de produtos retorna 401 com esta auth — derivar é robusto).
   - Sem duplicatas; ordenado por `name`.

8. **`computeHotmartDashboard(token, { productId, months })`** retorna objeto com **todas** as métricas do contrato, calculadas pelas fórmulas exatas (ver Dev Notes), agrupadas por moeda quando aplicável (MRR, LTV, reembolsos), primária BRL:
   - Total de assinaturas, Vigentes/Ativas, Canceladas, Inadimplentes, Reembolsadas, MRR, LTV, LT (meses), Retenção %, Churn %, Renovações do próximo mês (contagem + receita prevista), Distribuição por status.

9. **Helpers de cripto** `encryptHotmartSecret` / `decryptHotmartSecret` reusam `services/encryption.ts` (AES-256-GCM), seguindo o padrão `mauticConnections`. O `Basic` é **derivado em runtime**, nunca armazenado.

10. **Helpers de data** convertem corretamente epoch **milissegundos UTC** ↔ os formatos esperados pela API (`accession_date` etc.), e calculam a janela do **próximo mês** (1º dia 00:00 → último dia 23:59:59).

11. **Testes unitários** das fórmulas e conversões passam (`pnpm --filter @loyola-x/api test`), cobrindo: MRR mensalizado com `recurrency_period` em dias (30=mensal, 360=anual), LT, LTV, janela de renovações do próximo mês, conversão epoch ms, agregação multi-moeda, contagem de canceladas/inadimplentes.

12. **Segurança:** nenhum log/throw expõe `clientSecret` nem o Basic. PII do `subscriber` nunca é logada.

## 📊 Complexity Estimate

**Story Points:** 8 (ou **Large**)
**Estimated Duration:** 2-3 dias

**Rationale:**
- OAuth2 client_credentials com cache/refresh é estado mutável testável (Map por credencial).
- Auto-paginação por cursor exige laço robusto contra loop infinito.
- ~9 métricas com fórmulas próprias (Hotmart não expõe MRR/LTV/LT prontos) — risco aritmético.
- Multi-moeda em MRR/LTV/reembolsos (agrupar por `currency_code`).
- Conversão epoch ms e janela do próximo mês são fontes clássicas de off-by-one.
- Padrões já existem (encryption.ts + mautic.ts), o que reduz incerteza estrutural.

## ⚠️ Risks & Mitigations

| Risk | Severity | Mitigation |
|------|----------|-----------|
| **Vazamento de `clientSecret`/Basic em log ou mensagem de erro** | HIGH | Mensagens de erro genéricas (status + detalhe da Hotmart, nunca a credencial). Teste garante que `Error.message` não contém o secret. PII do subscriber nunca logada. |
| **Divisão por zero em MRR/LTV/LT/retenção (sem assinaturas ACTIVE ou total=0)** | HIGH | Guardas explícitas: retorno `0` (ou `null` documentado) quando denominador = 0. Testes cobrem dataset vazio. |
| **`/subscriptions` default = últimos 30 dias de adesão → métricas subestimadas** | HIGH | SEMPRE passar `accession_date` (default 12 meses atrás → hoje). Documentado em Dev Notes e coberto por chamada que sempre envia a janela. |
| **Off-by-one na janela de renovações do próximo mês / epoch ms** | MEDIUM | Helper dedicado `nextMonthWindow()` (1º dia 00:00:00.000 → último dia 23:59:59.999, UTC). Testes com datas fixas (mês com 28/30/31 dias, virada de ano). |
| **Loop infinito na auto-paginação se `next_page_token` não zerar** | MEDIUM | Guarda de máximo de páginas + break quando `next_page_token` ausente/igual ao anterior. |
| **MRR errado por não mensalizar `recurrency_period` (dias)** | MEDIUM | Fórmula fixa `value * 30 / recurrency_period`. Teste com plano anual (360) e mensal (30). |
| **Agregação multi-moeda misturando BRL+USD num só número** | MEDIUM | Sempre agrupar por `currency_code`; BRL primária. Teste com 2 moedas. |

## 🔧 Dev Notes

### Contexto Epic 34 (decisões travadas)
- Aba **unificada "Assinaturas"** (Hotmart agora, Kiwify depois).
- Produto escolhido via **dropdown** no dashboard.
- Foco = **assinaturas/recorrência** (espelha o "Dashboard de assinaturas" da Hotmart).

### Autenticação Hotmart (confirmada ao vivo)
- OAuth2 `client_credentials`.
- `POST https://api-sec-vlc.hotmart.com/security/oauth/token?grant_type=client_credentials&client_id=..&client_secret=..`
  header `Authorization: Basic base64(client_id:client_secret)`.
- Resposta `{ access_token, token_type: "bearer", expires_in (segundos) }`.
- Só o `access_token` expira → cachear e renovar ~5 min antes do `expires_in`.
- Chamadas de dados: `Authorization: Bearer <access_token>`.
- Base das APIs de dados: `https://developers.hotmart.com/payments/api/v1`.

### Endpoints (confirmados ao vivo)
- `GET /payments/api/v1/subscriptions?product_id=&status=&accession_date=&end_accession_date=&date_next_charge=&end_date_next_charge=&max_results=&page_token=`
  → `{ items: [...], page_info: { total_results, next_page_token } }`.
  Contagem por status SEM baixar tudo: chamar com `status=` e ler `page_info.total_results`.
  Item: `{ subscriber_code, subscription_id, status, accession_date(ms), date_next_charge(ms), trial, plan{name,id,recurrency_period(DIAS),max_charge_cycles}, price{value,currency_code}, product{id,name,ucode}, subscriber (PII — NÃO logar) }`.
- `GET /payments/api/v1/subscriptions/summary?product_id=&accession_date=&max_results=&page_token=`
  → items com `{ lifetime (nº de ciclos pagos), status, plan{name,recurrency_period}, product{id,name}, last_recurrency, date_next_charge }`. `lifetime` = base de LT/LTV.
- `GET /payments/api/v1/sales/summary?transaction_status=&product_id=&start_date=&end_date=`
  → `items: [{ total_items, total_value{value,currency_code} }]` (1 por moeda).
  PEGADINHA: SEM `transaction_status` retorna só APPROVED+COMPLETE. Reembolsos = `transaction_status=REFUNDED`.
  Status de venda: APPROVED, REFUNDED, PARTIALLY_REFUNDED, CHARGEBACK, CANCELLED, COMPLETE.

### Pegadinhas críticas (confirmadas)
- (a) `/subscriptions` DEFAULT = últimos 30 dias de adesão → SEMPRE passar `accession_date` (default do dashboard = 12 meses atrás até hoje).
- (b) Datas são **epoch MILISSEGUNDOS UTC**.
- (c) `recurrency_period` em **DIAS** (30=mensal, 360=anual).
- (d) Multi-moeda em `sales/summary` (agrupar por `currency_code`; primária BRL).
- (e) Paginação por `page_token`/`next_page_token` (cursor, não offset).
- (f) STATUS de assinatura: ACTIVE, INACTIVE, DELAYED, OVERDUE, STARTED, CANCELLED_BY_CUSTOMER, CANCELLED_BY_SELLER, CANCELLED_BY_ADMIN.

### Métricas — Hotmart NÃO expõe LTV/MRR/LT prontos → CALCULAR (documentar como aproximação nossa)
- **Total assinaturas** = `page_info.total_results` (sem status, com `accession_date` do período).
- **Vigentes/Ativas** = `total_results` com `status=ACTIVE`.
- **Canceladas** = `CANCELLED_BY_CUSTOMER` + `CANCELLED_BY_SELLER` + `CANCELLED_BY_ADMIN`.
- **Inadimplentes** = `OVERDUE` + `DELAYED`.
- **Reembolsadas** = `sales/summary?transaction_status=REFUNDED` (`total_items` + `total_value`).
- **MRR** = soma sobre assinaturas ACTIVE de `price.value * 30 / plan.recurrency_period` (mensalizar). Por moeda.
- **LT (meses)** = média de `(lifetime * recurrency_period / 30)` das assinaturas do summary.
- **LTV** = média de `(price.value * lifetime)` por assinante (do summary), por moeda. (alternativa: ticket médio × LT.)
- **Retenção %** = ativas / total ; **Cancelamento%/churn** = canceladas / total.
- **Renovações próximo mês** = assinaturas ACTIVE com `date_next_charge` dentro de `[1º dia 00:00 .. último dia 23:59:59]` do próximo mês → contagem + soma de `price.value` (receita prevista).
- **Distribuição de status** = contagem por cada status (tabela do dashboard Hotmart).
- **Produtos (dropdown)** = DERIVAR distinct `product{id,name}` varrendo `/subscriptions/summary`.

### Contrato de dados — tabela `hotmart_connections` (Story 34.1, referência)
> A tabela é criada na Story 34.1. Este service apenas consome/produz os valores criptografados.

`id uuid pk default gen_random_uuid; project_id uuid NOT NULL UNIQUE FK->projects(id) ON DELETE cascade; client_id_encrypted text NOT NULL; client_id_iv text NOT NULL; client_secret_encrypted text NOT NULL; client_secret_iv text NOT NULL; created_at timestamptz default now; updated_at timestamptz default now. index idx_hotmart_connections_project on (project_id).`
Basic = `base64(client_id:client_secret)` é DERIVADO em runtime, NÃO armazenado. Reusar `services/encryption.ts` `encrypt`/`decrypt` (AES-256-GCM), padrão `mauticConnections`.

### Contrato de service (`packages/api/src/services/hotmart.ts` — CRIAR)
Funções (assinaturas-guia; tipos exatos a critério do dev, sem `any` implícito):

```ts
// Cripto (reusa services/encryption.ts)
export function encryptHotmartSecret(plaintext: string): { encrypted: string; iv: string };
export function decryptHotmartSecret(encrypted: string, iv: string): string;

// Auth — cache em memória (Map por par de credenciais), refresh 5 min antes do expires_in
export async function getHotmartToken(clientId: string, clientSecret: string): Promise<string>;

// Client HTTP genérico — base developers.hotmart.com/payments/api/v1, Bearer
export async function hotmartGet<T>(token: string, path: string, params?: Record<string, string | number | undefined>): Promise<T>;

// Fetchers
export async function fetchSubscriptionCount(token: string, args: { productId: string; status?: string; accessionFrom: number; accessionTo?: number }): Promise<number>; // page_info.total_results
export async function fetchSubscriptionsSummary(token: string, args: { productId?: string; accessionFrom: number }): Promise<HotmartSummaryItem[]>; // auto-paginação
export async function fetchSalesSummaryByStatus(token: string, args: { productId: string; status?: string; from: number; to: number }): Promise<HotmartSalesSummaryItem[]>; // por moeda

// Derivação de produtos (endpoint v2 dá 401 — derivar das assinaturas)
export async function listHotmartProducts(token: string, accessionFrom: number): Promise<Array<{ id: string; name: string }>>;

// Agregação de métricas (todas as fórmulas acima)
export async function computeHotmartDashboard(token: string, args: { productId: string; months: number }): Promise<HotmartDashboard>;
```

Forma sugerida do `HotmartDashboard` (chaves estáveis — a rota 34.3 e o front 34.4 dependem):
```ts
interface MoneyByCurrency { currency: string; value: number } // currency = currency_code; BRL primária
interface HotmartDashboard {
  totalSubscriptions: number;
  activeSubscriptions: number;     // ACTIVE
  cancelledSubscriptions: number;  // CANCELLED_BY_*
  overdueSubscriptions: number;    // OVERDUE + DELAYED
  refunded: { totalItems: number; totalValue: MoneyByCurrency[] };
  mrr: MoneyByCurrency[];          // por moeda
  ltv: MoneyByCurrency[];          // por moeda
  ltMonths: number;                // LT médio em meses
  retentionRate: number;           // 0..1 (ativas/total)
  churnRate: number;               // 0..1 (canceladas/total)
  nextMonthRenewals: { count: number; expectedRevenue: MoneyByCurrency[] };
  statusDistribution: Array<{ status: string; count: number }>;
  currencyPrimary: string;         // "BRL"
}
```

### Helpers de data
- `nowMs()` → `Date.now()`.
- `monthsAgoMs(months)` → epoch ms de `hoje − months` (default dashboard: `months=12`).
- `nextMonthWindow(refMs?)` → `{ startMs, endMs }` do próximo mês em UTC: 1º dia 00:00:00.000 e último dia 23:59:59.999. Cobrir meses de 28/30/31 dias e virada de ano. Usar `Date.UTC(...)`.
- Lembrar: API espera/retorna epoch **ms UTC** — não converter pra segundos.

### Padrões a reusar (NÃO reinventar)

- **`services/mautic.ts`** — padrão de service de integração: `normalizeBaseUrl`, `fetch` com header de auth, tratamento de `!res.ok` lendo `errors[0].message`, helper `mapWithConcurrency` (se precisar paralelizar contagens por status), pattern de fetcher tipado. - **Tests** ficam em `packages/api/src/__tests__/*.test.ts` (Vitest). Espelhar `encryption.test.ts` (setup de `ENCRYPTION_KEY` em `beforeAll`/`afterAll`).
