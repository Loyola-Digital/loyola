## User Story

Como **desenvolvedor da plataforma**, preciso de um endpoint API e schema de banco que permita **conectar uma planilha Google Sheets de vendas a um funil de tipo perpétuo** (uma planilha por funil, sem stage), com mapper de colunas igual ao usado em etapas pagas, para que stories subsequentes (29.2–29.5) possam construir UI e dashboards em cima.

---

## Acceptance Criteria

- [x] **AC1:** Enum `funnel_spreadsheet_type` recebe novo valor `"perpetual_sales"` (migration que faz `ALTER TYPE ... ADD VALUE`).
- [x] **AC2:** Type do `columnMapping` em `funnelSpreadsheets` é estendido para incluir `valorBruto`, `valorLiquido`, `formaPagamento` (campos que hoje só existem em `stageSalesSpreadsheets`).
- [x] **AC3:** Endpoint `GET /api/projects/:projectId/funnels/:funnelId/perpetual-spreadsheet` retorna a planilha conectada ao funil (1 por funil) ou `null` se nenhuma.
- [x] **AC4:** Endpoint `POST /api/projects/:projectId/funnels/:funnelId/perpetual-spreadsheet` cria ou atualiza (upsert) a planilha do funil. Body: `{ spreadsheetId, spreadsheetName, sheetName, columnMapping }`. Quando já existe planilha `perpetual_sales` pro funil, atualiza; senão cria. `stageId` é sempre `NULL` pra esse tipo.
- [x] **AC5:** Endpoint `DELETE /api/projects/:projectId/funnels/:funnelId/perpetual-spreadsheet` remove a planilha conectada.
- [x] **AC6:** Todos os endpoints validam membership do user no projeto (mesma pattern de `stage-sales-spreadsheets.ts`). Guests recebem 403.
- [x] **AC7:** `columnMapping.email` é obrigatório (Zod `.min(1)`); demais campos são opcionais.
- [x] **AC8:** Upsert chama `clearSheetDataCache(spreadsheetId, sheetName)` ao criar/atualizar pra forçar resync no próximo read.
- [x] **AC9:** Rotas registradas em `packages/api/src/app.ts` via `fp()` closure.
- [x] **AC10:** `pnpm typecheck` passa em packages/api sem erros.

## Dev Notes

### Pattern de referência

`packages/api/src/routes/stage-sales-spreadsheets.ts` é o template mais próximo (mesmo mapper de colunas, mesma estrutura de auth/membership, mesmo `clearSheetDataCache`).

Diferenças-chave a aplicar:

| Aspecto | `stage-sales` | `perpetual-spreadsheets` |
|---|---|---|
| Tabela | `stageSalesSpreadsheets` | `funnelSpreadsheets` (reuso) |
| Escopo | por `stageId` | por `funnelId` |
| Subtype | `capture` / `main_product` / `sales` | N/A (sempre `perpetual_sales` no campo `type`) |
| Cardinalidade | N por stage | 1 por funil |
| `stageId` | obrigatório | sempre `NULL` |

### Por que reusar `funnelSpreadsheets`?

Decisão validada com Lucas: evita criar nova tabela + reusa toda a infra de sync (`google-sheets.ts`), cache (`clearSheetDataCache`), e o type union de `columnMapping` que já está bem estabelecido.

### Convenção do enum (cuidado Postgres)

`ALTER TYPE ... ADD VALUE` precisa rodar fora de transação. Os scripts de migration 0034/0035/0036 já fazem isso (rodam um `await client.query(...)` direto). Seguir o mesmo padrão.
