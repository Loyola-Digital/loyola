## Story

**As a** Loyola Digital employee chatting with an AI Mind,
**I want** to upload a document (.txt, .docx, .pdf) alongside my message,
**so that** the mind receives the document content as context and can analyze, summarize, or reference it in the conversation.

---

## Acceptance Criteria

1. [ ] `POST /api/upload` endpoint accepts multipart/form-data with a single file field
2. [ ] Supported MIME types: `text/plain`, `application/vnd.openxmlformats-officedocument.wordprocessingml.document` (.docx), `application/pdf`
3. [ ] Returns 400 for unsupported file types with clear error message
4. [ ] Maximum file size: 10MB raw file, 200KB extracted text (truncated with notice if exceeded)
5. [ ] Text extraction: `.txt` read as UTF-8, `.docx` via `mammoth`, `.pdf` via `pdf-parse`
6. [ ] Response returns `{ filename, mimeType, textLength, extractedText }` as JSON
7. [ ] Endpoint requires authentication (Clerk JWT, same as /api/chat)
8. [ ] Rate limit: 10 uploads/min/user
9. [ ] `messages.metadata` type updated to include optional `attachments` array: `{ filename, mimeType, textLength }`
10. [ ] `POST /api/chat` request schema updated to accept optional `attachmentContext` string field (max 200000 chars)
11. [ ] When `attachmentContext` is present, it's prepended to the user message as `[Documento: {filename}]\n{text}\n\n---\n\n{user message}`
12. [ ] Persisted message `content` includes the full document text (for future reference)
13. [ ] Persisted message `metadata.attachments` stores file metadata (not the full text)
14. [ ] Error handling: corrupted files return 422 with `"Could not extract text from file"`
15. [ ] Unit tests for text extraction (txt, docx, pdf) and endpoint validation

## Dev Notes

### Architecture Decisions

- **No file storage**: Files are NOT stored permanently. Text is extracted in-memory and returned to the frontend. The frontend then sends the extracted text with the chat message. This keeps the architecture simple — no S3/R2 needed.
- **Text in message content**: The full document text is stored as part of the message `content` field. This means it's available when loading conversation history and will be sent to Claude as context in future messages within the same conversation.
- **Metadata for display**: The `metadata.attachments` array stores only file metadata (name, type, length) for UI display purposes — not the full extracted text (that's in `content`).

### API Contract

**Request:**
```
POST /api/upload
Content-Type: multipart/form-data
Authorization: Bearer <clerk-jwt>

file: <binary file data>
```

**Response (200):**
```json
{
  "filename": "briefing-campanha.docx",
  "mimeType": "application/vnd.openxmlformats-officedocument.wordprocessingml.document",
  "textLength": 4523,
  "extractedText": "Briefing da campanha..."
}
```

**Error (400):**
```json
{
  "error": "Unsupported file type. Accepted: .txt, .docx, .pdf"
}
```

**Error (422):**
```json
{
  "error": "Could not extract text from file"
}
```

### Chat Request Schema Update

Current:
```typescript
{ mindId: string, conversationId: string | null, message: string }
```

Updated:
```typescript
{
  mindId: string,
  conversationId: string | null,
  message: string,
  attachmentContext?: string,   // extracted text from upload
  attachmentMeta?: {            // file metadata for persistence
    filename: string,
    mimeType: string,
    textLength: number,
  }
}
```
