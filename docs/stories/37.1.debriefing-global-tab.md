## 📖 Story

**Como** membro do time Loyola (qualquer usuário que NÃO seja guest),
**Quero** uma aba global "Debriefing" onde cadastro o nome da campanha e subo um documento HTML de debriefing, que é renderizado fielmente na plataforma, com opção de editar o doc (caso tenha dado errado) e de deixar comentários visíveis a todos com autoria,
**Para** centralizar os debriefings de campanhas na plataforma, com histórico, discussão do time e rastreabilidade de quem comentou o quê.

---

## ✅ Acceptance Criteria

1. **Schema — 2 tabelas novas em `packages/api/src/db/schema.ts`** (globais, sem `projectId`):
   - `debriefings`: `id` uuid PK, `campaignName` text NOT NULL, `html` text NOT NULL (conteúdo do doc), `fileName` text (nome original do arquivo), `createdBy` uuid → `users.id` (`onDelete: "restrict"`), `createdAt` timestamptz default now, `updatedBy` uuid → `users.id` (nullable), `updatedAt` timestamptz default now. Index em `createdAt`.
   - `debriefingComments`: `id` uuid PK, `debriefingId` uuid → `debriefings.id` (`onDelete: "cascade"`), `userId` uuid → `users.id` (`onDelete: "restrict"`), `text` text NOT NULL, `createdAt` timestamptz default now. Index em (`debriefingId`, `createdAt`).
   - Migration gerada via drizzle-kit, **aditiva** (só CREATE TABLE). ⚠️ Ver risco "DB dev = prod" abaixo.

2. **API — `packages/api/src/routes/debriefings.ts`** (registrado em `app.ts`):
   - `GET /api/debriefings` → lista (sem o campo `html` — payload leve): id, campaignName, fileName, autor (nome/avatar), datas, contagem de comentários. Ordenado por `createdAt` desc.
   - `GET /api/debriefings/:id` → detalhe completo com `html`.
   - `POST /api/debriefings` → multipart (`request.file()`, padrão de `upload.ts`): campos `campaignName` + arquivo. Aceita apenas `text/html` (e `.html`/`.htm` por extensão como fallback de mimetype); limite 5MB; 400 se faltar nome ou arquivo, 413 se exceder.
   - `PUT /api/debriefings/:id` → edição: aceita JSON `{ campaignName?, html? }` **ou** multipart com novo arquivo (substitui o `html`). Atualiza `updatedBy`/`updatedAt`.
   - `DELETE /api/debriefings/:id` → remove (comentários caem em cascade).
   - `GET /api/debriefings/:id/comments` → lista com nome + avatar do autor.
   - `POST /api/debriefings/:id/comments` → cria comentário (`text` não-vazio, `userId` = request.userId).
   - `DELETE /api/debriefings/:id/comments/:commentId` → apenas o **autor do comentário** pode deletar (403 caso contrário).

3. **Bloqueio de guests (server-side):** guests recebem 403 em **qualquer** método/rota `/api/debriefings*`. Adicionar o bloqueio na seção "Block global routes" do `guest-guard.ts` (prefixo `startsWith("/api/debriefings")`, todos os métodos). Teste em `guest-guard.test.ts`.

4. **Sidebar:** item "Debriefing" (ícone `FileText` de lucide) na seção **Global** do `app-sidebar.tsx` (`navItems`, antes de Settings), href `/debriefings`. Guests usam `GuestSidebar` e não veem o item.

5. **Página lista — `app/(app)/debriefings/page.tsx`:**
   - Guard client-side igual à `sprint-dashboard/page.tsx`: `useUserRole() === "guest"` → mensagem de acesso restrito (defesa em profundidade junto com AC 3).
   - Lista dos debriefings (card ou tabela): nome da campanha, autor, data, nº de comentários; click abre o detalhe.
   - Botão "Novo debriefing" → dialog com input "Nome da campanha" + file picker (aceita `.html,.htm`); toast de sucesso/erro; invalida a query da lista.
   - Empty state amigável quando não há debriefings.

6. **Página detalhe — `app/(app)/debriefings/[id]/page.tsx`:**
   - **Renderização fiel do HTML**: `<iframe srcDoc={html} sandbox="allow-scripts" ...>` — SEM `allow-same-origin` (o doc roda isolado: sem acesso a cookies/sessão/DOM da plataforma, mas scripts internos do doc — gráficos, etc. — funcionam). Iframe com altura generosa/auto-ajustável e largura total.
   - Header: nome da campanha, autor, "criado em"/"editado em" (quando houver edição, mostrar quem editou).
   - **Editar**: botão abre dialog com (a) edição do nome da campanha, (b) re-upload de novo arquivo HTML substituindo o atual, e (c) editor do código-fonte HTML (textarea monoespaçada) para correções pontuais. Salvar → PUT + invalidação + toast.
   - **Excluir**: com confirmação (AlertDialog).

7. **Comentários (na página detalhe, abaixo do doc):**
   - Lista de comentários com avatar, **nome do autor** e data/hora — todos os não-guests veem quem deixou cada comentário.
   - Campo de novo comentário (textarea + botão), otimista ou com invalidação; autor pode excluir o próprio comentário.

8. **Hooks — `lib/hooks/use-debriefings.ts`** (React Query + `useApiClient`, espelha padrão de `use-kiwify.ts`): `useDebriefings()`, `useDebriefing(id)`, `useCreateDebriefing()`, `useUpdateDebriefing(id)`, `useDeleteDebriefing()`, `useDebriefingComments(id)`, `useAddDebriefingComment(id)`, `useDeleteDebriefingComment(id)`.

9. **Qualidade:** `pnpm --filter @loyola-x/api typecheck` + testes da API passam; `pnpm --filter @loyola-x/web build` passa (⚠️ não usar value import de `@loyola-x/shared` no web — inline dos tipos/helpers no web se necessário); lint ok.

## ⚠️ Risks & Mitigations

| Risk | Severity | Mitigation |
|------|----------|-----------|
| HTML arbitrário com scripts (XSS na plataforma) | HIGH | Nunca usar `dangerouslySetInnerHTML`. Renderizar SEMPRE em `<iframe sandbox="allow-scripts">` **sem** `allow-same-origin` — o doc não acessa cookies/sessão/origin da app. |
| Migration em banco de PRODUÇÃO (DB dev = prod) | HIGH | Migration puramente aditiva (2 CREATE TABLE, zero ALTER). Revisar SQL gerado antes de aplicar. Não rodar `drizzle-kit push` sem revisão. |
| Docs HTML grandes inflando payload da lista | MEDIUM | `GET /api/debriefings` NÃO retorna `html`; detalhe carrega sob demanda. Limite 5MB no upload. |
| Guest acessando rota global nova | MEDIUM | Bloqueio explícito no `guest-guard.ts` (AC 3) + guard client-side (AC 5) + teste automatizado. |
| Usuário subir arquivo não-HTML por engano | LOW | Validação de mimetype/extensão com erro claro; accept no file picker. |

### Decisões técnicas desta story

- **Armazenamento:** HTML como coluna `text` no Postgres (não filesystem/S3) — mesmo racional do snapshot do relatório mensal do Instagram; simples, transacional, sem infra nova. 5MB de teto mantém a tabela saudável.
- **Fidelidade de renderização:** `iframe srcDoc + sandbox="allow-scripts"` preserva CSS/JS internos do doc "igual ele é", isolado da app. Não sanitizar/reescrever o HTML (sanitização quebraria a fidelidade); o isolamento vem do sandbox.
- **Edição:** três formas no mesmo dialog — nome, re-upload (caso mais comum: gerar o doc de novo e substituir) e editor de código-fonte (correção rápida de um dado errado). Sem WYSIWYG no MVP.
- **Autorização de escrita:** qualquer não-guest pode criar/editar/excluir debriefings e comentar; excluir comentário é só do autor. Sem papel de "dono" do debriefing no MVP (time pequeno, edição colaborativa desejada).

### Project Structure Notes

- API: `packages/api/src/routes/debriefings.ts` + registro em `packages/api/src/app.ts` (seguir ordem alfabética/agrupamento existente).
- Web: `packages/web/app/(app)/debriefings/page.tsx`, `packages/web/app/(app)/debriefings/[id]/page.tsx`, componentes em `packages/web/components/debriefings/` (dialog de criação/edição, seção de comentários, iframe viewer).
