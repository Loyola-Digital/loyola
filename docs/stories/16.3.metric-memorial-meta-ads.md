## Story

Como usuário do dashboard de tráfego (Meta Ads), quero ver o memorial de cálculo dos KPI cards agregados (spend, impressions, reach, frequency, clicks, CTR, leads, CPL, qualified, CPL Qual, sales, ROAS) e do gráfico diário de spend/clicks, para defender resultados de campanha em reuniões com clientes — **preservando o design atual dos cards e tabelas**.

### Diferenças vs 16.2 (importantes)

1. **Filtro de período é `days: number`** (não `PeriodConfig.since/until`). Factories Meta Ads precisam aceitar `{ days }` e gerar string tipo "Últimos 30 dias" no memorial.
2. **Cards são condicionais** por flags `hasCrm`, `hasQualification`, `hasSales` (vindas de `CampaignAnalyticsResponse`). Total possível: 12 cards na page global, 6 fixos na page por empresa.
3. **Não há filtro "campanhas selecionadas"** — `SummaryCards` soma todas as campanhas do accountId/projectId ativo. O `campaignId` existe no código mas é usado só para drill-down visual da tabela, não para filtrar o SummaryCards.
4. **KpiCard inline no arquivo monolítico** — não é componente separado. Mantém o mesmo visual do Instagram (rounded-xl + gradiente `card/80 → card/40` + border `border-border/30`), mas **sem gradientes coloridos por métrica**. Preserva-se wrap, não replace.

### KPI Cards reais no `SummaryCards`

| # | Label | Fórmula | Condicional | Fonte |
|---|-------|---------|:---:|-------|
| 1 | Spend | `Σ spend` de todas campaigns | sempre | Meta Ads API · `insights.spend` |
| 2 | Impressões | `Σ impressions` | sempre | Meta Ads API · `insights.impressions` |
| 3 | Alcance | `Σ reach` | sempre | Meta Ads API · `insights.reach` |
| 4 | Frequência | `Σ impressions ÷ Σ reach` | sempre (se reach>0) | Derivado |
| 5 | Cliques | `Σ clicks` | sempre | Meta Ads API · `insights.clicks` |
| 6 | CTR | `Σ clicks ÷ Σ impressions × 100` | sempre (se impressions>0) | Derivado |
| 7 | Leads | `Σ leads` por campaign | `hasCrm` | CRM externo / actions |
| 8 | CPL | `Σ spend ÷ Σ leads` | `hasCrm` (se leads>0) | Derivado |
| 9 | Qualificados | `Σ qualifiedLeads` | `hasQualification` | CRM externo |
| 10 | CPL Qual | `Σ spend ÷ Σ qualifiedLeads` | `hasQualification` (se >0) | Derivado |
| 11 | Vendas | `Σ sales` | `hasSales` | CRM externo |
| 12 | ROAS | `Σ revenue ÷ Σ spend` | `hasSales` (se spend>0) | Derivado |

### AC4: Formatação monetária BRL

## Dev Notes

- **Reaproveitar máximo da 16.2** — `<MetricTooltip>` e `<FormulaChartTooltip>` sem alterações. `formulasByKey` já suportado.
- **Não tocar** no `FunnelCampaignTable` nem na tabela inline (500+ linhas de tabela no page.tsx) — Scope OUT.
- **Formatação BRL:** centralizar em `meta-ads.ts`. Não usar formatadores do `traffic/page.tsx` (que já formatam o card) dentro das factories — as factories devem ser self-contained.
- **Nome da conta:** em `traffic/page.tsx` vive no state (accountId selecionado → buscar `accounts.find(a => a.id === accountId)?.name`). Em `projects/[id]/traffic/page.tsx` vem de `linkedAccount.name`.
- **Cuidado com números como string no `Meta Ads API`:** `spend`, `clicks` vêm frequentemente como string do Meta. Os reducers do código atual já fazem `Number()` implícito ou via `safeNum`. Factories recebem `number` limpo — não precisam de re-coerção.
