## Story

**As a** gestor de tráfego da Loyola Digital,
**I want** que os dados Meta (adsets, ads, campaigns, insights diários) sejam persistidos no Postgres e re-sincronizados via TTL,
**so that** o app não bata no rate limit da Meta API a cada reload, restart de backend ou novo funil aberto — e o histórico de dias passados não precise ser refetchado já que a Meta não muda atribuição depois de 7 dias.

---

## Contexto e Motivação

Hoje o backend cacheia respostas Meta em **memória** (`getCached`/`setCache`, TTL 15min):
- Restart do backend = cache zerado = reconsulta Meta de tudo
- Múltiplas instâncias = cada uma reconsulta separadamente
- A cada novo funil aberto = novo cache key = nova consulta Meta

Hotfix #173 (PR 2026-05-20) já reduziu de N→1 requests por funil, mas a próxima camada de ganho é parar de bater na Meta pra dados que já consultamos antes.

**Característica favorável:** metadata Meta é estável (adset_name muda raramente; insights de dias passados ficam estáveis após 7 dias da atribuição). Persistir no Postgres com TTL adequado por tipo de dado elimina a maior parte das chamadas.

**Decisões validadas com Lucas (2026-05-20):**
- Persistir 4 entidades: adsets, ads (+creative), campaigns, insights diários
- Estratégia: **Lazy + TTL no banco** (sem cron). Cada leitura checa `last_synced_at`; se stale, refetch e upsert.
- Prioridade: P1 (atacar agora)

---

## Faseamento

A story entrega valor incrementalmente em **3 fases independentes** — cada fase pode ser mergeada e validada antes da seguinte.

### Fase 1 — Adsets (resolve o tooltip de leads por adset)

Foco: tabela `meta_adsets` + service `getAdsetsByCampaigns` DB-first. Refactor `getAllAdSetsForProject` pra usar essa camada.

### Fase 2 — Ads + Creative metadata

Tabela `meta_ads` armazenando ad_id, name, adset_id, campaign_id, creative jsonb (image_url, thumbnail_url, video_id, title, body, link_url, cta_type, object_type). Refactor `fetchAdCreatives` + `fetchAllAdInsights` (parte metadata).

### Fase 3 — Insights diários (campaign-level + ad-level)

Tabela `meta_ad_insights_daily` (date, ad_id, spend, impressions, clicks, reach, actions jsonb, action_values jsonb) e `meta_campaign_insights_daily`. TTL diferenciado:
- Dias > 7 dias atrás: **infinito** (Meta não muda atribuição)
- Dias 1–7 atrás: 24h
- Dia atual: 30min

## Dev Notes

### Estrutura de TTL (constantes)

Em `packages/api/src/config/meta-sync.ts` (novo):

```ts
export const META_SYNC_TTL = {
  ADSET_METADATA: 24 * 60 * 60 * 1000, // 24h
  AD_METADATA: 24 * 60 * 60 * 1000,    // 24h
  INSIGHTS_TODAY: 30 * 60 * 1000,      // 30min
  INSIGHTS_RECENT: 24 * 60 * 60 * 1000, // 24h (dias 1-7 atrás)
  INSIGHTS_HISTORIC: Infinity,         // dias > 7 atrás (Meta não muda mais)
};
```

### Padrão de upsert (Drizzle)

```ts
await db.insert(metaAdsets).values(rows).onConflictDoUpdate({
  target: metaAdsets.adsetId,
  set: {
    adsetName: sql`excluded.adset_name`,
    campaignId: sql`excluded.campaign_id`,
    lastSyncedAt: sql`now()`,
  },
});
```

### Por que não cron?

Cron rodaria mesmo sem usuário usando o produto = consome rate limit Meta à toa. Lazy + TTL só consulta quando alguém realmente precisa do dado, e ainda assim só uma vez por janela.

### Por que não invalidar via webhook Meta?

A Meta tem webhooks mas eles têm setup complexo (validation, retry, secrets). Pra primeira iteração, TTL é mais simples e suficiente.

### Risco: cold start em conta nova

Primeira sync de uma conta com 1000+ ads pode demorar. **Mitigação Fase 1:** sincronizar só os campaign_ids passados como filter (não todo o pool da conta). Fases 2/3: mesmo padrão.

## Risks & Open Questions

- **R1:** Cold start em conta grande na primeira sync. Mitigação: filter por campaign_ids do funil (não conta inteira). Aceita-se latência maior na primeira vez por funil.
- **R2:** Race condition se 2 requests stale chegarem ao mesmo tempo — ambos vão refetchar. Mitigação: aceitar o overhead duplicado nesse caso raro. Se virar problema, adicionar lock via `pg_advisory_lock(hashtext(accountId))`.
- **R3:** Migração inicial: tabelas novas precisam de runtime que aplique. Verificar se `drizzle-kit push --force` no startup do container roda OK (Dockerfile já faz `npx drizzle-kit push --force && node dist/server.js`).
- **Q1:** Atribuição da Meta pode mudar até 28 dias depois (não só 7)? Confirmar — se sim, ajustar `TTL_INSIGHTS_RECENT` pra cobrir 28d. **Decisão default:** 7d (mais conservador pro lado da economia de API). Se houver discrepância detectada, ajustar.
- **Q2:** Política de retenção? Insights diários crescem ~N_ads × 30 rows/mês por funil. Pra 100 funis × 50 ads × 90 dias = 450k rows/conta. OK pra Postgres por anos. Mas devemos truncar após X meses? **Decisão Fase 3:** sem retenção automática agora; revisar quando tabela passar de 10M rows.
