## Contexto e MotivaĂ§ĂŁo

Hoje o backend cacheia respostas Meta em **memĂłria** (`getCached`/`setCache`, TTL 15min):
- Restart do backend = cache zerado = reconsulta Meta de tudo
- MĂșltiplas instĂąncias = cada uma reconsulta separadamente
- A cada novo funil aberto = novo cache key = nova consulta Meta

Hotfix #173 (PR 2026-05-20) jĂĄ reduziu de Nâ1 requests por funil, mas a prĂłxima camada de ganho Ă© parar de bater na Meta pra dados que jĂĄ consultamos antes.

**CaracterĂ­stica favorĂĄvel:** metadata Meta Ă© estĂĄvel (adset_name muda raramente; insights de dias passados ficam estĂĄveis apĂłs 7 dias da atribuiĂ§ĂŁo). Persistir no Postgres com TTL adequado por tipo de dado elimina a maior parte das chamadas.

**DecisĂ”es validadas com Lucas (2026-05-20):**
- Persistir 4 entidades: adsets, ads (+creative), campaigns, insights diĂĄrios
- EstratĂ©gia: **Lazy + TTL no banco** (sem cron). Cada leitura checa `last_synced_at`; se stale, refetch e upsert.
- Prioridade: P1 (atacar agora)
