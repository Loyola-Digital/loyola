# Story 19.9 — Lançamento manual de vendas PIX Direto

**Epic:** 19 — Funnel Stages
**Status:** Ready to Ship
**Priority:** High
**Estimate:** M
**Depende de:** 19.7 (sales-only stage type), projectMembers (já existe)

---

## User Story

Como gestor do Loyola X, quero **lançar manualmente vendas que aconteceram via PIX direto** (fora da plataforma de checkout, sem entrar na planilha do Google Sheets), preenchendo nome/email/telefone do cliente, valor e o vendedor responsável (puxado dos membros do projeto), para que essas vendas apareçam no dashboard da etapa **Vendas** em uma seção **separada** ("Vendas PIX Direto") logo abaixo das vendas vindas da planilha — assim o time enxerga: (1) vendas normais da plataforma e (2) vendas PIX direto, sem misturar fontes.

---

## Contexto Técnico

### Modelo atual

- Etapa `stage_type = "sales"` (Story 19.7) renderiza `SalesStageView` em `packages/web/components/funnels/sales-stage-view.tsx`
- Vendas vem 100% da planilha Google Sheets (`stage_sales_spreadsheets` + `readSheetData`)
- Hoje **não existe** entrada manual no DB — toda venda é lida do Sheets
- `projectMembers` (schema.ts:305) é a fonte dos usuários do projeto (owner / admin / member / guest)

### Mudança proposta

- Nova tabela `manual_sales` no Postgres pra persistir vendas lançadas manualmente
- Endpoints REST CRUD em `/api/projects/:projectId/funnels/:funnelId/stages/:stageId/manual-sales`
- Botão "Lançar venda manual" no header de `SalesStageView` (apenas quando `stage.stageType === "sales"`)
- Modal `ManualSaleDialog` com form: nome cliente, email (opcional), telefone (opcional), valor, vendedor (Select de `projectMembers`), data
- Nova seção `ManualPixSalesSection` no dashboard, **abaixo** das `StageSalesSection` existentes, com seus próprios cards/tabela/ranking de vendedores

### Decisão arquitetural — vendedor

- Campo `seller_user_id` (FK opcional pra `users`) **+** `seller_name` (snapshot do nome)
- Mantém histórico íntegro se o usuário sair do projeto ou ser deletado (cascade SET NULL na FK, mas nome persiste)
- Frontend popula dropdown a partir de `projectMembers` → name resolvido via join com `users`

### Por que tabela nova (não reusar `stage_sales_spreadsheets`)

- Conceito diferente: planilha = fonte externa, manual = digitação interna no app
- Schema diferente: planilha guarda mapping de colunas, manual guarda os campos da venda em si
- Filtros e agregação diferentes: PIX direto fica em seção separada por requisito do usuário
- Reusar misturaria responsabilidades e exigiria criar "mapping fake"

## Acceptance Criteria

1. **AC1 — Schema:** Migration cria tabela `manual_sales` com colunas:
   - `id` uuid PK default random
   - `stage_id` uuid NOT NULL FK → `funnel_stages.id` ON DELETE CASCADE
   - `customer_name` varchar(255) NOT NULL
   - `customer_email` varchar(255) NULL
   - `customer_phone` varchar(50) NULL
   - `value` numeric(12,2) NOT NULL
   - `seller_user_id` uuid NULL FK → `users.id` ON DELETE SET NULL
   - `seller_name` varchar(255) NOT NULL (snapshot do nome do vendedor no momento do lançamento)
   - `sale_date` timestamptz NOT NULL
   - `created_by` uuid NOT NULL FK → `users.id` ON DELETE RESTRICT
   - `created_at` timestamptz NOT NULL DEFAULT now()
   - Index em `(stage_id, sale_date desc)`
   - Drizzle schema atualizado em `packages/api/src/db/schema.ts`

2. **AC2 — Botão "Lançar venda manual":** Visível **apenas** quando `stage.stageType === "sales"`. Renderizado no header do `SalesStageView`, ao lado do `DayRangePicker` e do botão `Configurar`. Ícone `Plus` (lucide-react) + label "Lançar venda manual".

3. **AC3 — Modal `ManualSaleDialog`:** Ao clicar no botão, abre um dialog com os campos:
   - Nome do cliente (text, required, min 2 chars)
   - Email do cliente (email, optional, validação básica se preenchido)
   - Telefone (text, optional, mask br opcional)
   - Valor da venda (currency input, required, > 0, formato pt-BR com R$)
   - Vendedor (Select, required, carregado de `projectMembers` do projeto via hook `useProjectMembers`)
   - Data da venda (DatePicker, required, default = hoje)
   - Botões: "Cancelar" / "Lançar venda" (primary)
   - Após sucesso: toast verde "Venda lançada", fecha modal, invalida cache de manual-sales e refetch
   - Após erro: toast vermelho com mensagem do backend

4. **AC4 — POST endpoint:** `POST /api/projects/:projectId/funnels/:funnelId/stages/:stageId/manual-sales`
   - Body Zod: `{ customerName, customerEmail?, customerPhone?, value, sellerUserId, saleDate }`
   - Valida que `sellerUserId` está em `projectMembers` do projeto (senão 403)
   - Resolve `seller_name` via join com `users.name`
   - Valida que `stage.stageType === "sales"` (senão 400 com erro claro)
   - Retorna 201 com a venda criada
   - Role `guest` → 403

5. **AC5 — GET endpoint (listar):** `GET /api/projects/:projectId/funnels/:funnelId/stages/:stageId/manual-sales?days=N`
   - Retorna array de vendas filtradas por `sale_date >= now() - N days` (default 30)
   - Ordenado por `sale_date` desc
   - Inclui agregação no response:
     ```ts
     {
       sales: ManualSale[];
       summary: {
         totalSales: number;
         totalRevenue: number;
         avgTicket: number;
         sellersRanking: Array<{
           sellerUserId: string | null;
           sellerName: string;
           totalSales: number;
           totalRevenue: number;
         }>;
       };
     }
     ```

6. **AC6 — DELETE endpoint:** `DELETE /api/projects/:projectId/funnels/:funnelId/stages/:stageId/manual-sales/:saleId`
   - Soft delete não é necessário — hard delete está OK pra v1
   - Role `guest` → 403
   - Retorna `{ success: true }`

7. **AC7 — Seção "Vendas PIX Direto" no dashboard:**
   - Componente novo `ManualPixSalesSection` (`packages/web/components/funnels/manual-pix-sales-section.tsx`)
   - Renderizado **abaixo** das `StageSalesSection` existentes (capture / main_product), **dentro** do `TabsContent value="dashboard"`
   - Header da seção: título "Vendas PIX Direto" + badge contador `{totalSales} vendas`
   - Mostra:
     - 3 cards no topo: Total Vendas, Faturamento Bruto, Ticket Médio
     - Tabela "Ranking de Vendedores" (vendedor, qtd vendas, faturamento, ticket médio) ordenada por faturamento desc
     - Tabela detalhada de vendas (data, cliente, email, telefone, valor, vendedor, ação de deletar — botão lixeira com confirmação)
   - Empty state: "Nenhuma venda PIX direto lançada nos últimos {days} dias" + CTA "Lançar venda manual" (mesmo botão que abre o modal)

8. **AC8 — Respeita filtro de período:** A seção `ManualPixSalesSection` usa o mesmo `days` do `DayRangePicker` do `SalesStageView` — quando user muda o filtro, refetch acontece automático.

9. **AC9 — NÃO mistura com vendas da planilha:** Os cards e tabelas das `StageSalesSection` (capture/main_product) continuam **idênticos** — manuais ficam isoladas na sua seção. Em nenhum momento manual entra no agregado de `StageSalesData` da planilha.

10. **AC10 — Hook frontend:** `useManualSales(projectId, funnelId, stageId, days)` em `packages/web/lib/hooks/use-manual-sales.ts` — React Query com `queryKey: ["manual-sales", projectId, funnelId, stageId, days]`.

11. **AC11 — Hook criar/deletar:** `useCreateManualSale(...)` e `useDeleteManualSale(...)` com mutations que invalidam a query acima após sucesso.

12. **AC12 — Sem regressão:** Etapas `free`, `paid` e `sales` continuam funcionando idêntico ao atual quando não há vendas manuais. Botão de venda manual NÃO aparece em `free` e `paid`.

13. **AC13 — Typecheck zero erros:** `pnpm -r typecheck` passa limpo em shared/api/web.

```ts
// POST body
type CreateManualSaleBody = {
  customerName: string;        // min 2 chars
  customerEmail?: string;      // email format se presente
  customerPhone?: string;
  value: number;               // > 0, numeric
  sellerUserId: string;        // uuid, deve estar em projectMembers do project
  saleDate: string;            // ISO date

// GET response
type GetManualSalesResponse = {
  sales: ManualSale[];
  summary: {
    totalSales: number;
    totalRevenue: number;
    avgTicket: number;
    sellersRanking: Array<{
      sellerUserId: string | null;
      sellerName: string;
      totalSales: number;
      totalRevenue: number;
    }>;
  };
};

type ManualSale = {
  id: string;
  stageId: string;
  customerName: string;
  customerEmail: string | null;
  customerPhone: string | null;
  value: number;
  sellerUserId: string | null;
  sellerName: string;
  saleDate: string;             // ISO
  createdBy: string;
  createdAt: string;            // ISO
};
```

```sql
-- Migration 0041_manual_sales.sql
CREATE TABLE IF NOT EXISTS manual_sales (
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  stage_id        uuid NOT NULL REFERENCES funnel_stages(id) ON DELETE CASCADE,
  customer_name   varchar(255) NOT NULL,
  customer_email  varchar(255),
  customer_phone  varchar(50),
  value           numeric(12,2) NOT NULL CHECK (value > 0),
  seller_user_id  uuid REFERENCES users(id) ON DELETE SET NULL,
  seller_name     varchar(255) NOT NULL,
  sale_date       timestamptz NOT NULL,
  created_by      uuid NOT NULL REFERENCES users(id) ON DELETE RESTRICT,
  created_at      timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX idx_manual_sales_stage_date ON manual_sales (stage_id, sale_date DESC);
```

## Risks

- **Vendedor deletado da plataforma:** FK `ON DELETE SET NULL` preserva a venda. `seller_name` snapshot garante que UI continua exibindo nome. Risco baixo.
- **Confusão com vendas da planilha:** Usuário pode achar que vendas manuais somam com planilha. Resolver com **título claro** "Vendas PIX Direto" + separação visual forte (border / spacing). Tooltip opcional.
- **Lançamento duplicado:** Não há dedupe automático. Usuário responsável por não lançar 2x. Aceitável pra v1.
- **Performance:** Tabela cresce no tempo. Index em `(stage_id, sale_date)` + filtro de `days` mantém queries leves. Sem paginação na v1 (até 1k vendas/etapa é OK).
- **`value` em numeric vs centavos int:** Optei por `numeric(12,2)` por consistência com `parseNumber` da planilha (que retorna float). Trade-off: pequena perda de precisão em ops aritméticas — aceitável pra valores monetários até R$ 10 bilhões.

- Padrão de RBAC seguido de `sellers-breakdown.ts` (helper `getStageContext` valida guest em `projectMembers` + confirma stage pertence ao funnel/projeto).
