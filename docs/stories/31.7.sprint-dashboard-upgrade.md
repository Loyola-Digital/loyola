# Story 31.7 — Sprint Dashboard Upgrade (Card Resumo + Tasks Atraso/Hoje)

**Epic:** 31 — Sprint Dashboard
**Status:** Done
**Priority:** Medium
**Estimate:** L (consolidada — 3 features em 1 story)
**Depende de:** 31.1, 31.2, 31.4, 31.6 (todas Done)

---

## User Story

Como gestor da Loyola, quero:
1. Na **Visão Geral**, ver um **card resumo executivo** por bloco (estilo "anúncio no canal Geral pós-sprint"). Conteúdo vem **automaticamente** das tasks do ClickUp marcadas com `📢` no título, OU eu preencho **manual** num botão "Contexto" — manual sempre sobrescreve.
2. No **Calendário Macro**, **embaixo de cada CampaignCard** ver as tasks daquele bloco que estão **em atraso ou com vencimento hoje** — assim vejo de relance o que precisa de ação imediata por campanha.

---

## Contexto Técnico

### Estado atual (Epic 31 base)

- `SprintDashboardBlock` (shared) tem `id`, `title`, `subtitle`, `color`, `clickupListIds`, `filters`, `groupBy`, `sortOrder`, `campaignPhases[]`.
- `SprintBlockCard` (web): grid de tasks ClickUp do bloco com checkbox/edit.
- `MacroCalendarView` → `CampaignCard`: renderiza `block.campaignPhases` como tags coloridas com status derivado das datas.
- Backend: cache 5min em-memória das tasks por listId combo.
- Tasks expostas: `{ id, name, status, statusColor, tags, dueDate, startDate, assignees, listId, listName, folderId, folderName }`.

### Mudança proposta

1. **Card Resumo (Visão Geral)**:
   - Adicionar campo `manualContext: string | null` em `SprintDashboardBlock` (persistido em `sprint_dashboard_config.blocks[].manualContext`).
   - Frontend computa `resolvedSummary` na ordem: (a) `manualContext` se truthy; (b) senão, pega tasks do bloco cujo `name` contém `📢` (qualquer posição) e usa nome+descrição da PRIMEIRA encontrada (ordem por `dueDate` ASC).
   - `SprintBlockCard` ganha header expandido com o resumo (quando existe) acima da lista de tasks.
   - Botão "Contexto" no header do bloco abre dialog `BlockContextDialog` com `Textarea` pra editar `manualContext`. Salva via PUT.
   - Se `manualContext` for `""` (string vazia) explícita, considera "limpou manual" → volta a olhar para tasks `📢`.

2. **Tasks Atraso/Hoje (Calendário Macro)**:
   - `CampaignCard` ganha section `<PendingTasksList>` ao fim com:
     - Filtro: `dueDate` parseável + `dueDate <= endOfToday` (overdue) OU `mesma data de hoje` (today).
     - Aplica também `block.filters` (statuses/tags/assignees) — coerente com o que aparece na Visão Geral.
     - Exclui tasks com status `done/closed/concluído` (helper `isDoneStatus` já existe em `sprint-block-card.tsx`).
   - Cada item: nome + status badge + due relativa ("Hoje" / "Há 2 dias") + border-l colorida (red pra overdue, amber pra today).
   - Render limitado a 5 tasks; se >5, mostra "+N pendentes" como link/expansor.
   - Empty state: "Nenhuma task em atraso ou pra hoje".

### Decisões trancadas com Lucas

| Pergunta | Resposta |
|----------|----------|
| Como sistema lê o card resumo | Emoji `📢` no nome da task |
| Botão Contexto manual | Sim, sempre disponível, sobrescreve auto |
| Filtro de tasks no Macro | Todas as tasks das listas do bloco, due_date <= hoje |
| Empacotar | 1 story consolidada |

### Layout

Layout dos cards a ser iterado após user enviar as 2 imagens de referência. Versão v1 usa estética existente do `CampaignCard` (border-rounded, bolinha de cor, badges compactos). Visual ajustável sem mudar funcional.

## Acceptance Criteria

1. **AC1 — Schema do bloco:** `SprintDashboardBlock` (shared) ganha `manualContext?: string`. Zod `blockSchema` aceita `manualContext: z.string().max(2000).optional().nullable()`. Backend persiste no `sprint_dashboard_config.blocks` (jsonb — sem migration nova).

2. **AC2 — Endpoint PUT bloco:** Reusa `PUT /api/sprint-dashboard/config` existente (não cria endpoint novo). Frontend monta o array completo com o `manualContext` editado e chama. Guest → 403 (já existe).

3. **AC3 — Helper `extractAutoSummary`:** Função em `packages/web/components/sprint-dashboard/summary-utils.ts`:
   - Input: `ClickUpTaskShape[]`
   - Output: `{ taskId: string; name: string; url: string; dueDate: string | null } | null`
   - Filtra tasks cujo `name` contém `📢` (também aceita `📣` como alias)
   - Ordena por `dueDate` ASC (sem due vai pro fim)
   - Retorna a primeira ou `null`.

4. **AC4 — Resolver de resumo:** Componente computa `resolvedSummary`:
   - Se `block.manualContext` é truthy (após trim) → usa esse texto, marca origem `manual`
   - Senão → chama `extractAutoSummary(tasksDoBloco)`, marca origem `auto`
   - Senão (nada nenhum dos 2) → não renderiza o card resumo

5. **AC5 — Card Resumo na Visão Geral:** Em `SprintBlockCard`, quando `resolvedSummary !== null`:
   - Renderiza acima da lista de tasks
   - Mostra: ícone `📢` (ou `Megaphone` da lucide) + texto
   - Se `origem === auto`: link pra task ClickUp (`ExternalLink`)
   - Se `origem === manual`: badge sutil "manual"
   - Suporta line breaks em `manualContext` (`whitespace-pre-wrap`)

6. **AC6 — Botão "Contexto":** Header do `SprintBlockCard` ganha botão "Contexto" (ícone `MessageSquare` ou `Megaphone`) com tooltip "Editar contexto do bloco". Click abre `BlockContextDialog`.

7. **AC7 — `BlockContextDialog`:**
   - Dialog com Textarea (rows=6, maxLength=2000)
   - Header: "Contexto do bloco" + subtítulo com nome do bloco
   - Help text: "Texto livre que aparece no card resumo. Sobrescreve qualquer task marcada com 📢 no ClickUp. Deixe vazio pra voltar ao automático."
   - Botões: Cancelar / Salvar
   - Ao salvar: chama mutation que monta `blocks` atualizado e faz PUT config; toast sucesso, fecha dialog, invalida cache

8. **AC8 — PendingTasksList no Macro:** `CampaignCard` em `macro-calendar-view.tsx` ganha `<PendingTasksList block={block} />` ao fim, abaixo das phases.

9. **AC9 — Filtro de pendentes:**
   - Pega tasks via `tasksByListId` (mesma fonte do `SprintBlockCard`)
   - Aplica `block.filters` (reusa `applyFilters` de `sprint-block-card.tsx` — vai pra `summary-utils.ts` pra ser compartilhado)
   - Filtra: `!isDoneStatus(task.status) && dueDate parseável && dueDate <= endOfToday()`
   - Ordena por `dueDate` ASC (vencidas há mais tempo primeiro)

10. **AC10 — Visual de pendentes:** Cada item:
    - `border-l-2` red-500 (overdue) ou amber-500 (hoje)
    - Status badge nativo ClickUp (mesma estética do BlockCard)
    - Due relative em pt-BR: "Hoje", "Ontem", "Há N dias"
    - Truncate nome longo
    - Click no nome → abre task no ClickUp (target="_blank")

11. **AC11 — Limit + expand:** Mostra primeiras 5; se `count > 5` renderiza linha "+N pendentes" (clicável apenas pra revelar todas, sem navegação).

12. **AC12 — Empty state:** Se nenhuma task pendente no bloco, mostra texto sutil "Nenhuma task pendente" (não esconde a section pra manter consistência visual).

13. **AC13 — Sem regressão:**
   - Tasks `📢` continuam aparecendo na lista normal do BlockCard (não some)
   - `campaignPhases` continuam funcionando como antes
   - Cache 5min de tasks segue ativo
   - Demais blocos sem `manualContext` nem `📢` renderizam idêntico ao atual

14. **AC14 — Typecheck + lint zero:** `pnpm -r typecheck` + `pnpm -r lint` passam.

## Risks

- **Emoji match falha com variações Unicode:** Já vi `📢` e `📣` em uso. Hardcoded; se user usar outro emoji, não pega. Aceitável pra v1 — adicionamos mais se precisar.
- **Tasks `📢` ainda aparecem na lista geral:** Decisão consciente (não sumir). Se incomodar visual, podemos esconder duplicata no `SprintBlockCard` numa v2.
- **Performance no Macro:** `getPendingTasks` roda por bloco, em cima do `tasksByListId` que já está em memória. O(n) por bloco — irrelevante pra N de tasks da Loyola.
- **manualContext maior que 2000 chars:** Validação Zod max 2000. UI mostra contador.
- **Layout v1 ≠ visual final:** User vai enviar prints depois. Funcional fica estável; visual itera num PR de polish.

## Dev Notes

- `manualContext` persiste no jsonb existente — sem migration nova. Zod `blockSchema` aceita `.max(2000).nullable().optional()`. Backend já passou batch validation no PUT.
- `summary-utils.ts` consolida `isDoneStatus`, `applyFilters`, `extractAutoSummary`, `getPendingTasks`, `collectBlockTasks`, `formatRelativeDue` — usados por `SprintBlockCard` E `PendingTasksList`. `SprintBlockCard` deletou as cópias locais e importa do utils.
- Resumo: `manualContext.trim() truthy` ⇒ origin=`manual`. Senão tenta auto via `extractAutoSummary(tasks)`. Empty string explícita = limpa manual e volta a usar auto.
- `extractAutoSummary` busca `📢` E `📣` (alias). Ordem por `dueDate` ASC com `+Infinity` no fallback pra tasks sem due.
- `getPendingTasks` exclui done e tasks sem due parseável. Marca `isOverdue` (antes do start of today) e `isToday` (entre start e end of today).
- Border-l-2 com hex direto (red-500 #ef4444 / amber-500 #f59e0b) pra evitar dependência de classes dinâmicas do Tailwind.
- `BlockContextDialog` recebe `allBlocks` por prop, monta a versão atualizada e chama `useUpdateSprintDashboardConfig` (já existia) — sem hook novo.
- Layout v1 propositalmente conservador. Quando user enviar prints reais (canal Geral / cards de Campanha), polish visual fica fácil — toda lógica está estável.
