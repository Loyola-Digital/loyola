# Story 29.11 — Funil Perpétuo: mapeamento de "Data da Venda" não persiste (e quebra o filtro de período)

**Epic:** 29
**Status:** InReview
**Points:** 2
**Tipo:** Bug Fix (P0 — crítico, bloqueador)
**Prioridade:** P0

## Contexto (por que existe)

Dois sintomas reportados por Danilo, **mesma causa raiz**:

1. **"Selecionei 'Data de Criação' no campo Data da Venda mas não salva."** Ao mapear a coluna de data no wizard de vinculação da planilha do perpétuo, a seleção não persiste — reabre e volta pra "— Não mapear —".
2. **Dados all-time inflados (#2):** filtrando 01/06–30/06/2026, a planilha tem 146 vendas / R$ 8.614, mas o dash mostrava 1.5K vendas / R$ 83.1K (≈10x = histórico completo, ignorando o período).

**Causa raiz (confirmada no código):** O schema Zod de validação do save no endpoint do perpétuo (`packages/api/src/routes/perpetual-spreadsheets.ts:23-34`) **não incluía `dataVenda` nem `transactionId`** (e tinha um campo `date` fantasma que nenhum leitor usa). Como o Zod **descarta silenciosamente** chaves não declaradas no `.parse()`, o front enviava `columnMapping.dataVenda` corretamente, mas o backend **removia o campo antes de gravar no banco**.

Consequência em cadeia:
- `dataVenda` nunca chegava ao DB → `mapping.dataVenda === undefined`.
- No endpoint de agregação (`perpetual-sales-data.ts:191`), `dataIdx = colIdx(mapping.dataVenda)` retornava `-1`.
- As guardas `&& dataIdx !== -1` (`perpetual-sales-data.ts:199, 202, 225`) **desligavam o filtro de data inteiro** → agregava a planilha toda (all-time).

Ou seja: era **impossível** filtrar por período no perpétuo, porque era impossível salvar a coluna de data.

**Referência do fix correto:** o schema da planilha de **etapa** (`stage-sales-spreadsheets.ts:28-46`) espelha o tipo canônico `SaleColumnMapping` (`packages/shared/src/types/funnel.ts:26`) e inclui `dataVenda`/`transactionId` — funciona. O do perpétuo tinha divergido.

## User Story

Como usuário de funil perpétuo, quero que o mapeamento de "Data da Venda" (e "ID da Transação") seja salvo corretamente, para que o filtro de período funcione e os KPIs mostrem os números do intervalo selecionado — não o histórico inteiro.

## Dev Notes

### Fix mínimo, cirúrgico
Uma única correção de schema resolve os dois sintomas. Não precisou tocar a lógica de agregação (o filtro de data já funciona quando `dataIdx !== -1` — ele só nunca era alcançado porque a coluna nunca era salva).

### Precisa de deploy da API pra valer
O fix é no `packages/api` (backend, roda no Railway). Para testar:
- **Local:** reiniciar o processo da API (`pnpm dev`). ⚠️ Cuidado com o watcher tsx zumbi (a API às vezes não recarrega — matar processos na porta 3001 + watchers órfãos e subir de novo).
- **Deploy:** só terá efeito após @devops mergear + Railway deployar.

### Passo do usuário após o deploy
Re-abrir o wizard e **mapear "Data da Venda"** (ex.: coluna "Data de Criação"). Só então o filtro de período passa a valer. Se depois de mapear os números **ainda** não baterem, há um 2º bug latente (formato de data no `parseDate`) — investigar à parte.

### AC7 adiado
A guarda de integridade (avisar em vez de mostrar all-time) é defesa em profundidade — evita a mentira silenciosa caso alguém não mapeie a data. Não é necessária pro fix do bug (agora dá pra mapear), então deixei como follow-up pra manter este P0 atômico e desbloquear o teste rápido. Conecta com a Story 29.12 (auto-map, que já deixaria a data mapeada de cara).
