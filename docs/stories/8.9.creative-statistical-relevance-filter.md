## Story

**As a** gestor de tráfego da Loyola Digital,
**I want** que a galeria de criativos esconda por padrão criativos sem relevância estatística (gasto baixo demais para tirar conclusões),
**so that** eu enxergue apenas criativos cuja performance é estatisticamente significativa e tome decisões de otimização sobre dados confiáveis — sem precisar configurar filtros manualmente toda vez.

---

## Contexto e Motivação

Hoje a galeria de criativos (`TopCreativesGallery`) mostra todos os criativos com qualquer gasto no período filtrado. Isso polui a análise com:
- Criativos novos com poucos reais gastos e 0 ou 1 venda sortuda (CPA irrealista)
- Criativos pausados/legados com spend residual
- Variações sem volume suficiente pra concluir nada

A regra de mercado para um criativo "falar estatisticamente" é ter gasto suficiente para gerar **≥3-5 conversões** ou **≥50-100 cliques** — o que tipicamente equivale a um múltiplo do CPA da conta.

**Decisão (validada com Lucas em 2026-05-19):** aplicar limiar adaptativo baseado no CPA agregado do período: `spend ≥ 2 × CPA_médio`. Fallback se 0 vendas: `spend ≥ 2 × gasto_médio`.

---

## Acceptance Criteria

### Cálculo do Limiar (Threshold)

1. [ ] Calcular **CPA agregado do período** como `sum(spend) / sum(vendas)` sobre todos os criativos visíveis (antes do filtro), onde `vendas` = `salesLegacy` do `AggregatedCreative`.
2. [ ] Calcular **threshold de relevância** = `2 × CPA_agregado`.
3. [ ] **Fallback (sum(vendas) === 0):** se não há vendas no período, calcular `gasto_médio` = `sum(spend) / count(criativos com spend > 0)` e usar `threshold = 2 × gasto_médio`.
4. [ ] **Fallback duplo (zero criativos ou zero spend total):** se não dá pra calcular nem CPA nem gasto médio (lista vazia ou spend total = 0), filtro fica desabilitado automaticamente (mostra tudo).
5. [ ] Cálculo é feito **client-side**, em util pura testável, sobre a lista de `AggregatedCreative` já agregada (não no backend).

### Aplicação do Filtro

6. [ ] Criativos com `spend < threshold` são ocultados por padrão da galeria.
7. [ ] A regra se aplica **após a agregação por nome** (`aggregateCreativesByName`) — ou seja, o spend considerado é o spend agregado do criativo (soma de todas as variações com mesmo nome).
8. [ ] A regra **não filtra ordenação** — apenas esconde linhas. A ordenação atual (por ROAS/spend/leads/etc.) continua intacta sobre o conjunto filtrado.

### UI — Toggle e Transparência

9. [ ] Adicionar um toggle **"Mostrar todos os criativos"** no header da galeria (próximo aos controles de ordenação/filtros existentes).
10. [ ] Toggle **default = OFF** (filtro de relevância ATIVO por padrão).
11. [ ] Quando o filtro está ativo e há criativos ocultos, mostrar um indicador discreto: **"X criativos ocultos por baixo gasto (< R$ Y)"** — onde Y é o threshold calculado, formatado em BRL com 0 casas decimais (ex: "R$ 287").
12. [ ] Ao clicar no indicador ou no toggle, o filtro desliga e todos os criativos aparecem.
13. [ ] Quando o filtro está desligado, esconder o indicador de "X criativos ocultos" e mostrar visualmente que o toggle está ativo.
14. [ ] Em caso de fallback duplo (AC 4), o toggle fica desabilitado/oculto com tooltip: "Sem dados suficientes para calcular relevância estatística".

### Acessibilidade e Persistência

15. [ ] Toggle tem `aria-label` claro: "Mostrar criativos com baixo gasto" / "Esconder criativos com baixo gasto".
16. [ ] O estado do toggle **não persiste** entre sessões (volta pro default OFF a cada page load). Justificativa: o critério é dinâmico (depende do período), persistir confundiria.

### Testes

17. [ ] Unit tests para a util de cálculo cobrindo: caso normal (com vendas), fallback 1 (sem vendas mas com spend), fallback 2 (lista vazia ou spend total = 0), criativo no threshold exato (>= passa, < não passa).
18. [ ] Unit test para a util de filtragem: lista de N criativos → retorna subset correto + threshold + count de ocultos.

### Fórmula resumida

```typescript
function computeRelevanceThreshold(creatives: AggregatedCreative[]):
  { threshold: number; mode: 'cpa' | 'spend' | 'disabled'; cpaMedio: number | null } {
  const totalSpend = creatives.reduce((s, c) => s + c.spend, 0);
  const totalSales = creatives.reduce((s, c) => s + c.salesLegacy, 0);
  const withSpend = creatives.filter(c => c.spend > 0).length;

  if (totalSpend === 0 || creatives.length === 0) {
    return { threshold: 0, mode: 'disabled', cpaMedio: null };
  }
  if (totalSales > 0) {
    const cpaMedio = totalSpend / totalSales;
    return { threshold: 2 * cpaMedio, mode: 'cpa', cpaMedio };
  }
  // Fallback: sem vendas no período
  const gastoMedio = totalSpend / withSpend;
  return { threshold: 2 * gastoMedio, mode: 'spend', cpaMedio: null };
}
```

### Considerações

- **Onde inserir o filtro no fluxo da galeria:** depois do `enrichWithPaidLeads` (preserva todas as métricas calculadas) e **antes** da ordenação/paginação. Isso garante que ordenação opera sobre o conjunto visível.
- **Não tocar no backend.** O cálculo é leve e roda no client sobre dados que já chegaram.
- **Não criar nova rota.** Não há mudança de API.
- **Edge case importante:** criativos com `spend === 0` ainda entram no cálculo de threshold? Não — são ignorados no cálculo do `gastoMedio` (denominador `withSpend`), mas continuam aparecendo na galeria se o toggle estiver ligado.
- **Compat:** comportamento atual (mostrar tudo) está disponível via toggle "Mostrar todos". Não é breaking change funcional.

### Por que não filtrar "Top N por vendas" (opção descartada)

Análise feita com Lucas: filtrar por top vendas confunde *ordenação* com *filtragem de ruído*. Um criativo com R$30 gastos e 1 venda sortuda passaria, enquanto um criativo "queimando dinheiro" (R$500 e 0 vendas — justamente o que ele quer detectar pra pausar) sumiria. O filtro por gasto adaptativo resolve isso: mantém visíveis tanto os winners quanto os perdedores estatisticamente relevantes.

## Risks & Open Questions

- **R1:** Se o usuário filtrar período muito curto (ex: 1 dia) com poucos criativos, o CPA agregado fica volátil. Mitigação: o toggle resolve — basta desligar.
- **R2:** A escolha de `N=2` (multiplicador) é heurística. Caso o usuário ache muito agressivo ou muito permissivo no uso real, podemos: (a) tornar configurável em uma próxima story; (b) ajustar para 1.5 ou 3. Lucas validou N=2 como ponto de partida.
- **Q1:** Faz sentido aplicar a mesma regra na drill-down (lista de ads dentro de uma campanha)? **Decisão:** fora de escopo desta story — focar só na galeria principal. Se houver demanda, abrir 8.10.
