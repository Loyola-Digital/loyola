**Epic:** 19 — Funnel Stages

## User Story

**Como** gestor do Loyola X que fecha vendas num **evento presencial**,
**quero** que, ao **lançar a venda** na etapa de Evento Presencial, o comprador seja **matriculado automaticamente no MemberKit** (na turma/curso configurado), por meio de uma **integração nativa** (eu cadastro a API key do MemberKit uma vez no projeto e escolho a turma),
**para que** o aluno ganhe acesso à área de membros na hora, sem ninguém precisar matricular manualmente no painel do MemberKit.

---

## Contexto Técnico

### Esclarecimento sobre o "webhook para o MemberKit"

O pedido fala em "disparar um webhook para o MemberKit". Tecnicamente, **o MemberKit não recebe webhooks de matrícula** — quem quer matricular **chama a API REST dele** (`POST /api/v1/users`, que faz upsert do membro por email e libera o acesso). Então a integração nativa é **outbound (nós → API do MemberKit)**, exatamente o padrão que Hotmart/Kiwify/Digital Manager Guru usam. Mantemos o termo "disparo" no sentido de "ao lançar a venda, disparamos a chamada de matrícula". (O MemberKit **também** emite webhooks de eventos via `POST /api/v1/hooks` — útil no futuro para reagir a `enrollment_created` etc., mas **fora de escopo** aqui.)

### Contrato da API MemberKit (pesquisado — fonte: `ajuda.memberkit.com.br`, confiança ALTA)

- **Base URL:** `https://memberkit.com.br/api/v1/`
- **Autenticação:** API key na **query string** `?api_key=...` em **toda** chamada (NÃO é header `Authorization`/`X-Api-Key`). ⇒ **sempre server-side**; cuidado com a key vazar em logs de URL.
- **Matrícula / upsert de membro:** `POST /api/v1/users?api_key=...`
  - Body: `full_name` (**obrigatório**), `email` (**obrigatório** — chave de upsert), `status` (**obrigatório**: `active|inactive|pending|expired`), e **um** dos modos de acesso: `classroom_ids: number[]` (turmas) **ou** `membership_level_id: number` **ou** `unlimited: boolean`.
  - Opcionais cadastrais: `cpf_cnpj`, `phone_local_code`, `phone_number`, `blocked`, `expires_at`.
  - Retorno: `201` com `UserDetails` (`enrollments[]`, `memberships[]`).
- **Listar turmas (para o picker):** `GET /api/v1/classrooms?api_key=...` → `id`, `name`, `master`, `course_name`, `users_count`, ...
- **Listar cursos:** `GET /api/v1/courses?api_key=...` → `id`, `name`, `category{...}`, ...
- **Níveis de assinatura:** `GET /api/v1/membership_levels?api_key=...`
- **Magic link (acesso direto pós-matrícula):** `POST /api/v1/tokens?api_key=...` body `{ email }` → `{ token, authenticated_url }`.
- **Rate limit:** **120 req/min** (global). Paginação: 50/página, metadados em headers (`Total-Pages`, etc).
- **Spec oficial:** `https://ajuda.memberkit.com.br/referencia-api/openapi.yml` (baixar na implementação para catálogo de erros).

> 🔴 **BLOQUEADOR de produto — email é obrigatório para matrícula.** Não existe matrícula por telefone/CPF. A planilha do evento (Story 19.10) **não tem email** — só `Nome` + `Telefone`. Por isso a matrícula automática depende de **capturar o email no lançamento da venda manual** (ver Decisão arquitetural). Vendas vindas só da planilha (sem email) **não** podem ser matriculadas automaticamente.

### Mudança proposta

1. **Tabela `memberkit_connections`** (1 por projeto, API key cifrada AES).
2. **Tabela `stage_memberkit_enrollment`** (config por etapa: quais `classroom_ids`/`status` matricular, e se o disparo é automático).
3. **Service `memberkit.ts`**: `encryptMemberkitKey`/`decryptMemberkitKey`, `testMemberkitConnection`, `listClassrooms`, `listCourses`, `enrollMember`.
4. **Rota `memberkit.ts`**: CRUD da connection (`{connected}` no GET), picker de turmas/cursos, e CRUD da config de enrollment por etapa.
5. **UI `memberkit-stage-tab.tsx`** na `EventStageView`: conectar API key + escolher turma(s) + status.
6. **Gancho de matrícula** no `POST .../manual-sales` (e edição): se etapa `event` + connection ativa + config de enrollment + venda com **email** → chamar `enrollMember`. Registrar status no log; **nunca** falhar a venda por erro do MemberKit.

### Decisão arquitetural — o bloqueador do email

- **Default escolhido (recomendado):** na etapa de evento, o `ManualSaleDialog` torna **email obrigatório** (o time coleta o email do comprador no momento da venda). Assim a matrícula automática sempre tem chave válida e o aluno recebe o magic link real.
- Vendas **só da planilha** (sem email) **não** disparam matrícula — ficam marcadas como **"pendente de email"** na UI, com ação manual "Matricular" disponível depois que o email for preenchido.
- **Rejeitado:** gerar email-placeholder a partir do telefone (`tel@placeholder`) — tecnicamente matricula, mas quebra o login/magic-link real do aluno.

## Acceptance Criteria

1. **AC1 — Conectar o MemberKit ao projeto.**
   - Existe `memberkit_connections` (1 por projeto, API key cifrada via `encryption.ts`).
   - `PUT /api/projects/:projectId/memberkit/connection` valida a key **antes de salvar** (chama `testMemberkitConnection` → `GET /classrooms?api_key=`), cifra e faz upsert por `projectId`.
   - `GET .../connection` retorna **apenas `{ connected: boolean }`** (nunca a key). `DELETE` remove.
   - Guest sem vínculo: 403.

2. **AC2 — Escolher a turma/curso de matrícula por etapa.**
   - `GET .../memberkit/classrooms` e `.../memberkit/courses` listam turmas/cursos do MemberKit (para o picker), usando a connection do projeto.
   - Existe config por etapa (`stage_memberkit_enrollment`): `classroomIds: number[]`, `status` (default `active`), `autoEnroll: boolean` (default `true`).
   - A UI (`memberkit-stage-tab` dentro da `EventStageView`) permite: conectar/desconectar a key, escolher uma ou mais turmas, definir status e ligar/desligar o auto-enroll.

3. **AC3 — Matrícula automática ao lançar venda manual (com email).**
   - Ao criar uma venda manual numa etapa `event`, **se** a connection está ativa **e** há config de enrollment **e** `autoEnroll` ligado **e** a venda tem `customerEmail` → o backend chama `POST /api/v1/users` com `full_name`, `email`, `status` e `classroom_ids` configurados.
   - O resultado é registrado (sucesso/erro) e exibível na UI (badge "Matriculado ✓" / "Falha" / "Pendente de email").
   - **A venda é persistida e retornada com sucesso mesmo se a matrícula falhar** (erro do MemberKit nunca derruba o lançamento) — falha é logada e marcada como retryável.

4. **AC4 — Venda sem email não bloqueia.**
   - Venda manual em etapa `event` **sem** email → é criada normalmente, **sem** chamar o MemberKit, marcada "pendente de email".
   - Existe ação manual **"Matricular no MemberKit"** numa venda já existente (`POST .../manual-sales/:saleId/memberkit-enroll`) que reaproveita o mesmo service.

5. **AC5 — Robustez (rate limit + erros).**
   - As chamadas ao MemberKit são server-side, com timeout e tratamento de erro legível; respeitam o rate limit de 120/min (throttle/serialização ao matricular em lote).
   - Nenhuma API key aparece em logs (a key vai na query string — mascarar nos logs).

6. **AC6 — Idempotência.**
   - Reenviar a mesma venda (mesmo email/turma) não cria acesso duplicado (o `POST /users` do MemberKit já é upsert por email — confirmar e não tratar 200/201 repetido como erro).

## Risks

| Risco | Prob. | Impacto | Mitigação |
|-------|-------|---------|-----------|
| **Email obrigatório no MemberKit × planilha de evento sem email.** | Alta | Alto | Email obrigatório (recomendado) no lançamento manual; vendas sem email = "pendente" com re-matrícula manual. **Confirmar a regra com o Lucas na validação.** |
