# Story 29.13 — Funil Perpétuo: Receita por Público/Criativo mostrar nome (não ID)

**Epic:** 29
**Status:** InReview
**Points:** 3
**Tipo:** Feature / Data Clarity
**Prioridade:** P1

## Contexto

Nos gráficos "Receita por Público" (utm_medium) e "Receita por Criativo" (utm_content) do Funil Perpétuo, os rótulos mostravam **IDs crus** (adset id / ad id) em vez dos nomes — no padrão Loyola, `utm_medium` carrega o adset_id e `utm_content` carrega o ad_id. O backend do perpétuo (`perpetual-sales-data.ts:307-312`) entregava a chave crua sem resolver nome, diferente da seção por-etapa (`stage-sales-data`) que já resolve.

## User Story

Como usuário de funil perpétuo, quero ver o **nome do público (adset)** e do **criativo (anúncio)** nos gráficos de receita — não o ID — para entender de imediato qual público/criativo está performando.

## Acceptance Criteria

- [x] **AC1:** "Receita por Público" exibe o **nome do adset** resolvido de `utm_medium` (adset id). Fallback pro ID quando não resolver.
- [x] **AC2:** "Receita por Criativo" exibe o **nome do anúncio** resolvido de `utm_content` (ad id). Fallback pro ID quando não resolver.
- [x] **AC3:** Adsets/ads com o mesmo nome têm a receita **somada** (re-agrupamento por nome resolvido), ordenado por receita desc, top 8.
- [x] **AC4:** Resolução via cache de nomes Meta (`POST /meta-names/resolve`, DB 24h) — resolve **qualquer** id, não só os com insights na janela selecionada.
- [x] **AC5:** Sem regressão no bloco legacy (sem planilha → gráficos Meta por campanha continuam iguais).
- [x] **AC6:** Typecheck OK.

## Dev Notes

### Por que frontend (e não backend)
Escolhi resolver no **frontend via endpoint existente** (`/meta-names/resolve`) em vez de espelhar o backend do stage-sales-data. Motivos: (1) reusa infra pronta (o hook `useResolveAdsetNames` já batia nesse endpoint pro adset), (2) testável no Local sem reiniciar API, (3) resolve qualquer id via cache 24h — não limitado à janela. Se um dia quiser consolidar perpétuo == etapa no backend, a Abordagem C (adicionar `name` no `PerpetualSalesData` e resolver server-side) fica como refactor futuro.

### Fallback
IDs que não resolvem (ex: utm com valor não-Meta, conta diferente) caem no ID cru — não quebram o gráfico.
