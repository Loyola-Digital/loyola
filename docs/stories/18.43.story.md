## 📖 User Story

Como gestor de tráfego, quero que a tabela **"Desempenho de Criativos"** na **Captação Gratuita** mostre o número correto de **leads gerados por criativo**, obtido pelo cruzamento de:
1. **Conversões (linhas) da planilha de captação gratuita conectada**
2. **IDs de anúncios (Meta Ads API)**

em vez de exibir zeros, para que eu possa avaliar a performance real de cada criativo em tráfego orgânico.

---

**T-Shirt Size:** M-L (8-13 Story Points)
- **Drivers:** Hook novo + integração em tabela existente, lógica de cruzamento, validação de agregação
- **Estimated Timeline:** 3-5 dias (@dev)
- **Primary Complexity:** Cruzamento de 2 fontes de dados (planilha + Meta Ads), agregação correta, performance com 1000+ registros

## 📐 Scope

### IN (Inclusos)
- Leitura de todas as linhas da planilha de Captação Gratuita vinculada
- Cruzamento com Meta Ads API usando `ad_id` como chave
- Agregação/soma de leads por `ad_id` (mesmo criativo = múltiplos registros → somar)
- Cálculo de CPL recalculado (`Spend ÷ Leads`)
- Integração com filtro de período (DayRangePicker) — cruzamento recalcula ao alterar dias
- Normalização de IDs (trim, lowercase) se houver inconsistências

### OUT (Exclusos)
- Criação de nova tabela (reutilizar componente existente `StageCreativePerformanceTable`)
- Alteração de estrutura de dados na tabela (colunas, ordem)
- Mudança de Captação Paga (manter comportamento antigo)
- Backend novo — usar dados já disponíveis ou hook frontend-only
- UI de configuração de mapeamento de campos (MVP hardcoded)

## ⚠️ Risks & Mitigations

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|-----------|
| Coluna `ad_id` na planilha tem nome diferente | **High** | **Medium** | Danilo confirma nome exato em kick-off; suportar aliases no código |
| Cruzamento lento com 1000+ linhas + 100 criativos | **Low** | **Medium** | Usar hash/indexação em vez de .filter().map(); testar performance |
| Planilha não tem `ad_id` para alguns leads (NULL ou vazio) | **Medium** | **Medium** | AC5: empty state claro "Sem dados de ad_id", sem crash; logar warning |
| Drift entre ID na planilha e ID Meta Ads | **Medium** | **High** | Normalizar IDs: `.toLowerCase().trim()`; validar matches no kick-off |
| CPL cálculo com Leads=0 causa divisão por zero | **Low** | **Low** | Já coberto em AC4: exibir "—" quando Leads=0 |
| DayRangePicker muda → leads não recalcula (cache stale) | **Low** | **High** | Hook deve ter deps array correto: `[projectId, funnelId, stageId, days]` |

### AC2: Agregação Correta
- [x] Se um mesmo `ad_id` aparecer múltiplas vezes na planilha → **SOMAR** todos os leads
- [x] Se um mesmo `ad_id` tiver múltiplos registros na API do Meta Ads → usar **um único registro agregado** (não duplicar)
- [x] Exemplo:
  ```
  Planilha:
    ad_id=123: 5 linhas = 5 leads
    ad_id=123: aparece de novo 3 linhas = +3 leads
    Total para ad_id=123: 8 leads
  ```

### AC3: Referência Implementação (Story 18.6 — Top Criativos)

- [x] Hook ou função reutilizável: `crossReference(planilhaRows, metaAdsData, keyField)` 
- [x] Normalizar IDs (trim, lowercase) se houver inconsistências
- [x] Filtro por data deve ser aplicado ANTES do cruzamento (se DayRangePicker ativo)

### AC4: Coluna "Leads" Atualizada
- [x] Coluna "Leads" na tabela exibe **valor calculado do cruzamento** (não mais zero)
- [x] CPL (Cost-Per-Lead) recalculado: `CPL = Spend / Leads` (ou "—" se Leads = 0)
- [x] Validação: CPL de um criativo = seu Spend ÷ seus Leads
- [x] Ordem de cálculo: buscar leads via cruzamento → aplicar CPL → exibir

### AC5: Empty State Handling
- [x] Se planilha não tiver campo `ad_id` → log warning em dev, mostrar "Sem dados de ad_id" na coluna Leads
- [x] Se cruzamento retorna 0 matches → exibir "0" (não "—"), pois significa nenhuma conversão daquele criativo
- [x] Se planilha está vazia → tabela exibe "Nenhuma conversão no período" ou similar

### AC6: Performance
- [x] Cruzamento executa em < 500ms para 1000+ linhas de planilha + 100 criativos
- [x] Usar indexação/hash se necessário para busca rápida de `ad_id`
- [x] Sem recompilação desnecessária quando DayRangePicker muda

### AC7: Filtro de Período Efetivo

- [x] Exemplo: alterar de "últimos 7 dias" → "últimos 30 dias" → Leads na tabela aumentam
- [x] Sem regressão em outros filtros (hot/cold público)

### AC8: Sem Regressão

- [x] Colunas suppressed (Faturamento, ROAS) continuam suprimidas

## 📝 Technical Context

### Data Source
**Planilha de Captação Gratuita:**
- Cada linha = 1 conversão
- Campo chave: `ad_id` ou coluna equivalente
- Filtro de data disponível via `DayRangePicker`

**Meta Ads API:**
- Dados já integrados (endpoint `/ads` ou similar)
- Fields: `meta_adid`, `meta_adname`, `spend`, `impressions`, `clicks`, etc.

### Arquitetura Sugerida

**Frontend Hook:**
```typescript
// Hook que cruza dados
useCrossReferenceLeads(projectId, funnelId, stageId, days) → {
  leads: Record<string, number>,  // { ad_id: 8, ... }
  isLoading: boolean,
  error?: string,
}

// Uso:
const { leads } = useCrossReferenceLeads(...)
const metaAdsData = useStageCreativePerformance(...)

// Combinar leads no CPL:
data.map(ad => ({
  ...ad,
  leads: leads[ad.adId] || 0,
  cpl: ad.spend / (leads[ad.adId] || 1),
}))
```

**Backend (se necessário):**
- Endpoint: `GET /api/funnels/:funnelId/stages/:stageId/creative-leads-crossref?days=30`
- Response:
  ```json
  {
    "leads": {
      "123": 8,
      "124": 12,
      "125": 0
    }
  }
  ```

### Referência: Story 18.6 Pattern

De Story 18.6 (Surveys), usar padrão:
```typescript
// normalizar se necessário:
function normalizeAdId(raw: string): string {
  return raw.toLowerCase().trim();
}

// cruzamento:
const byAdId = {};
planilhaRows.forEach(row => {
  const adId = normalizeAdId(row.ad_id);
  byAdId[adId] = (byAdId[adId] || 0) + 1;  // SOMAR
});

// resultado: { "123": 8, "124": 5 }
```
