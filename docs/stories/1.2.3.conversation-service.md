## Story

**As a** authenticated Loyola Digital user,
**I want** to list my conversations, view message history, and delete conversations I no longer need,
**so that** I can manage my interactions with minds and easily return to previous conversations.

---

## Acceptance Criteria

1. [ ] `ConversationService` como Fastify plugin (`packages/api/src/services/conversation.ts`)
2. [ ] `GET /api/conversations` — lista conversas do user autenticado
   - Query params: `?limit=20&offset=0&mindId=optional`
   - Ordenado por `updated_at DESC`
   - Exclui soft-deleted (`deleted_at IS NULL`)
   - Retorna `{ conversations: [...], total: number }`
3. [ ] `GET /api/conversations/:id/messages` — mensagens da conversa

6. [ ] Resolução de user: `request.userId` (Clerk ID) deve ser resolvido para `users.id` (UUID) via lookup na tabela `users`

## Dev Notes

### Arquitetura do Backend Existente

O backend Fastify está configurado (Story 1.1.1) com:
- Plugin architecture (`fastify-plugin`)
- Drizzle ORM com schema em `packages/api/src/db/schema.ts`
- Auth middleware em `packages/api/src/middleware/auth.ts` — usa `getAuth(request)` e seta `request.userId` como **Clerk ID** (string como `user_2abc123`)
- DB connection via `fastify.db` (Drizzle instance)
- Env config validada com Zod em `packages/api/src/config/env.ts`

[Source: packages/api/src/middleware/auth.ts, packages/api/src/db/client.ts]

### IMPORTANTE: Resolução de User ID

`request.userId` contém o **Clerk ID** (ex: `user_2abc123`), NÃO o UUID do banco. A tabela `conversations` referencia `users.id` (UUID) via FK. Portanto, TODAS as queries de conversations devem:

1. Primeiro resolver `clerkId → users.id` via SELECT na tabela `users`
2. Usar o UUID resultante para filtrar conversations

```typescript
// Pattern para resolver user:
const [user] = await fastify.db
  .select({ id: users.id })
  .from(users)
  .where(eq(users.clerkId, request.userId))
  .limit(1);

if (!user) {
  reply.code(404).send({ error: "User not found" });
  return;
}
// Usar user.id (UUID) para queries de conversations
```

[Source: packages/api/src/middleware/auth.ts, packages/api/src/db/schema.ts]

### Schema da Tabela `conversations`

```typescript
// packages/api/src/db/schema.ts (existente)
export const conversations = pgTable("conversations", {
  id: uuid("id").defaultRandom().primaryKey(),
  userId: uuid("user_id").notNull().references(() => users.id, { onDelete: "cascade" }),
  mindId: text("mind_id").notNull(),
  mindName: text("mind_name").notNull(),
  squadId: text("squad_id").notNull(),
  title: text("title"),
  messageCount: integer("message_count").notNull().default(0),
  totalTokens: integer("total_tokens").notNull().default(0),
  deletedAt: timestamp("deleted_at", { withTimezone: true }),
  createdAt: timestamp("created_at", { withTimezone: true }).defaultNow().notNull(),
  updatedAt: timestamp("updated_at", { withTimezone: true }).defaultNow().notNull(),
}, (table) => [
  index("idx_conversations_user_updated").on(table.userId, table.updatedAt).where(sql`deleted_at IS NULL`),
  index("idx_conversations_user_mind").on(table.userId, table.mindId).where(sql`deleted_at IS NULL`),
]);
```

**Partial indexes:** `idx_conversations_user_updated` e `idx_conversations_user_mind` já filtram `deleted_at IS NULL`, alinhado com as queries desta story.

[Source: packages/api/src/db/schema.ts]

### Schema da Tabela `messages`

```typescript
export const messages = pgTable("messages", {
  id: uuid("id").defaultRandom().primaryKey(),
  conversationId: uuid("conversation_id").notNull().references(() => conversations.id, { onDelete: "cascade" }),
  role: messageRoleEnum("role").notNull(),
  content: text("content").notNull(),
  tokensUsed: integer("tokens_used"),
  metadata: jsonb("metadata"),
  createdAt: timestamp("created_at", { withTimezone: true }).defaultNow().notNull(),
```

### Plugin Registration Pattern

Todas as rotas usam `fp` (fastify-plugin) exceto `webhooks.ts` (encapsulado para isolar content parser). As rotas de conversations devem usar `fp` para ter acesso a `fastify.db` e `fastify.config`.

[Source: packages/api/src/routes/chat.ts, packages/api/src/routes/webhooks.ts]

### Cursor Pagination para Messages

Para o cursor `before` no endpoint de messages:
- Client envia `?before=<messageId>` do último message visível
- Server resolve o `createdAt` desse message e filtra `createdAt < resolved_timestamp`
- Isso funciona com o index `idx_messages_conversation_created`

### Performance

- Queries usam partial indexes existentes (`deleted_at IS NULL`)
- Count query separada é aceitável para o volume esperado (< 1000 conversas/user)
- LIMIT/OFFSET para conversations, cursor para messages (melhor performance em datasets grandes)
