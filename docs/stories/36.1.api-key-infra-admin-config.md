## Valor de Negócio

A ideia maior do épico é expor a performance de criativos do Meta Ads como uma **API consumível por um MCP** (a IA do Lucas consulta "qual criativo da campanha SCA tem melhor CTR/CPA?"). Hoje **toda** autenticação do LoyolaX é via Clerk (JWT de humano logado) — não existe um caminho máquina-a-máquina. Esta story entrega a **fundação**: um sistema de API Keys que um admin gera/revoga numa tela de config, sem o qual nenhum endpoint público (Story 36.3) nem o MCP (Story 36.6) podem existir com segurança. Sem chave revogável e escopada, a única alternativa seria expor token fixo no `.env` — sem rastreabilidade, sem revogação granular, sem controle de quem criou.

## Dependências

- **Sem dependências bloqueantes** — é a primeira story do épico e usa apenas infra já existente (Drizzle, Fastify, gate de admin, padrão de rotas).
- **Habilita:** Story 36.2 (middleware de auth por `X-API-Key`), que consome o hash gerado aqui.
- ⚠️ **Banco = produção** — o `DATABASE_URL` local aponta para o banco de PRODUÇÃO e o Coolify **não** roda migrations. A migration desta story precisa ser aplicada com cuidado (ver Riscos + Definição de Pronto).

## Contexto

Mapeamento do código atual (validado via @sm):

- **Auth hoje:** pipeline Clerk → DB em `packages/api/src/middleware/auth.ts` popula `request.userId` e `request.userRole`. Não há nenhuma forma de autenticação não-Clerk.
- **Gate de admin existente:** `packages/api/src/routes/admin.ts:27` usa `if (request.userRole !== "admin" && request.userRole !== "manager")`. Roles em `schema.ts:27-33` (`copywriter`, `strategist`, `manager`, `admin`, `guest`).
- **Padrão de rotas:** Fastify + Zod `safeParse` + `reply.code(...).send(...)`, ex. `packages/api/src/routes/meta-ads.ts:59-105`.
- **Storage de credenciais de integrações:** tabelas dedicadas com colunas criptografadas AES-256-GCM (`meta_ads_accounts`, etc.) via `packages/api/src/services/encryption.ts`.
- **Tela de config admin-only no front:** `packages/web/app/(app)/settings/users/page.tsx:28` faz `const isAdmin = role === "admin" || role === "manager"` e renderiza card "Acesso restrito" se não for.

**Decisão de design importante (diferença vs. integrações existentes):** as integrações guardam tokens **criptografados** (AES, reversível) porque precisam *recuperar* o token para chamar a API externa. Uma API Key é o oposto: o servidor **valida** a chave que chega, nunca precisa recuperá-la. Portanto guardamos **só o hash SHA-256** (irreversível) + um prefixo legível para exibição. A chave em texto puro é mostrada **uma única vez** na criação.

> **Escopo desta story = fundação apenas.** O *consumo* da chave (middleware que valida `X-API-Key` numa request) é a Story 36.2. Aqui criamos a tabela, a geração/revogação e a tela. O esquema de hash precisa ficar pronto para a 36.2 plugar.

## Aceitação

### AC1: Tabela `api_keys` + migration Drizzle
- [ ] Adicionar tabela `api_keys` em `packages/api/src/db/schema.ts`, seguindo as convenções das tabelas existentes (mesmo estilo de `meta_ads_accounts`):
  - `id` — PK (mesmo tipo das outras tabelas: uuid/serial conforme padrão do schema)
  - `name` — `text`, not null (label legível, ex.: "MCP IA Lucas")
  - `keyPrefix` — `text`, not null (primeiros caracteres legíveis p/ exibição, ex.: `lyx_live_a1b2c3d4`)
  - `keyHash` — `text`, not null, **unique** (SHA-256 hex da chave completa)
  - `scopes` — `text[]` (ou `jsonb`), default `['meta:read']`
  - `createdBy` — FK → `users.id` (mesmo padrão de `meta_ads_accounts.createdBy`)
  - `createdAt` — timestamp default now
  - `lastUsedAt` — timestamp, nullable (atualizado pela 36.2; aqui só existe a coluna)
  - `revokedAt` — timestamp, nullable
- [ ] Gerar a migration via `pnpm drizzle-kit generate` (não escrever SQL à mão).
- [ ] Migration aplicada ao banco e validada (ver Riscos — banco é produção).
- [ ] Índice em `keyHash` (lookup por hash na validação da 36.2).

### AC2: Endpoints de gestão de chaves (admin-only) — `routes/api-keys.ts`
- [ ] Novo arquivo `packages/api/src/routes/api-keys.ts` registrado no app (mesmo padrão de registro das outras rotas).
- [ ] **POST `/api/api-keys`** — gera chave nova. Body Zod: `{ name: string (min 1), scopes?: string[] }`.
  - Gate: `request.userRole !== "admin"` → `403` (ver Decisão de escopo nas Notas — **admin only**, não manager).
  - Gera chave segura, calcula prefixo + hash, persiste com `createdBy: request.userId`.
  - **Resposta `201` retorna a chave em texto puro UMA vez** + metadata (id, name, keyPrefix, scopes, createdAt).
- [ ] **GET `/api/api-keys`** — lista chaves. Gate admin. Retorna **mascarado**: `{ id, name, keyPrefix, scopes, createdAt, lastUsedAt, revoked: boolean }`. **Nunca** retorna `keyHash` nem a chave completa.
- [ ] **DELETE `/api/api-keys/:id`** (ou `PATCH /api/api-keys/:id/revoke`) — seta `revokedAt = now`. Gate admin. Não deleta a linha (mantém histórico/rastreabilidade — método Pedro Valério: log completo).
- [ ] Todos os handlers seguem o padrão `meta-ads.ts` (Zod `safeParse`, erros `400`/`403`/`404` com `{ error, details }`).

### AC3: Geração segura da chave
- [ ] Chave gerada com `crypto.randomBytes` (≥ 32 bytes), formato `lyx_live_<base64url ou hex>`.
- [ ] Persistir **apenas** `keyHash` = `sha256(chaveCompleta)` em hex. Texto puro **nunca** vai para o banco nem para logs.
- [ ] `keyPrefix` = `lyx_live_` + primeiros ~8 chars (suficiente para o admin reconhecer a chave na lista, insuficiente para uso).
- [ ] Helper de hash isolado (ex.: `packages/api/src/services/api-key.ts`) com `hashApiKey(raw)` — reutilizável pela 36.2, que vai comparar com `crypto.timingSafeEqual` (constant-time).

### AC4: Tela de config "API Keys" (admin-only) no front
- [ ] Nova tab `packages/web/app/(app)/settings/api-keys/page.tsx`, seguindo o padrão de `settings/users/page.tsx`:
  - Gate `isAdmin` (role === "admin") → card "Acesso restrito" se não for.
  - Adicionar entrada na navegação de settings (mesmo lugar onde estão users/audit/instagram/etc.).
- [ ] **Listagem:** tabela com `name`, `keyPrefix` mascarado (ex.: `lyx_live_a1b2c3d4…`), `createdAt`, `lastUsedAt`, status (Ativa/Revogada), e os `scopes`.
- [ ] **Gerar nova chave:** botão → form (nome + scopes) → ao criar, exibe a chave completa **uma vez** num bloco com botão "Copiar" e aviso claro: *"Guarde agora — você não verá esta chave novamente."*
- [ ] **Revogar:** ação por linha com confirmação ("Tem certeza? Qualquer integração usando esta chave para de funcionar.").
- [ ] Estados de loading/erro consistentes com as outras telas de settings.

## Notas Técnicas

- **Hash vs. criptografia:** ao contrário de `meta_ads_accounts` (AES reversível via `encryption.ts`), API key guarda **só hash SHA-256**. Não reutilizar `encrypt()`/`decrypt()` aqui — o helper novo (`services/api-key.ts`) faz `crypto.createHash('sha256')`.
- **Decisão de escopo do gate:** a gestão de chaves é **admin only** (`role === "admin"`), mais restrita que o gate de `admin.ts` (que aceita admin **ou** manager). Justificativa: uma API key é uma credencial de máquina de longa duração — só o admin deve emitir/revogar. Confirmar com Lucas se manager também deve poder.
- **Prefixo legível:** `lyx_live_` deixa espaço para `lyx_test_` no futuro, sem comprometer este MVP.
- **Comparação constant-time:** o helper de hash já deve expor o que a 36.2 precisa para usar `crypto.timingSafeEqual` e evitar timing attack na validação.
- **Padrão de rota:** copiar a estrutura de `meta-ads.ts:59-105` (guard → Zod `safeParse` → operação → `reply.code().send()`).
