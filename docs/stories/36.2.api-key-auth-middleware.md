# Story 36.2: Middleware de autenticação por API Key (`X-API-Key`) para rotas públicas read-only

**Epic:** 36 - Meta Ads Creative Intelligence API + MCP
**Story:** 36.2
**Status:** Ready for Review
**Priority:** High
**Complexity:** Medium (~1 dia)
**Points:** 5
**Assignee:** @dev

---

## Valor de Negócio

A Story 36.1 cria e guarda as API Keys (hash). Esta story faz elas **funcionarem**: um middleware que valida o header `X-API-Key` numa request real, **sem** passar pelo Clerk, autorizando o MCP a ler dados. Sem ela, a chave é só um registro no banco. É o "portão" pelo qual a IA do Lucas entra — e por isso precisa ser à prova de timing attack, read-only e com rate limit para não virar vetor de abuso.

## Dependências

- **Bloqueante:** Story 36.1 (tabela `api_keys` + `hashApiKey()` em `services/api-key.ts`).
- **Habilita:** Story 36.3 (endpoints públicos protegidos por este middleware).
- **Design:** @architect deve validar o vocabulário de `scopes` e a política de rate limit antes do merge.

---

## Contexto

- Auth atual: `packages/api/src/middleware/auth.ts` resolve usuário via Clerk e popula `request.userId`/`request.userRole`. Esse pipeline **não** deve rodar para rotas públicas de API key.
- Padrão Fastify de hooks/plugins: ver `app.ts` (registro do plugin Clerk) e `plugins/` (ex.: `meta-names-scheduler.ts`, `guestGuardPlugin`).
- Webhook Clerk (`routes/webhooks.ts`) já é exemplo de rota **sem** auth Clerk, protegida por assinatura — provando que dá pra ter rotas fora do pipeline padrão.

---

## Aceitação

### AC1: Plugin/preHandler de API Key isolado
- [ ] Criar `packages/api/src/middleware/api-key-auth.ts` (ou plugin Fastify) que:
  - Lê o header `X-API-Key`.
  - Calcula `hashApiKey(raw)` (reutiliza helper da 36.1) e busca em `api_keys` por `keyHash` (índice criado na 36.1).
  - Compara com **`crypto.timingSafeEqual`** (constant-time) — nunca comparação de string direta.
  - Rejeita com `401` se ausente/inválida; `403` se `revokedAt != null`.
  - Em sucesso, popula `request.apiKey = { id, scopes }` e atualiza `lastUsedAt = now` (fire-and-forget, sem bloquear a resposta).

### AC2: Escopo de aplicação restrito
- [ ] O middleware só roda no prefixo **`/api/public/*`** (ou via `scope`/encapsulamento do registro de rotas) — **nunca** nas rotas Clerk existentes.
- [ ] As rotas públicas **não** passam pelo pipeline Clerk (sem `request.userId`).
- [ ] Garantir read-only: o middleware (ou as rotas) rejeita métodos não-GET com `405`.

### AC3: Verificação de scope
- [ ] Helper `requireScope(scope)` que checa se `request.apiKey.scopes` inclui o scope necessário (ex.: `'meta:read'`); senão `403`.
- [ ] Vocabulário inicial de scopes: `meta:read` (validar com @architect se precisa granularidade tipo `meta:creatives:read`).

### AC4: Rate limiting
- [ ] Rate limit por chave (ex.: usar `@fastify/rate-limit` com `keyGenerator` baseado em `request.apiKey.id`, ou implementação simples in-memory).
- [ ] Limite default razoável (ex.: 120 req/min) — confirmar número com @architect.
- [ ] Resposta `429` com header `Retry-After` quando estourar.

## Out of Scope
- Os endpoints de dados em si (`/campaigns`, `/creatives`, …) → Story 36.3.
- Quotas mensais / billing por chave.
- Geração/revogação de chaves (já é 36.1).

---

## Riscos

| Risco | Prob. | Impacto | Mitigação |
|-------|-------|---------|-----------|
| Timing attack na comparação de hash | Baixa | Alto | `crypto.timingSafeEqual` obrigatório (AC1) |
| Middleware vazar para rotas Clerk | Baixa | Crítico | Escopo por prefixo `/api/public/*` + teste cobrindo isolamento |
| `lastUsedAt` write a cada request degradar performance | Média | Baixo | fire-and-forget / debounce; não bloqueia resposta |
| Rate limit in-memory não funcionar com múltiplas instâncias | Média | Médio | Documentar; se houver multi-instância, migrar p/ store compartilhado (story futura) |

---

## Notas Técnicas
- **Isolamento Clerk:** registrar as rotas públicas num escopo Fastify separado onde o hook de auth Clerk **não** é aplicado. Conferir como o plugin Clerk é registrado em `app.ts` para garantir que ele não é global, ou usar `fastify.register` aninhado.
- **Constant-time:** buscar por `keyHash` (que já é hash) e ainda assim usar `timingSafeEqual` na comparação do hash recebido vs. armazenado.
- **lastUsedAt:** `void fastify.db.update(...).catch(() => {})` para não acoplar à latência da resposta.
