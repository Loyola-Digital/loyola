## 📖 Contexto

Na tabela **Desempenho de Criativos** (Story 18.24), cada linha representa **um `ad_name`** — mas o mesmo criativo é veiculado em vários adsets e a Meta cria **N `ad_id` distintos com o mesmo `ad_name`**. Hoje a tabela mostra métricas de performance (invest., CTR, leads/ingressos, faturamento), mas **não mostra se o criativo ainda está rodando**. O gestor de tráfego precisa saber, ao bater o olho, quais criativos estão **ativos** e quais foram **pausados**, sem abrir o Gerenciador de Anúncios.

Como um `ad_name` mapeia para vários `ad_id`, a regra de negócio é uma **agregação OR**:

> Um criativo (linha) é considerado **Ativo** se **pelo menos um** dos seus `ad_id` estiver ativo na Meta. Só é **Pausado** quando **todos** os `ad_id` daquele `ad_name` estiverem não-ativos.

Incremento pedido pelo usuário (avaliado e **aprovado** — custo baixo): quando a linha está **Ativa**, um **tooltip** mostra **em qual(is) adset** o criativo ainda está rodando (os adsets dos `ad_id` ativos).

### 🔬 Fonte de dados (investigado no código)

- **Agrupamento `ad_name` → `ad_id[]` já existe no backend.** `packages/api/src/routes/stage-creative-performance.ts:553-606` monta `AdNameGroup` com `adIds: string[]` (todos os ad_ids que compartilham o nome). O `ad_id` representativo da linha é `group.adIds[0]` (`:657`). **Esta é a base exata da regra OR** — já temos a lista de ad_ids por linha.
- **O status NÃO é buscado nem persistido hoje.** `fetchAllAdInsights` (`meta-ads.ts:880`) traz insights por ad (com `adset_name`/`adset_id`) mas **sem `status`/`effective_status`**. `effective_status` só aparece como *filtro* em `fetchCampaigns` (`meta-ads.ts:389`) e no backfill (`meta-names-backfill.ts:67`) — **nunca é pedido nos `fields` nem retornado**. Nenhuma tabela de cache Meta do banco tem coluna de status (`schema.ts:1651-1805`).
- **`adset_name` JÁ está salvo por `ad_id`** em `meta_ad_insights_daily.adset_name` (`schema.ts:1735`, gravado em `meta-insights-cache.ts:377`) e também vem em cada insight de `fetchAllAdInsights`. Portanto o tooltip de adset **não exige chamada nova** à Meta.
- **Cache "atual" por entidade:** `meta_entity_names_cache` (`schema.ts:1651-1666`) guarda `project_id, entity_type (ad/adset/campaign), entity_id, entity_name, last_synced_at`. É o lugar natural para o `effective_status` (estado atual, não histórico por dia). Populado em lote de 50 ids por `resolveEntityNames` (`meta-ads.ts:1396-1487`, padrão `/?ids=<csv>&fields=id,name`).
- **Regra obrigatória batch + cache:** só o SYNC chama a Meta; o dashboard lê do banco (`meta-db-source.ts:67-80`; memória `feedback_meta_api_rate_limit_batching`). O status seguirá o mesmo caminho.
- **Padrão de UI de badge de status já existe** em `perpetual-dashboard.tsx:217-233` (`STATUS_BADGE`/`StatusBadge`: ACTIVE→Ativo, PAUSED→Pausado, fallback "—") — reusar o padrão visual.

## 🗳️ Decisões de escopo (elicitação com o usuário, 2026-07-18)

| Dúvida | Decisão |
|--------|---------|
| Como obter o status por `ad_id` (não existe hoje)? | **Cache no banco via sync.** Estender `meta_entity_names_cache` com coluna `effective_status`, populada em lote de 50 ids pelo caminho de nomes já existente (`resolveEntityNames`/backfill). Dashboard lê do banco. Respeita a regra batch+cache. |
| Tooltip "em qual adset está ativo"? | **Incluir nesta story.** `adset_name` já está no banco/insights → custo baixo. Tooltip aparece só quando a linha está Ativa, listando os adsets dos `ad_id` ativos. |
| Como tratar `ARCHIVED` e outros estados? | **Binário Ativo/Pausado.** Só `ACTIVE` = "Ativo"; qualquer outro (`PAUSED`, `ARCHIVED`, `CAMPAIGN_PAUSED`, `ADSET_PAUSED`, `DISAPPROVED`, etc.) = "Pausado". |
| Regra de agregação por `ad_name`? | **OR:** Ativo se ≥1 `ad_id` ACTIVE; Pausado só se todos não-ativos. |
| Alcance da coluna (quais etapas)? | **Todas** as etapas onde a tabela aparece (paga/gratuita/normal). |
| Linhas sem `ad_id` da Meta (só planilha) ou status desconhecido? | Exibir **"—"** (badge neutro), nunca "Pausado" por falta de dado. |
| Ordenação/filtro pela nova coluna? | Coluna **ordenável** (Ativo antes de Pausado por padrão de sort). Filtro dedicado por status fica **OUT** (follow-up). |

## ✅ Acceptance Criteria

**AC1 — Coluna "Status" na tabela (todas as etapas)**
**Given** a tabela Desempenho de Criativos carregada para qualquer etapa (paga, gratuita ou normal), **when** os dados são renderizados, **then** existe uma coluna **"Status"** exibindo um badge **"Ativo"** (verde) ou **"Pausado"** (cinza/neutro) por linha de criativo.
- Posição: logo após a coluna **Ad Name** (primeira coluna de dados), antes de Invest.
- Reusar o padrão visual de `STATUS_BADGE`/`StatusBadge` (`perpetual-dashboard.tsx:217-233`); não recriar do zero. ⚠️ Reconciliar a cor: o `STATUS_BADGE` existente mapeia `PAUSED` para **amarelo** (`text-yellow-...`, `:222`); alinhar Ativo=verde / Pausado=neutro-ou-amarelo de forma consistente (decidir na implementação, sem divergir no QA).

**AC2 — Regra de agregação OR por `ad_id`**
**Given** uma linha que agrupa múltiplos `ad_id` sob o mesmo `ad_name` (`group.adIds`, `stage-creative-performance.ts:553-606`), **when** o status é resolvido, **then** a linha é **"Ativo"** se **pelo menos um** `ad_id` do grupo tiver `effective_status === "ACTIVE"`, e **"Pausado"** somente quando **todos** os `ad_id` do grupo forem não-ativos.
- Qualquer `effective_status` diferente de `ACTIVE` conta como não-ativo (`PAUSED`, `ARCHIVED`, `CAMPAIGN_PAUSED`, `ADSET_PAUSED`, `DISAPPROVED`, `PENDING_REVIEW`, etc.).

**AC3 — Status "—" quando desconhecido**
**Given** uma linha sem `ad_id` da Meta (linha só de planilha) **ou** cujo(s) `ad_id` não têm `effective_status` no cache, **when** renderizada, **then** exibe **"—"** (neutro) — **nunca** "Pausado" por ausência de dado.

**AC4 — Tooltip de adsets ativos (só quando Ativo)**
**Given** uma linha com status **"Ativo"**, **when** o usuário passa o mouse sobre o badge, **then** um tooltip exibe a lista **distinta** de `adset_name` correspondentes aos `ad_id` **ativos** daquela linha (ex.: "Ativo em: Público Lookalike 1%, Interesses — Empreendedorismo").
- `adset_name` vem dos insights já buscados / do banco (`meta_ad_insights_daily.adset_name`) — **sem** nova chamada à Meta.
- Linhas "Pausado" ou "—" não têm tooltip de adset (ou têm tooltip vazio/omitido).

**AC5 — Backend: persistir `effective_status` no cache de entidades (batch + banco)**
**Given** o sync de nomes/entidades Meta em execução, **when** os `ad_id` são resolvidos em lote de 50 (`resolveEntityNames`/`meta-names-backfill`), **then** o request à Graph API passa a incluir `effective_status` nos `fields` (`/?ids=<csv>&fields=id,name,effective_status`) e o valor é gravado numa **nova coluna `effective_status`** em `meta_entity_names_cache` (migration nova).
- Só o SYNC chama a Meta; o endpoint/dashboard lê do banco (regra `feedback_meta_api_rate_limit_batching`).
- `last_synced_at` existente serve de TTL — sem nova política de expiração.

**AC6 — Backend: expor status por linha no endpoint**
**Given** `GET /api/funnels/:funnelId/stages/:stageId/creative-performance`, **when** monta cada linha por `ad_name` (`:608-679`), **then** inclui na resposta:
- `status: "active" | "paused" | "unknown"` — resolvido pela regra OR (AC2/AC3) cruzando `group.adIds` com o `effective_status` do cache.
- `activeAdsets?: string[]` — `adset_name` distintos dos `ad_id` ativos (para o tooltip da AC4); ausente/vazio quando não Ativo.
- Campos adicionados de forma **aditiva** (não quebram consumidores existentes).

**AC7 — Coluna ordenável**
**Given** a coluna Status, **when** o usuário clica no header, **then** ordena por status (Ativo → Pausado → "—" no sentido padrão, invertendo no segundo clique), consistente com o mecanismo de sort client-side já existente na tabela (`stage-creative-performance-table.tsx`).

**AC8 — Regressão zero**
**Given** as demais colunas e comportamentos (métricas Único/Total 18.55, CTR link_click 18.59, crossref de leads, temperatura hot/cold, preview Meta Library, imposto 12,15%), **when** a coluna Status é adicionada, **then** nada regride: mesmos números, mesma ordenação padrão das outras colunas, mesmo layout responsivo.
- Gates: `pnpm --filter @loyola/api exec tsc --noEmit` e `pnpm --filter web exec tsc --noEmit` verdes; `next build` (web) verde.

## 📊 Complexidade & Valor

- **Estimativa:** M (medium), tendendo a M-L — **toca backend (Meta fields + migration + cache read/write), endpoint e frontend**. O agrupamento por `ad_name` e o `adset_name` já existirem reduz o esforço; o custo real está em (a) adicionar `effective_status` ao caminho de resolução de entidades sem estourar rate-limit e (b) a migration de coluna nova.
- **Valor de negócio:** Alto para operação diária — o gestor identifica na hora criativos pausados (sem investir mais tempo no Gerenciador) e, via tooltip, sabe exatamente onde um criativo vencedor ainda está rodando para escalar/duplicar.

---

## ⚠️ Riscos

- **R1 (médio)** — `effective_status` no endpoint de entidade `/?ids=` pode não estar disponível para todos os tipos ou retornar estados inesperados. **Mitigação:** tratar qualquer valor ≠ `ACTIVE` como "Pausado" e ausência como "—" (AC3); validar em conta real antes do QA.
- **R2 (médio)** — Custo/rate-limit da Graph API ao adicionar `effective_status` ao batch (a Story 29.21 cortou isso por esse motivo). **Mitigação:** reutilizar o **mesmo** batch de 50 ids do caminho de nomes (`resolveEntityNames`), sem chamadas adicionais — apenas um campo a mais no `fields`. Ler sempre do banco no request do dashboard.
- **R3 (baixo)** — Status "atual" pode divergir do período de datas da tabela (um criativo pausado hoje teve performance no período). **Mitigação:** deixar explícito que Status = estado **atual** na Meta (não do período); comportamento idêntico ao badge do dashboard perpétuo (29.21).
- **R4 (baixo)** — Linha compilada no modo "Todos" (`compileCreativeMetrics.ts`) reagrupa por `ad_name` no frontend. **Mitigação:** garantir que `status`/`activeAdsets` sobrevivam à compilação (propagar via OR ao mesclar, não sobrescrever).

"paused" : "unknown")` e `activeAdsets` = distinct dos `adset_name` dos ad_ids ativos. O `adset_name` por ad_id já vem em cada insight de `fetchAllAdInsights` (`meta-ads.ts:880`) — monte um `Map<adId, adsetName>` a partir dos insights que o endpoint já busca (`:226-231`), evitando novo I/O.
- **Onde vem o `effective_status`:** ele **não** vem de `/insights`; vem do endpoint de **entidade** (`/ads`) ou do batch `/?ids=<csv>&fields=id,name,effective_status`. Reuse `resolveEntityNames` (`meta-ads.ts:1396-1487`) — ele já faz batch de 50 (`META_NAMES_BATCH_LIMIT`, `:1374`) com throttle 200ms para `entity_type='ad'`. Basta adicionar `effective_status` ao `fields` e gravar na coluna nova. **Não confundir** com o uso de `effective_status` em `filtering` de `fetchCampaigns` (`:389`) — aquilo é filtro, não field.
- **Migration:** coluna `effective_status varchar` nullable em `meta_entity_names_cache` (`schema.ts:1651-1666`). Seguir o padrão das migrations existentes (`0034_meta_entity_names_cache.sql`, `0058_...`).
