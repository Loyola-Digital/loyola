# Story 36.3: Endpoints públicos de leitura de performance de criativos Meta Ads

**Epic:** 36 - Meta Ads Creative Intelligence API + MCP
**Story:** 36.3
**Status:** 🔨 Implementado (Lote 1: discovery + Meta) — aguardando testes/validação real/PR
**Priority:** High
**Complexity:** Medium (~1,5 dia)
**Points:** 8
**Assignee:** @dev

---

## Valor de Negócio

Esta é a **carne** do épico: os endpoints que entregam, de forma estável e documentável, a performance dos criativos por campanha. É o contrato que o MCP (36.6) vai consumir e a IA do Lucas vai usar para responder "qual criativo da SCA tem melhor CTR/CPA?". Lê do **cache já existente** (sync+cache, decisão do Lucas), então é rápido e barato.

## Dependências

- **Bloqueante:** Story 36.2 (middleware `X-API-Key` + escopo `/api/public/*`).
- **Reuso direto:** `packages/api/src/services/meta-ads.ts` (criativos + insights), `meta-insights-cache.ts`, tabelas `meta_insights_daily_cache`, `meta_ad_creatives_cache`, `meta_entity_names_cache`.
- **Complementa:** Story 36.4 (mantém o cache fresco) e 36.5 (adiciona ROAS real).

---

## Contexto

- O service `meta-ads.ts` já puxa por criativo: `adId`, `thumbnailUrl`, `imageUrl`, `title`, `body`, `linkUrl`, `ctaType`, `objectType`, `videoId`, além de impressões, reach, clicks, spend, CTR, CPC, CPM, conversões e métricas de vídeo (p25/p50/p75/p100, thruplay). Janela até 365d (chunk 90d).
- Nomes de campaign/adset/ad vêm de `meta_entity_names_cache` (TTL 24h).
- O contrato precisa ser **estável e versionado** (`/api/public/meta/v1/...`) porque um MCP/IA externo vai depender dele.

---

## Aceitação

### AC1: `GET /api/public/meta/v1/campaigns`
- [ ] Retorna lista de campanhas com agregado: `campaignId`, `campaignName`, `spend`, `impressions`, `clicks`, `ctr`, `cpc`, `cpm`, `conversions`, `cpa`, e contagem de criativos ativos.
- [ ] Query params: `from`, `to` (datas ISO; default últimos 30d), `accountId` (opcional, filtra conta Meta).
- [ ] `requireScope('meta:read')`.
- [ ] Lê do cache; se faltar dado no range, retorna o que tem + flag `partial: true` (não chama Graph API ao vivo aqui — isso é trabalho do 36.4).

### AC2: `GET /api/public/meta/v1/creatives`
- [ ] Retorna performance **por criativo**: todos os campos de criativo (thumbnail, title, body, cta, tipo) + métricas (spend, impressions, ctr, cpc, cpm, conversions, cpa) + métricas de vídeo quando aplicável.
- [ ] Query params: `campaignId` (opcional, filtra), `from`, `to`, `orderBy` (ex.: `ctr`, `cpa`, `spend`), `limit`.
- [ ] `requireScope('meta:read')`.

### AC3: `GET /api/public/meta/v1/creatives/:adId/timeseries`
- [ ] Retorna **série temporal diária** das métricas de um criativo (de `meta_insights_daily_cache`, `time_increment=1`): array de `{ date, spend, impressions, clicks, ctr, conversions, ... }`.
- [ ] Query params: `from`, `to`.
- [ ] `requireScope('meta:read')`. `404` se `adId` desconhecido.

### AC4: Contrato estável + erros consistentes
- [ ] Responses tipados (schema compartilhado em `packages/shared` se fizer sentido) — chaves estáveis, datas ISO, números (não strings).
- [ ] Erros padronizados `{ error, details }`, códigos `400/401/403/404/429`.
- [ ] Documentar o shape de cada response (alimenta o `llms.txt` da 36.6).

## Riscos

| Risco | Prob. | Impacto | Mitigação |
|-------|-------|---------|-----------|
| Cache vazio/desatualizado no range pedido | Média | Médio | Flag `partial`; 36.4 mantém fresco; documentar comportamento |
| Contrato mudar depois quebra o MCP | Média | Alto | Versionar `/v1/`; mudanças aditivas; nunca remover campo sem `/v2/` |
| Volume de criativos grande sem paginação | Baixa | Médio | `limit` + ordenação default; paginação se necessário |

---

## Notas Técnicas
- **Reuso máximo:** estes endpoints devem ser uma **camada fina de leitura** sobre os services/caches existentes — não reimplementar a lógica de agregação que já existe em `meta-ads.ts`/`meta-insights-cache.ts`. Extrair função compartilhada se a lógica hoje estiver acoplada às rotas Clerk.
- **Mesma fonte da verdade:** as métricas aqui devem bater com o que o dashboard interno mostra (mesma agregação) — evitar divergência de números entre dashboard e API.
