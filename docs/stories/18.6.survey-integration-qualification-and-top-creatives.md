## Story

Como gestor de tráfego do Loyola X, quero integrar as respostas da **pesquisa Tally captação** (já vinculada via aba Pesquisas) em dois lugares do dashboard:

1. **Nova seção no fim do dash** — mostra as % de respostas das 3 perguntas de qualificação (faturamento, profissão, nº funcionários) em barras horizontais, indicando o perfil do público captado
2. **Dentro dos cards do Top Criativos** — mostra a resposta mais frequente de **faturamento** e **profissão** para cada criativo (cruzando `utm_content` da pesquisa = `ad_id` do Meta)

Para que eu consiga avaliar não só se o criativo "funciona" (CPL), mas **que tipo de público ele atrai**.

## Contexto

### Dados disponíveis

- Pesquisas vinculadas ao funil via `useFunnelSurveys(projectId, funnelId)` — retorna lista de `FunnelSurvey[]` (cada uma é 1 aba do Google Sheet Tally captação)
- Dados das abas via `useSheetData(spreadsheetId, sheetName)` — retorna `{ headers: string[], rows: string[][] }` (sem tipagem de cada coluna)
- `SurveySummary` já traz `totalResponses` e `responseRate` — só precisa processar breakdown por pergunta no frontend

### Perguntas alvo (confirmadas com Danilo)

Colunas esperadas nos headers da planilha Tally:

| Coluna | Pergunta | Usado em |
|--------|----------|----------|
| "Qual seu faturamento mensal?" | Faturamento | 3.a + 3.b |
| "Qual sua profissão?" | Profissão | 3.a + 3.b |
| "Quantos funcionários você tem?" | Nº funcionários | 3.a apenas |
| "utm_content" | match com `ad_id` | 3.b (cruzamento) |
| `<coluna de timestamp>` | filtro por data | ambos |

**Nota:** se houver múltiplas surveys vinculadas ao funil (cenário multi-aba), **todas** são processadas e agregadas (respostas somadas).

### Decisão arquitetural

- **Frontend-only:** processamento todo no cliente via `useMemo` sobre `useSheetData`
- **Hook novo:** `useSurveyAggregation(projectId, funnelId, days)` — busca surveys vinculadas, agrega respostas, normaliza, aplica filtro por data (com fallback)
- **MVP hardcoded** (approach α confirmado): perguntas-alvo definidas em constante no código; configuração via UI fica pra story futura
- **Normalização de respostas:** `lowercase + trim + remove accents` pra agrupar variações ("Médico", "medico", "MEDICO" → mesma opção)

### Normalização de respostas

```ts
function normalizeAnswer(raw: string): string {
  return raw
    .toLowerCase()
    .trim()
    .normalize("NFD").replace(/[\u0300-\u036f]/g, "")  // remove acentos
    .replace(/\s+/g, " ");
}
```

**Display:** sempre usar a versão normalizada como chave de agrupamento, mas exibir a versão **mais comum** (moda) na UI — ex: agrupar ("Médico", "medico", "MEDICO") vira label "Médico" se foi o mais frequente.

### Fallback de filtro de data

```
1. Filtrar surveyRows por data (usando helper da 18.2)
2. Se filteredRows.length < 10:
   → usar surveyRows (sem filtro)
   → retornar { usingFallback: true, fallbackReason: "Apenas N respostas no período" }
3. Senão:
   → usar filteredRows
```

Badge visual quando `usingFallback: true`:
- Na seção 3.a: alerta amarelo no topo do bloco
- Nos cards do Top Criativos: ícone `⚠️` discreto (sem poluir)

## Dev Notes

- **Headers da planilha Tally:** perguntas-alvo são "Qual seu faturamento mensal?", "Qual sua profissão?", "Quantos funcionários você tem?". Os `matchers` no `SURVEY_QUESTION_MAP` usam termos parciais case-insensitive pra serem resilientes a pequenas variações de nomenclatura.
- **Coluna de timestamp:** Tally geralmente chama de "Submitted at". Incluir matchers alternativos ("timestamp", "data"). Se @dev identificar outro nome padrão durante implementação, adicionar ao matcher.
- **Match `utm_content` = `ad_id`:** Danilo confirmou match exato (case-sensitive). Se na prática houver drift (ex: spaces, null), sinalizar no PR.
- **Multi-survey:** `useFunnelSurveys` retorna N surveys do funil. O hook chama `useSheetData` pra cada uma e concatena as linhas. Usar `useQueries` do react-query pra paralelizar.
- **Top-8 "Outros":** se futuramente quiser ver a lista completa, expandir vira story de UX. Agora é agregado simples.
- **Performance:** agregação é O(surveys × rows × perguntas). Funil típico tem 1-2 surveys × poucos milhares de rows × 3 perguntas = trivial. `useMemo` com deps `[surveyData, days]`.
- **Privacidade:** respostas podem conter dados pessoais (email na Coluna 2). O hook deve ignorar colunas não mapeadas (não processar email, telefone, etc).

## Risks

- **Alto:** Nome dos headers da planilha Tally pode não bater com matchers. Mitigação: matchers generosos + warning em dev quando header esperado ausente. Backup: Danilo pode ajustar matchers no constants file rapidamente.
- **Médio:** Respostas muito variadas (muitas profissões únicas) podem gerar barras pequenas e poluir UI. Top-8 + "Outros" mitiga.
- **Médio:** Multi-survey com estruturas diferentes entre as abas. Se uma aba não tem coluna de faturamento e outra tem, o agregado fica enviesado pra quem tem. Mitigar com warning em dev + plano de story futura pra configuração explícita.
- **Baixo:** Normalização pode agrupar demais (ex: "Médico veterinário" ≠ "Médico" mas uma fuzzy match poderia errar). Usando apenas normalize básico (sem fuzzy), só agrupa exatas após lowercase/trim/remove-accents — é o comportamento seguro.
- **Baixo:** Performance — agregação O(n × m) local, com 2k-5k rows típicos é trivial.
