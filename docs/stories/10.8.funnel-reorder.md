# Story 10.8 — Reordenação de funis no sidebar (perpétuos > lançamentos)

**Epic:** 10 — Funnel Architecture
**Status:** Done
**Priority:** Medium
**Estimate:** M
**Depende de:** 10.4 (sidebar-funnels) — `FunnelItem` em `project-folder.tsx`

---

## User Story

Como gestor do Loyola X, quero **arrastar funis** dentro do sidebar do projeto pra escolher a ordem que aparecem, **mantendo a regra de que perpétuos sempre aparecem antes de lançamentos**, pra ver primeiro os negócios sempre-ativos e logo abaixo os lançamentos em ordem cronológica/relevância da minha escolha. Dentro de cada grupo (perpétuos entre si, lançamentos entre si) eu controlo a ordem livremente; entre grupos a regra é fixa.

---

## Contexto Técnico

### Modelo atual

- Tabela `funnels` (schema.ts:540) não tem coluna de ordem — `GET /api/projects/:projectId/funnels` retorna na ordem natural do DB (insertion order).
- Lista no sidebar renderiza em `packages/web/components/layout/project-folder.tsx:658` via `funnelList?.map((funnel) => <FunnelItem .../>)`.
- `funnelList` vem de `useFunnels(projectId)` (hook em `use-funnels.ts:65`).
- Já existe drag-and-drop em outras partes do app: `sortable-stage-grid.tsx` (reorder de stages dentro de um funil) — pattern com `@dnd-kit/core` + `@dnd-kit/sortable` (já em `package.json`).
- `FunnelType = "launch" | "perpetual"` (shared/types/funnel.ts:1).

### Mudança proposta

1. Adicionar coluna `sort_order` (integer, default 0) na tabela `funnels`.
2. Backend ordena response do GET por `(type DESC NULLS LAST, sort_order ASC, created_at ASC)` — `"perpetual"` > `"launch"` alfabeticamente em ordem decrescente coloca perpetuais primeiro.
3. Novo endpoint `PUT /api/projects/:projectId/funnels/reorder` body: `{ ids: string[] }` — recebe array com a nova ordem, valida que respeita a regra (perpétuos contíguos no início), persiste `sort_order` igual ao índice.
4. Frontend envolve a lista de `FunnelItem` em 2 `SortableContext` separados (um por grupo: perpétuos, lançamentos) — drag livre dentro do grupo, bloqueado entre grupos.
5. Update otimista no cache do React Query; reverte se backend rejeitar.

### Por que 2 SortableContext separados

- Isola fisicamente os grupos: arrastar um funil só vê o grupo dele como drop target.
- Implementa a hard rule no nível da UI — usuário **não consegue** soltar um lançamento no meio dos perpétuos (sem precisar de feedback de erro).
- Padrão suportado nativamente pelo `@dnd-kit`.

## Acceptance Criteria

1. **AC1 — Schema:** Migration `0042_funnels_sort_order.sql` adiciona coluna `sort_order integer NOT NULL DEFAULT 0` na tabela `funnels`. Drizzle schema atualizado em `packages/api/src/db/schema.ts`.

2. **AC2 — Backfill:** Migration popula `sort_order` pra funis existentes baseado em `created_at` ASC, agrupado por `type` (perpétuos recebem 0,1,2,... e lançamentos recebem 0,1,2,...). Garante que ordem inicial após deploy seja perpetuais-por-data ASC + lançamentos-por-data ASC.

3. **AC3 — GET ordenação:** `GET /api/projects/:projectId/funnels` retorna funis ordenados:
   - 1º critério: `type = 'perpetual'` antes de `type = 'launch'`
   - 2º critério: `sort_order ASC`
   - 3º critério (tiebreak): `created_at ASC`

4. **AC4 — Endpoint reorder:** `PUT /api/projects/:projectId/funnels/reorder`:
   - Body: `{ ids: string[] }` (array com TODOS os funis do projeto na nova ordem)
   - Valida: todos os IDs pertencem ao projeto (senão 400 "ID desconhecido")
   - Valida: array contém TODOS os funis do projeto, sem duplicatas (senão 400 "Lista incompleta ou com duplicatas")
   - Valida: ordem respeita a hard rule — todos os perpétuos vêm antes de qualquer lançamento (senão 400 "Perpétuos devem vir antes de lançamentos")
   - Persiste: `UPDATE funnels SET sort_order = idx WHERE id = ids[idx]` em transação
   - Role `guest` → 403
   - Retorna `{ success: true }`

5. **AC5 — Hook frontend:** `useReorderFunnels(projectId)` em `use-funnels.ts` — mutation que chama o PUT acima e invalida `["funnels", projectId]` após sucesso.

6. **AC6 — UI separação por grupo:** Em `project-folder.tsx`, a lista atual de `FunnelItem` é dividida em 2 grupos visuais sequenciais:
   - Primeiro renderiza `<SortableContext items={perpetualIds}>` com `FunnelItem` dos `type === "perpetual"`
   - Logo abaixo `<SortableContext items={launchIds}>` com `FunnelItem` dos `type === "launch"`
   - Sem header de seção (visual idêntico ao atual, mantém continuidade) — usuário percebe a separação pelo comportamento do drag.

7. **AC7 — Drag handle:** Cada `FunnelItem` ganha um drag handle (ícone `GripVertical` da lucide) visível em hover (mesma estética de `sortable-stage-grid.tsx`). Click no handle não navega; arrasta o item.

8. **AC8 — Update otimista:** Ao soltar o item, ordem local atualiza imediatamente. Se backend rejeitar (erro 4xx), ordem reverte e toast vermelho "Erro ao reordenar — voltei pra ordem anterior".

9. **AC9 — Bloqueio entre grupos:** Não é possível arrastar um lançamento pra dentro do grupo de perpétuos (e vice-versa). Sortable contexts isolam.

10. **AC10 — Cross-context drop:** Tentar arrastar pra fora do próprio `SortableContext` (entre os 2 grupos) simplesmente solta no fim do grupo de origem (sem mudança) — nenhum erro, nenhum toast.

11. **AC11 — Empty state:** Se o projeto tem só 1 tipo de funil (só perpetuais ou só lançamentos), o outro `SortableContext` não renderiza nada (sem espaço extra).

12. **AC12 — Sem regressão:** `FunnelItem` continua exibindo todos os detalhes atuais (label, ícone, dropdown menu, compareFunnel, etc). Click no nome do funil continua navegando. Click no menu de ações continua funcionando. Stages dentro do funil mantêm seu próprio drag (sortable-stage-grid intocado).

13. **AC13 — Persistência:** Ao recarregar a página, ordem definida pelo usuário se mantém (validado via `sort_order` no DB).

14. **AC14 — Typecheck zero erros:** `pnpm -r typecheck` passa limpo em shared/api/web.

## API Changes

```ts
// PUT /api/projects/:projectId/funnels/reorder
// Request body
{ ids: string[] }  // UUIDs de TODOS os funis do projeto na nova ordem

// Response 200
{ success: true }

// Response 400 — perpétuos depois de lançamentos
{ error: "Perpétuos devem vir antes de lançamentos" }

// Response 400 — ids incompletos/com dup
{ error: "Lista incompleta ou com duplicatas" }
```

```sql
-- Migration 0042_funnels_sort_order.sql
ALTER TABLE funnels ADD COLUMN sort_order integer NOT NULL DEFAULT 0;

-- Backfill: ordem inicial = posição cronológica dentro do tipo (0-indexed)
WITH ranked AS (
  SELECT
    id,
    row_number() OVER (PARTITION BY type ORDER BY created_at ASC) - 1 AS rn
  FROM funnels
)
UPDATE funnels f
SET sort_order = ranked.rn
FROM ranked
WHERE f.id = ranked.id;
```

## Risks

- **Hard rule confunde usuário:** Usuário pode tentar arrastar lançamento pra cima dos perpetuais e não entender por que não solta. Mitigação: drag handles visualmente distintos por grupo (cor sutil) e/ou um Separator entre os 2 SortableContexts (sem texto). Decidir em refinamento — começa sem separator.
- **Backfill em prod:** Migration roda em DB que já tem funis ativos. CTE com row_number é safe (não bloqueia) mas em projeto muito grande pode demorar. Risco baixo (Loyola tem <50 funis no total).
- **Race condition:** Dois admins reordenando ao mesmo tempo — última escrita ganha. Aceitável; React Query refetcha em foco/janela ativa.
- **Mobile sem hover:** Drag handle invisível em mobile pode bloquear feature. Mitigação: classe `sm:opacity-0 sm:group-hover/sortable:opacity-100` mantém handle visível em mobile (opacidade default 60%) — mesmo padrão de `sortable-stage-grid.tsx:115`.

## Dev Notes

- Hard rule "perpétuos antes de lançamentos" enforced em 2 lugares: backend (PUT reorder valida a ordem do array antes de gravar) **e** UI (2 SortableContexts separados — fisicamente não dá pra arrastar entre grupos).
- Update no UPDATE usa `CASE WHEN id THEN idx END` num único statement — N=10 funis = 1 query SQL em vez de 10.
- `SortableFunnelItem` injeta `disabled: !isAdmin` no `useSortable` — guests não conseguem nem iniciar drag.
- `sort_order` é populado por backfill (CTE com row_number()) na migration — funis existentes ganham ordem cronológica ASC dentro do tipo.
- Index `(project_id, type, sort_order)` casa exato com o ORDER BY do GET.
- Drag handle posicionado `-left-1` com `opacity-0 group-hover:opacity-100` — mesmo padrão visual do SortableStageNavItem que já existia.
