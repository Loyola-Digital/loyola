# Story 36.5: ROAS real — cruzar spend do Meta Ads com vendas reais (Kiwify/Hotmart)

## Valor de Negócio

Este é o **ouro do épico**: a IA do Lucas não vai só ver CTR/CPA do pixel — vai ver **ROAS real** (faturamento de vendas confirmadas Kiwify/Hotmart ÷ spend Meta) por campanha e por criativo. É a diferença entre "esse criativo tem bom CTR" e "esse criativo me deu R$ X de venda real para cada R$ 1 investido". É a pergunta de negócio que realmente importa para decisão de escala.

## Dependências

- **Bloqueante:** Story 36.3 (endpoints públicos onde os campos de ROAS serão adicionados).
- **Reuso:** lógica de classificação de origem já existente — `classifyFonte` / `PAID_UTM_SOURCES` (vista nas Stories 18.x, ex.: 18.29 notas técnicas, 18.4 hot/cold buyers por utm_term), services `kiwify.ts`/`hotmart.ts`, endpoint `sales-data`.
- ⚠️ **GATE DE DESIGN:** o **modelo de atribuição venda↔campanha/criativo** precisa de validação do **@architect + @data-engineer + ok do Lucas** ANTES de implementar. É o maior risco do épico.

---

## Contexto / Questão central de atribuição

Para cruzar uma venda Kiwify/Hotmart com um criativo Meta, precisamos de uma **chave de atribuição**. O LoyolaX já faz cruzamento UTM em outras telas (crossref de vendas, hot/cold buyers por `utm_term`). As hipóteses a confirmar no gate:

1. **Via UTM:** a venda carrega `utm_source`/`utm_campaign`/`utm_content`/`utm_term` que mapeiam para campanha/criativo Meta. Qual campo bate com qual nível (campanha vs. ad)?
2. **Janela de atribuição:** quantos dias entre clique e venda contam? (ex.: 7d, 30d).
3. **Granularidade:** dá pra atribuir até o **criativo** (ad) ou só até a **campanha**? Depende de `utm_content`/`utm_term` estarem populados nos links dos criativos.
4. **Origem:** `PAID_UTM_SOURCES` já define o que é "pago" — reusar.

**Esta story NÃO inventa atribuição nova** — formaliza e reusa o que já existe nas Stories 18.x. O gate é para confirmar o mapeamento exato, não para projetar do zero.

## Aceitação

> ⚠️ ACs abaixo assumem o modelo de atribuição aprovado no gate. Ajustar após decisão.

### AC1: Função de atribuição reutilizável
- [ ] Função que, dado um período e (campanha|adId), retorna vendas reais atribuídas: `{ vendas, faturamento }`, reusando `classifyFonte`/`PAID_UTM_SOURCES` e a fonte de vendas (`sales-data`/`kiwify.ts`/`hotmart.ts`).
- [ ] Respeita a janela de atribuição definida no gate.
- [ ] Documenta explicitamente o nível de granularidade alcançável (campanha sempre; criativo só se `utm_content`/`utm_term` disponível).

### AC2: ROAS real nos endpoints públicos
- [ ] `GET /api/public/meta/v1/campaigns` ganha campos: `realRevenue`, `realSales`, `realRoas` (= realRevenue ÷ spend), `attributionLevel` (`campaign`|`creative`).
- [ ] `GET /api/public/meta/v1/creatives` ganha os mesmos campos quando a atribuição por criativo é possível; senão `realRoas: null` + `attributionLevel: 'campaign'` (não inventar número).
- [ ] Mudanças **aditivas** (não quebram o contrato `/v1/` da 36.3).

### AC3: Honestidade do dado
- [ ] Quando não há atribuição confiável (UTM ausente), retorna `null` — **nunca** estimar/inventar ROAS.
- [ ] Campo `attributionCoverage` (ex.: % de vendas que puderam ser atribuídas) para a IA saber o quão confiável é o número.

## Out of Scope

- Reescrever a coleta de vendas Kiwify/Hotmart (já existe).
- UI de dashboard para ROAS real (o foco é a API/MCP; dashboard pode ser story futura).

## Riscos

| Risco | Prob. | Impacto | Mitigação |
|-------|-------|---------|-----------|
| UTM dos criativos não permite atribuição por ad | Alta | Alto | Gate confirma granularidade; fallback honesto p/ campanha + `null` por criativo |
| Janela de atribuição errada infla/deflaciona ROAS | Média | Alto | Definir janela no gate; documentar; deixar configurável |
| Divergência vs. crossref do dashboard interno | Média | Médio | Reusar exatamente `classifyFonte`/`PAID_UTM_SOURCES`; conferir números |
| Vendas sem UTM (orgânico/sem track) contadas como pagas | Média | Alto | `PAID_UTM_SOURCES` já filtra; `attributionCoverage` expõe a lacuna |

- **Decisão do gate vai aqui:** após @architect/@data-engineer, registrar o modelo aprovado nesta seção antes de @dev começar.
