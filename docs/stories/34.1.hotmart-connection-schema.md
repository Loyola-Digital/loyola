## 📖 Story

**Como** desenvolvedor da plataforma Loyola Digital X,
**Quero** uma tabela `hotmart_connections` (com credenciais OAuth2 criptografadas por projeto) declarada no schema Drizzle, com migration versionada e script de aplicação,
**Para** que as stories seguintes do Epic 34 (service, rotas, frontend de Assinaturas) tenham onde persistir, de forma segura, o `client_id` e `client_secret` da Hotmart de cada projeto.

---

## ✅ Acceptance Criteria

1. **Tabela declarada no schema Drizzle** (`packages/api/src/db/schema.ts`):
   - Nova export `hotmartConnections = pgTable("hotmart_connections", …)`.
   - Colunas exatas (nomes do contrato de dados):
     - `id` uuid PK default `gen_random_uuid()` NOT NULL
     - `project_id` uuid NOT NULL UNIQUE, FK → `projects(id)` ON DELETE cascade
     - `client_id_encrypted` text NOT NULL
     - `client_id_iv` text NOT NULL
     - `client_secret_encrypted` text NOT NULL
     - `client_secret_iv` text NOT NULL
     - `created_at` timestamptz default `now()` NOT NULL
     - `updated_at` timestamptz default `now()` NOT NULL
   - Index `idx_hotmart_connections_project` em `(project_id)`.
   - Declaração segue exatamente o padrão de `mauticConnections` (mesmas convenções de tipos, `.unique()`, `.references(() => projects.id, { onDelete: "cascade" })`, `index(...)`).

2. **Migration SQL criada** em `packages/api/src/db/migrations/0051_hotmart_connection.sql`:
   - `0051` confirmado como o próximo número livre (último ocupado = `0050_seller_aliases.sql`).
   - Idempotente: `CREATE TABLE IF NOT EXISTS`, guard de constraint FK via bloco `DO $$ … information_schema.table_constraints …`, `CREATE INDEX IF NOT EXISTS`.
   - Usa `--> statement-breakpoint` entre statements (padrão Drizzle do repo).
   - FK com nome `hotmart_connections_project_id_projects_id_fk` referenciando `public.projects(id)` ON DELETE cascade ON UPDATE no action.

3. **Aplicação via runner reutilizável** `packages/api/scripts/run-migration.mjs`:
   - Reusar o runner existente (`node scripts/run-migration.mjs <arquivo.sql>`), que carrega `.env` via dotenv, lê `DATABASE_URL` e aplica cada statement separado por `--> statement-breakpoint`.
   - Aplicar com o arquivo `0051_hotmart_connection.sql`: `node scripts/run-migration.mjs src/db/migrations/0051_hotmart_connection.sql`.
   - Após a aplicação, validar via `information_schema` / `pg_indexes` que a tabela `hotmart_connections`, o index `idx_hotmart_connections_project` e a FK `hotmart_connections_project_id_projects_id_fk` existem; imprimir os resultados.

4. **Reuso de criptografia documentado** (sem novo código de cripto nesta story):
   - Dev Notes referencia `packages/api/src/services/encryption.ts` (`encrypt`/`decrypt`, AES-256-GCM) como a fonte das colunas `*_encrypted` / `*_iv`.
   - O Basic Auth `base64(client_id:client_secret)` é DERIVADO em runtime e **NÃO** é armazenado (apenas client_id e client_secret criptografados).

5. **Migration aplicada e verificada** no banco de dev:
   - `node scripts/run-migration.mjs src/db/migrations/0051_hotmart_connection.sql` roda sem erro e imprime `Migration aplicada com sucesso.` + a tabela/index/FK existentes.

6. **Qualidade**:
   - `npm run typecheck` (ou `pnpm typecheck`) passa sem erros na API após a edição do schema.
   - `npm run lint` passa sem novos erros/warnings.

## 🔧 Dev Notes

### Contrato de dados (nomes EXATOS — não inventar)

Tabela `hotmart_connections`:

| Coluna | Tipo | Constraints |
|--------|------|-------------|
| `id` | uuid | PK, default `gen_random_uuid()`, NOT NULL |
| `project_id` | uuid | NOT NULL, UNIQUE, FK → `projects(id)` ON DELETE cascade |
| `client_id_encrypted` | text | NOT NULL |
| `client_id_iv` | text | NOT NULL |
| `client_secret_encrypted` | text | NOT NULL |
| `client_secret_iv` | text | NOT NULL |
| `created_at` | timestamptz | default `now()`, NOT NULL |
| `updated_at` | timestamptz | default `now()`, NOT NULL |

Index: `idx_hotmart_connections_project` em `(project_id)`.

> **Basic Auth NÃO armazenado.** A Hotmart usa OAuth2 `client_credentials` com header `Authorization: Basic base64(client_id:client_secret)`. Esse Basic é **derivado em runtime** a partir do client_id/client_secret descriptografados — apenas os dois valores criptografados (+ seus IVs) vão pro banco. Conexão é **por projeto** (1 linha por projeto → coluna `project_id` UNIQUE).

### Declaração Drizzle (espelhar `mauticConnections`, schema.ts ~linha 1099)

`mauticConnections` é o template canônico de "credencial criptografada por projeto". A nova export deve seguir o mesmo formato. Esboço de referência:

```ts
// ============================================================
// Story 34.1 — Integração Hotmart (Assinaturas / recorrência)
// Conexão é POR PROJETO (1 credencial OAuth2 client_credentials por cliente).
// client_id e client_secret criptografados via AES-256-GCM (services/encryption.ts).
// O Basic base64(client_id:client_secret) é derivado em runtime, NÃO armazenado.
// ============================================================
export const hotmartConnections = pgTable(
  "hotmart_connections",
  {
    id: uuid("id").defaultRandom().primaryKey(),
    projectId: uuid("project_id")
      .notNull()
      .unique()
      .references(() => projects.id, { onDelete: "cascade" }),
    clientIdEncrypted: text("client_id_encrypted").notNull(),
    clientIdIv: text("client_id_iv").notNull(),
    clientSecretEncrypted: text("client_secret_encrypted").notNull(),
    clientSecretIv: text("client_secret_iv").notNull(),
    createdAt: timestamp("created_at", { withTimezone: true }).defaultNow().notNull(),
    updatedAt: timestamp("updated_at", { withTimezone: true }).defaultNow().notNull(),
  },
  (table) => [index("idx_hotmart_connections_project").on(table.projectId)]
);
```

Imports já presentes em schema.ts (`pgTable`, `uuid`, `text`, `timestamp`, `index`) — nada novo a importar.

### Migration SQL (espelhar `0047_mautic_integration.sql` + guard de FK do `0049`)

`0051_hotmart_connection.sql` deve combinar:
- `CREATE TABLE IF NOT EXISTS "hotmart_connections" (...)` com `CONSTRAINT "hotmart_connections_project_id_unique" UNIQUE("project_id")` inline.
- FK em statement separado, protegida por bloco `DO $$ ... IF NOT EXISTS (SELECT 1 FROM information_schema.table_constraints WHERE constraint_name = 'hotmart_connections_project_id_projects_id_fk' ...) THEN ALTER TABLE ... ADD CONSTRAINT ... FOREIGN KEY ("project_id") REFERENCES "public"."projects"("id") ON DELETE cascade ON UPDATE no action; END IF; END $$;` (padrão idempotente do 0049).
- `CREATE INDEX IF NOT EXISTS "idx_hotmart_connections_project" ON "hotmart_connections" USING btree ("project_id");`
- `--> statement-breakpoint` entre cada statement.

### Script de apply (reusar o runner `run-migration.mjs`)

Reusar o runner reutilizável `packages/api/scripts/run-migration.mjs` (já carrega `.env` via dotenv, guarda `DATABASE_URL`, aplica cada statement separado por `--> statement-breakpoint`). Não criar script dedicado. Aplicar com:

```
node scripts/run-migration.mjs src/db/migrations/0051_hotmart_connection.sql
```

Após aplicar, verificar via psql/queries que os objetos existem:
- coluna/tabela: `information_schema.columns` WHERE `table_name = 'hotmart_connections'`.
- index: `pg_indexes` WHERE `indexname = 'idx_hotmart_connections_project'`.
- FK: `information_schema.table_constraints` WHERE `constraint_name = 'hotmart_connections_project_id_projects_id_fk'`.

### Reuso de criptografia (`packages/api/src/services/encryption.ts`)

`encrypt(plaintext) → { encrypted, iv }` e `decrypt(encrypted, iv) → plaintext`, AES-256-GCM (`ENCRYPTION_KEY` de 64 hex chars no env). O service grava `encrypted` → coluna `*_encrypted` e `iv` → coluna `*_iv`. Esta story **não** escreve código de cripto — só cria as colunas que a Story 34.2 vai popular via `encryptHotmartSecret`/`decryptHotmartSecret` (wrappers sobre `encrypt`/`decrypt`).

### Segurança

NUNCA commitar/logar `client_secret` nem o Basic. Em dev, credenciais só no `.env` (gitignored). As colunas guardam apenas valores já criptografados.

### Próxima migration livre

`0051` (último ocupado em `migrations/` = `0050_seller_aliases.sql`, criado no PR #217). `0050` **NÃO está livre** → próximo número livre real = **0051**. Observação: o diretório tem dois arquivos `0041_*` e dois `0001_*` (histórico), mas o maior numérico contínuo agora é 0050 → próximo = **0051**.
