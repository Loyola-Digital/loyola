### Plugin Registration Order (CRITICAL)

A order de registro dos plugins no Fastify IMPORTA. Seguir exatamente:

```typescript
// 1. Config (primeiro — tudo depende do env)
await app.register(envPlugin);

// 2. Infrastructure (CORS, rate-limit)
await app.register(corsPlugin);
await app.register(rateLimitPlugin);

// 3. Auth middleware
await app.register(authPlugin);

// 4. Database
await app.register(dbPlugin);

// 5. Routes (ultimo — consomem services)
await app.register(healthRoutes, { prefix: '/api' });
```

[Source: architecture/fullstack-architecture#5.3]

### Service Layer Pattern

Cada service e um **Fastify plugin** que decora a instancia:

```typescript
import fp from 'fastify-plugin';

export default fp(async function dbPlugin(fastify) {
  const pool = new Pool({ connectionString: fastify.config.DATABASE_URL });
  const database = drizzle(pool, { schema });
  fastify.decorate('db', database);
});
```

[Source: architecture/fullstack-architecture#5.4]

### Auth Middleware Pattern

```typescript
import { clerkPlugin, getAuth } from '@clerk/fastify';

export default fp(async function authPlugin(fastify) {
  await fastify.register(clerkPlugin);

  fastify.addHook('onRequest', async (request, reply) => {
    if (request.url === '/api/health') return; // Skip health
    const { userId } = getAuth(request);
    if (!userId) {
      reply.code(401).send({ error: 'Unauthorized' });
      return;
    }
    request.userId = userId;
  });
});
```

[Source: architecture/fullstack-architecture#9.2]

### Database Schema Summary

**4 tabelas:**
- `users` — Synced de Clerk via webhook. clerk_id UNIQUE, email UNIQUE, role enum
- `conversations` — Chat session user↔mind. Soft delete (deleted_at). Counters denormalizados
- `messages` — Append-only. Check constraint content > 0. JSONB metadata
- `delegated_tasks` — Link ClickUp task ↔ message. clickup_task_id UNIQUE

**4 enums:** `user_role`, `message_role`, `task_status`, `task_priority`

**9 indexes:** Incluindo partial indexes para conversations (WHERE deleted_at IS NULL)

**5 triggers:**
- 3x `update_updated_at()` (users, conversations, delegated_tasks)
- 1x `update_conversation_counters()` (messages INSERT → conversations counter)
- 1x `auto_title_conversation()` (messages INSERT → conversations title from first user msg)

**IMPORTANTE:** Drizzle-kit nao gera triggers automaticamente. O SQL dos triggers deve ser criado manualmente no migration file.

[Source: architecture/database-schema-design.md#4, #5]

### Database Security Pattern

- **Sem RLS** — Railway PostgreSQL nao expoe DB diretamente. Backend e o unico client
- **Query scoping obrigatorio:** TODA query deve incluir `WHERE user_id = ?`
- **SSL** obrigatorio em producao (`rejectUnauthorized: false` para Railway self-signed certs)
- **Connection pool:** max 20 connections para MVP

[Source: architecture/database-schema-design.md#10]

### Dockerfile Pattern (Multi-stage)

```dockerfile

RUN corepack enable && corepack prepare pnpm@9 --activate

FROM base AS build
WORKDIR /app
COPY pnpm-lock.yaml pnpm-workspace.yaml package.json ./
COPY packages/api/package.json packages/api/
COPY packages/shared/package.json packages/shared/
RUN pnpm install --frozen-lockfile
COPY packages/api packages/api
COPY packages/shared packages/shared
RUN pnpm --filter @loyola-x/api build

FROM base AS runtime
WORKDIR /app
COPY --from=build /app/packages/api/dist ./dist
COPY --from=build /app/node_modules ./node_modules
```
